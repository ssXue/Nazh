## 关键决策点（已确认 2026-04-26）

1. **Phase 3 全 `Json`，不引入 `Custom`**——避免"消费者孤岛"。`Custom("sql-row")` 之类要求生产者-消费者**对**同时落地，否则只在 `Any → Custom` 兜底通道工作（校验形同虚设）。留待 Phase 4 配套 row-formatter 节点登场时一并引入。
2. **`mqttClient` 用 `&self` 实例方法区分 mode**——publish/subscribe 数据流向镜像，强行用同一对 pin 会丢失类型信息。这正是 ADR-0010 把 `output_pins` 设计为实例方法的原因（已有 `switch_node` 范式）。
3. **`modbusRead` input 保留 `Any`**——它常作为根节点（无上游）或被 `timer`（输出 `Any`）触发，input 形状不重要。output 收紧到 `Json` 是 Phase 3 的实际增益点。
4. **`httpClient` 不按 webhook_kind 分支化 pin**——三种 body_mode（json / template / dingtalk_markdown）的 input 都能被 `Json` 涵盖（`Any → Json` 兜底，`Json → Json` 直连），引入 mode-conditional pin 收益不抵复杂度。
5. **反向兼容性测试**作为反退化保险：新工程师改 pin 声明时若破坏现有链路，CI 立刻红灯（Task 5）。

## 节点 pin 设计表（最终版）

| 节点 | mode | input | output | 设计意图 |
|------|------|-------|--------|---------|
| `modbusRead` | — | `Any` | `Json` | source；input 是 trigger 信号（常被 timer 触发或作根节点）；output 是寄存器读取后合并的 JSON 对象 |
| `mqttClient` | publish | `Json` | `Any` | sink + echo；publish 期待结构化 payload；output 仅 echo input（下游基本不消费） |
| `mqttClient` | subscribe | `Any` | `Json` | source via `on_deploy`；input 留 `Any` 给手动 dispatch；output 是 `normalize_subscribed_payload` 后的对象 |
| `httpClient` | — | `Json` | `Json` | sink + source；body 入是 JSON-shaped；response 出按 `body_mode` 通常也是 JSON |
| `sqlWriter` | — | `Json` | `Any` | sink；input 必须是有列结构的 row；output 是写入确认元信息（基本不消费） |

**未涉及节点**：`timer` / `serialTrigger` / `native` / `barkPush` / `debugConsole` / `template` 保持 Phase 1 默认（`Any` / `Any`）。理由：trigger 类节点（timer/serialTrigger）数据形状由配置决定不固定；native/template/debugConsole 是透传/格式化节点，强制 `Json` 会过窄；barkPush 与 httpClient 共享路径但功能更窄，等 Phase 4 一起处理更合适。

```rust
//! ADR-0010 Phase 3：与 `tests/fixtures/pin_compat_matrix.json` 合约一致性。
//!
//! 本测试保证 Rust `PinType::is_compatible_with` 实现与 fixture 一致。
//! Phase 2 落地后，前端 TS 实现也将消费同一份 fixture——任一方漂移，CI 立刻红。

use nazh_core::PinType;
use serde::Deserialize;

#[derive(Deserialize)]
struct CompatPair { from: PinType, to: PinType, compatible: bool }

#[derive(Deserialize)]
struct Fixture { pairs: Vec<CompatPair> }

#[test]
fn rust_实现与合约 fixture 一致() {
    let path = concat!(env!("CARGO_MANIFEST_DIR"), "/../../tests/fixtures/pin_compat_matrix.json");
    let raw = std::fs::read_to_string(path).expect("fixture 必须存在");
    let fixture: Fixture = serde_json::from_str(&raw).expect("fixture 反序列化失败");
    assert!(!fixture.pairs.is_empty(), "fixture 至少应包含 1 条断言");

    for (i, pair) in fixture.pairs.iter().enumerate() {
        let actual = pair.from.is_compatible_with(&pair.to);
        assert_eq!(
            actual, pair.compatible,
            "fixture pair #{i}: {:?} → {:?} 期望 {} 但 Rust 实现给出 {}",
            pair.from, pair.to, pair.compatible, actual
        );
    }
}
```
