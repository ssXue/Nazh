# ADR-0010 Pin 声明系统实施计划（Phase 2：前端可视化 + 连接期 UI 校验）

> **Status:** 立项中（2026-04-26）
> 实施完成后请同步：ADR-0010 → Phase 2 实施记录、`docs/adr/README.md` 索引（无须改状态）、根 `AGENTS.md`（Project Status / ADR Execution Order 标注 Phase 2 已落地）、`crates/tauri-bindings/AGENTS.md`（IPC 表新增 `describe_node_pins`）、本 plan 顶部 Status 改为 `merged in <SHA>`。

**Goal:** 在用户拖边连接节点的瞬间，UI 层即时拒绝不兼容连接（**第一层防御**），提示具体不兼容原因；保留 Phase 1 已落地的部署期校验作为 backstop（**第二层防御**），覆盖 UI 绕过场景（手改 JSON / 第三方导入 / AI 直 deploy）。两层即 *defense in depth*，UI 是 UX 加速器，后端是正确性保证，不可互相替代。

**Architecture:**

- **Backend IPC (`src-tauri/`, `crates/tauri-bindings/`)**：新增 `describe_node_pins(node_type, config) → { input_pins, output_pins }` Tauri 命令。复用 `standard_registry().create(node_def)` 拿到 `Arc<dyn NodeTrait>`，调 `input_pins(&self)` / `output_pins(&self)` 即可——`mqttClient` 这种实例方法 pin 走 config 路径自然支持。
- **Frontend pin schema cache (`web/src/lib/pin-schema-cache.ts` 新文件)**：模块级 `Map<nodeId, PinSchema>` 缓存。节点添加到画布时 + 节点 config 保存时调 `describeNodePins` 填缓存；节点删除时清缓存。
- **TS 兼容性判断 (`web/src/lib/pin-compat.ts` 新文件)**：~25 行 `isCompatibleWith(from: PinType, to: PinType): boolean`，与 Rust `PinType::is_compatible_with` 严格一致——通过 Phase 3 已落地的 `tests/fixtures/pin_compat_matrix.jsonc` 跑合约测试守住一致性。
- **FlowGram `canAddLine` 钩子 (`web/src/components/FlowgramCanvas.tsx`)**：注册 `canAddLine(ctx, fromPort, toPort, lines, silent?)` → 从缓存读两端 pin schema → 调 `isCompatibleWith` → 返回 `boolean`。`silent: false` 时通过 toast / 节点 `hasError` 提示。
- **类型着色（可选）**：每个 `PinType` 变体映射到 CSS 颜色变量，写入 `data-port-pin-type` 属性，CSS 选择器消费。

**Tech Stack:** TypeScript / React 18 / FlowGram.AI free-layout-editor / Tauri 2 IPC / `jsonc-parser`（Vitest 合约测试解析 fixture）/ Vitest unit / Playwright E2E（可选）/ Rust（IPC handler + ts-rs 导出新类型）

## 关键决策点（执行前请确认）

1. **缓存粒度按 nodeId**（不是 nodeType + config hash）：`mqttClient` 改 mode 后 pin 形态切换，**必须按节点实例缓存**才能区分。失效策略：节点 config 保存事件 → 重新 IPC 取 pin schema 更新缓存。

2. **`describeNodePins` 缺失节点时的兜底**：若 IPC 返回错误（节点 type 未注册 / config 无效），缓存写入 fallback `{ input: [Any], output: [Any] }`。这样 `canAddLine` 在不确定情况下回退为"放行"——UI 层宁可放过、不要误拒（部署期 backstop 兜底）。

3. **`canAddLine` 的 `silent` 参数**：FlowGram 调用时 `silent: false` = 用户拖边，可弹错误；`silent: true` = 内部检查（如复制粘贴整张图），不弹。我们的实现两种都返回 `boolean`，但仅 `silent: false` 时调用错误反馈通道（toast / 节点高亮）。

4. **JSONC 解析器选择**：用 `jsonc-parser`（Microsoft 出品，~5KB，VS Code 同款）。**不**用 `vite-plugin-jsonc` 改 build pipeline——影响面过大，单纯一份测试 fixture 不值得。`jsonc-parser` 加进 `web/package.json` 的 devDependency。

5. **类型着色范围**：Phase 2 范围只做"端口圆点的 background-color"。pin label tooltip / 端口类型 legend 等更深入的 UX 推迟。**着色方案**：`Any` = 灰、`Json` = 蓝、`String` = 紫、`Bool` = 黄、`Integer/Float` = 绿、`Binary` = 橙、`Array<T>` = 同 T + 边框、`Custom` = 红边框。

6. **不引入连接期"为什么不兼容"的 toast 组件**：FlowGram 默认的端口红色 + node `hasError` 视觉反馈已足够。如果用户问"为什么连不上"，鼠标 hover 端口看 tooltip 即可（Phase 2 添加 `data-port-pin-type` 属性供 CSS / hover 消费）。

```rust
async fn describe_node_pins(
    request: DescribeNodePinsRequest,
) -> Result<DescribeNodePinsResponse, String> {
    let registry = standard_registry();
    let node_def = WorkflowNodeDefinition::new(
        "_describe_pins_probe".to_owned(),
```

```typescript
// web/src/lib/pin-compat.ts
import type { PinType } from '../generated/PinType';

/**
 * 判断"上游产出 from → 下游期望 to"是否兼容。
 *
 * 与 Rust `PinType::is_compatible_with` (crates/core/src/pin.rs:96) 严格一致。
 * 任意一方漂移由 tests/fixtures/pin_compat_matrix.jsonc 合约测试拒绝。
 */
export function isCompatibleWith(from: PinType, to: PinType): boolean {
  // Any 双向吃一切——匹配 ADR-0010 矩阵的前两行
  if (from.kind === 'any' || to.kind === 'any') return true;

  // 标量精确相等
  if (from.kind === to.kind && (
    from.kind === 'bool' ||
    from.kind === 'integer' ||
    from.kind === 'float' ||
    from.kind === 'string' ||
    from.kind === 'json' ||
    from.kind === 'binary'
  )) return true;

  // Array 嵌套
  if (from.kind === 'array' && to.kind === 'array') {
    return isCompatibleWith(from.inner, to.inner);
  }

  // Custom 命名严格匹配
  if (from.kind === 'custom' && to.kind === 'custom') {
    return from.name === to.name;
  }

  return false;
}
```

> **类型守卫小心**：`PinType` 是可辨识联合（`{ kind: 'array'; inner: PinType }`），TS 通过 `from.kind === 'array'` 缩窄到带 `inner` 的分支。如果 ts-rs 导出的形态与此不符，调整这里的判断。

```typescript
// web/src/lib/pin-schema-cache.ts
import { describeNodePins } from './tauri';
import type { PinDefinition } from '../generated/PinDefinition';
import type { PinType } from '../generated/PinType';

interface CachedSchema {
  inputPins: PinDefinition[];
  outputPins: PinDefinition[];
}

const FALLBACK: CachedSchema = {
  inputPins: [{ id: 'in', label: 'in', pinType: { kind: 'any' } as PinType,
                direction: 'input', required: true }],
  outputPins: [{ id: 'out', label: 'out', pinType: { kind: 'any' } as PinType,
                 direction: 'output', required: false }],
};

const cache = new Map<string, CachedSchema>();

export async function refreshNodePinSchema(
  nodeId: string,
  nodeType: string,
  config: Record<string, unknown>,
): Promise<CachedSchema> {
  try {
    const response = await describeNodePins(nodeType, config);
    const schema = { inputPins: response.inputPins, outputPins: response.outputPins };
    cache.set(nodeId, schema);
    return schema;
  } catch (error) {
    console.warn(`[pin-cache] describe_node_pins 失败，节点 ${nodeId} 走 fallback：`, error);
    cache.set(nodeId, FALLBACK);
    return FALLBACK;
  }
}

export function getCachedPinSchema(nodeId: string): CachedSchema | undefined {
  return cache.get(nodeId);
}

export function findPin(
  nodeId: string,
  portId: string | number,
  direction: 'input' | 'output',
): PinDefinition | undefined {
  const schema = cache.get(nodeId);
  if (!schema) return undefined;
  const pins = direction === 'input' ? schema.inputPins : schema.outputPins;
  return pins.find((pin) => pin.id === String(portId));
}

export function invalidateNodePinSchema(nodeId: string): void {
  cache.delete(nodeId);
}

/** 测试钩子；生产路径不应直接调。 */
export function _resetCacheForTests(): void {
  cache.clear();
}
```

- [ ] **Step 2：实现 `canAddLine`**

```typescript
import { findPin } from '../lib/pin-schema-cache';
import { isCompatibleWith } from '../lib/pin-compat';

const canAddLine = useCallback(
  (
    _ctx: FreeLayoutPluginContext,
    fromPort: WorkflowPortEntity,
    toPort: WorkflowPortEntity,
    _lines: WorkflowLinesManager,
    _silent?: boolean,
  ): boolean => {
    const fromPin = findPin(fromPort.node.id, fromPort.portID, 'output');
    const toPin = findPin(toPort.node.id, toPort.portID, 'input');

    // 缓存未命中 = 还没拿到 schema → 放行（部署期 backstop 兜底）
    if (!fromPin || !toPin) return true;

    const compatible = isCompatibleWith(fromPin.pinType, toPin.pinType);
    if (!compatible && !_silent) {
      // 视觉反馈：标记目标端口 hasError = true，让默认 isErrorPort 渲染红色
      toPort.hasError = true;
      // 200ms 后清除；FlowGram 自身 isErrorLine 在拒收返回 false 后不会真创建线
      setTimeout(() => { toPort.hasError = false; }, 1500);
    }
    return compatible;
  },
  [],
);
```
