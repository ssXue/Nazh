# ADR-0005: ConnectionManager 改用细粒度连接锁

- **状态**: 已接受
- **日期**: 2026-04-09
- **决策者**: Niu Zhihong
- **关联**: ADR-0001

## 背景

当前 `ConnectionManager` 使用全局 `Arc<RwLock<ConnectionManager>>` 保护整个连接池：

```rust
pub type SharedConnectionManager = Arc<RwLock<ConnectionManager>>;
```

每次 `borrow()` 或 `release()` 都需要获取**全局写锁**（`write().await`），导致：

1. 两个节点并发借出**不同的**连接时互相阻塞——一个节点借 Modbus 连接时，另一个节点无法借 MQTT 连接。
2. 在 DAG 拓扑中，多个并行分支各自使用独立连接时形成不必要的串行瓶颈。
3. 未来接入真实 Modbus TCP / MQTT / OPC-UA 驱动后，连接操作耗时将显著增加（涉及网络 I/O），全局锁的影响会加剧。

当前这不是性能问题（连接池是骨架实现，操作都是内存操作），但在接入真实协议驱动**之前**解决锁粒度是低成本高收益的，可以避免未来的大规模锁重构。

## 决策

> 我们决定将 `ConnectionManager` 内部改为每连接独立锁，使 DAG 并行分支可真正并发访问不同连接，因为当前的全局写锁会在真实协议驱动接入后成为不必要的瓶颈。

具体结构变化：

```rust
// 旧：全局读写锁
pub type SharedConnectionManager = Arc<RwLock<ConnectionManager>>;

struct ConnectionManager {
    connections: HashMap<String, ConnectionRecord>,
}

// 新：外层只需 Arc，内部每连接独立锁
pub type SharedConnectionManager = Arc<ConnectionManager>;

struct ConnectionManager {
    connections: RwLock<HashMap<String, Arc<Mutex<ConnectionRecord>>>>,
}
```

- 注册连接（`upsert_connection`）：获取外层 `RwLock` 的写锁，修改 HashMap
- 借出/释放连接（`borrow` / `release`）：只获取外层 `RwLock` 的读锁定位到具体连接，然后锁单个连接的 `Mutex`
- 查询连接（`get` / `list`）：获取外层 `RwLock` 的读锁

## 可选方案

### 方案 A: 维持全局 RwLock

- 优势：最简单，当前骨架实现足够
- 劣势：引入真实驱动后成为性能瓶颈；DAG 并行分支被人为串行化

### 方案 B: `DashMap<String, Mutex<ConnectionRecord>>`

- 优势：零全局锁，分片并发，注册/查询更高效
- 劣势：引入新外部依赖 `dashmap`；对于当前连接数量（个位数）无实际性能差异

### 方案 C: `RwLock<HashMap<String, Arc<Mutex<ConnectionRecord>>>>`（已选）

- 优势：零新依赖（仅使用 `tokio::sync::RwLock` 和 `tokio::sync::Mutex`，均已可用）；注册操作走外层写锁，运行时借出/释放只锁单个连接
- 劣势：两层锁结构增加内部复杂度；注册新连接仍需外层写锁

### 方案 D: 每连接 Semaphore

- 优势：语义精确（信号量天然表达"资源可用性"）
- 劣势：需要额外数据结构保存连接元数据，Semaphore 无法传递 lease 信息

## 后果

### 正面影响

- DAG 并行分支可真正并发访问不同连接，无互相阻塞
- 为真实 Modbus / MQTT / HTTP 驱动铺路，避免未来的锁重构
- `helpers::with_connection` 的外部接口不变，所有节点代码无需修改
- `SharedConnectionManager` 类型别名不变，调用方无感知

### 负面影响

- `ConnectionManager` 内部复杂度增加（两层锁嵌套）
- `borrow` / `release` 从同步方法变为异步方法（内部 Mutex 是 tokio 异步锁）
- 注册操作（`upsert_connection`）需要外层写锁，但这只在 `deploy_workflow` 时调用一次，不影响运行时

### 风险

- 两个节点借出**同一个**连接仍然串行（这是设计意图：排他借出语义不变）。
- `register_connection` 存在 TOCTOU 窗口：先读锁检查存在性，释放后再写锁插入。两个并发注册同一 ID 理论上可同时通过检查。实际不影响——注册仅在 `deploy_workflow` 中串行调用。
- 如果将来需要连接复用（同一连接被多个节点读共享），需要进一步改造为读写锁或连接池模式。
