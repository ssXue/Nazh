# ADR-0009: 为 NodeTrait 增加 `on_deploy` 钩子与 `LifecycleGuard` RAII 清理

- **状态**: 已接受
- **日期**: 2026-04-24（提议）/ 2026-04-26（已接受，启动实施）
- **决策者**: Niu Zhihong
- **关联**: 依赖 ADR-0008 的 `transform` 签名；对应内部提案「提案-03 节点生命周期钩子」；后续 Pin 系统（提案-01）可在此基础上添加类型化引脚

## 背景

当前 `NodeTrait` 只有一个核心方法 `transform(trace_id, payload) → NodeExecution`。这对**纯变换型节点**（`code`、`if`、`switch`、`httpClient` 单次请求、`modbusRead` 单次读取）是合适的，但对**触发器型节点**和**长连接型节点**不够用：

- **MQTT 订阅者**：需要在部署时建立到 broker 的长连接、订阅指定 topic、维持心跳；每条收到的消息应触发下游一次执行；撤销部署时必须干净地断开连接、释放订阅。
- **Modbus 轮询根**：需要在部署时建立 TCP 连接、按配置周期轮询寄存器；撤销部署时必须停掉轮询线程。
- **串口监听（serial）**：需要打开串口、按帧格式解析；撤销部署时必须关闭串口句柄。
- **定时器根**：需要在部署时拉起一个按间隔 tick 的后台任务；撤销时必须取消任务。

### 当前处境（2026-04-24）：壳层代持长连接

这些节点目前都**没有**生命周期钩子的支撑，实际做法是在 Tauri 壳层（`src-tauri/src/lib.rs`）直接管理（行号截至 2026-04-24）：

| 节点类型 | 壳层代持函数（节选） | 精确行号 |
|---------|---------------------|---------|
| `timer` | `spawn_timer_root_tasks` | 2499 |
| `serial`（监听） | `spawn_serial_root_tasks` | 2545 |
| `mqttClient`（订阅模式） | `collect_mqtt_root_specs` | 2584 |
| `mqttClient`（订阅模式） | `spawn_mqtt_root_tasks` | 2680 |
| `mqttClient`（订阅模式） | `run_mqtt_root_subscriber` | 2719 |

所有这些后台任务都共享一条相同的取消惯例——**使用 `Arc<AtomicBool>` 作为取消信号**（见 `sleep_with_cancel(cancel: &AtomicBool, ...)` 辅助函数在 `src-tauri/src/lib.rs:3319`）。这些代码直接读取 `graph.nodes` 找出特定 `node_type`，借用 `ConnectionManager` 里的资源，自己 `tokio::spawn` 一个订阅/轮询任务，通过 `WorkflowDispatchRouter` 向 DAG 内部注入数据。这违反了几项设计契约：

1. **Ring 分层违反**：壳层（Ring 外）本不该了解某个 Ring 1 节点的协议细节（MQTT QoS、topic、keep-alive）
2. **Plugin 开闭原则失效**：每次加一种长连接节点，都得改 `src-tauri/src/lib.rs`——而 Plugin 系统的初衷就是"核心与壳层与节点三者互不相识"
3. **测试成本高**：订阅逻辑藏在 Tauri AppHandle 里，很难脱离桌面壳层做单元测试
4. **AI 复制粘贴场景不可行**：用户未来可能希望 AI 生成一个新的"Kafka 订阅节点"，如果需要改壳层才能用，AI 工作流就断了

## 决策

> 我们决定在 `NodeTrait` 上新增 **`on_deploy`**（带默认空实现）钩子，返回一个 **`LifecycleGuard`（RAII 句柄）**。Runner 在节点任务启动前调用 `on_deploy`，在工作流撤销时按逆拓扑序 drop 这些 guard——清理逻辑通过 `LifecycleGuard` 的 `Drop` 实现触发，不再需要单独的 `on_undeploy` 方法。配合 `NodeHandle` 抽象让长连接型节点能把消息推进 DAG 数据通道；升级现有壳层的 `Arc<AtomicBool>` 取消信号为 `tokio_util::sync::CancellationToken`（更适配异步等待）。整体目标：把 MQTT/Modbus/Serial/Timer 的壳层代持代码迁回引擎。

### 拟议 Trait 扩展

```rust
// crates/core/src/node.rs
#[async_trait]
pub trait NodeTrait: Send + Sync {
    fn id(&self) -> &str;
    fn kind(&self) -> &'static str;

    /// 纯变换路径：收到上游数据时被 Runner 调用。触发器节点可返回空输出。
    async fn transform(&self, trace_id: Uuid, payload: Value)
        -> Result<NodeExecution, EngineError>;

    /// 节点部署时调用，早于任何 `transform`。用于建立长连接、订阅、拉起后台任务。
    ///
    /// 返回的 `LifecycleGuard` 由 Runner 持有。**不存在单独的 `on_undeploy` 方法**——
    /// 清理逻辑通过 `LifecycleGuard` 的 `Drop` 实现触发（RAII 模式，与现有 `ConnectionGuard` 一致）。
    ///
    /// 默认实现返回 `LifecycleGuard::noop()`：纯变换节点无需清理，guard 析构时什么也不做。
    async fn on_deploy(
        &self,
        _ctx: NodeLifecycleContext,
    ) -> Result<LifecycleGuard, EngineError> {
        Ok(LifecycleGuard::noop())
    }
}

/// 部署钩子可用的受限上下文。
pub struct NodeLifecycleContext {
    pub resources: SharedResources,     // 与节点工厂同款的资源包
    pub handle: NodeHandle,              // 允许触发器节点向 DAG 推入数据
    pub shutdown: CancellationToken,     // 优雅停止信号（tokio_util）
}

/// 允许长连接型节点把外部消息 "喂" 进 DAG 数据通道。
/// 内部封装 DataStore 写入 + 下游广播 + 事件上报。
pub struct NodeHandle { /* ... */ }

impl NodeHandle {
    pub async fn emit(&self, payload: Value, metadata: Map<String, Value>)
        -> Result<(), EngineError>;
}

/// RAII 句柄。Drop 时负责：取消内部任务（通过持有的 `CancellationToken`）、
/// 等待后台任务完成、释放协议资源。
///
/// 支持可选的 `shutdown().await` 让调用方显式等待清理完成（否则只能依赖 Drop 异步隐式执行）。
pub struct LifecycleGuard { /* 持有 JoinHandle + CancellationToken + 资源释放闭包 */ }

impl LifecycleGuard {
    pub fn noop() -> Self;              // 纯变换节点用
    pub async fn shutdown(self);         // 显式清理并等待
}
```

### 取消信号：为什么升级 `Arc<AtomicBool>` 为 `CancellationToken`

现有壳层代码（见背景一节）使用 `Arc<AtomicBool>` + 轮询（如 `sleep_with_cancel`）做取消。这个模式在"同步循环或短 sleep"里够用，但对异步订阅者很别扭——MQTT 的 `EventLoop::poll().await` 不感知 AtomicBool，只能靠轮询超时间接响应。

`tokio_util::sync::CancellationToken` 提供原生 async 取消语义：`token.cancelled().await` 可与任意其他 future 用 `tokio::select!` 组合。迁移过程中可让两种原语共存，最终统一到 `CancellationToken`。

### Runner 侧调用时序

```
WorkflowInstance::deploy():
    for node in topological_order:
        guard = node.on_deploy(ctx).await?     // 建连/订阅；失败 → 整图回滚
        guards.insert(node.id, guard)          // 按部署顺序保存

    spawn per-node transform loop              // 现有路径

WorkflowInstance::undeploy():
    shutdown_token.cancel()                    // 通知所有后台任务退出
    for node in reverse_topological_order:
        guard = guards.remove(node.id)
        guard.shutdown().await                 // 显式等待清理完成
        // 若没走 shutdown，Drop 在 guards 容器销毁时兜底触发清理
```

### 错误与隔离策略

- `on_deploy` 失败 → 已完成的节点按**逆拓扑序** drop guard（RAII 自动清理）；整张图进入 `DeployFailed` 状态，不接受 `dispatch`
- `on_deploy` 调用同样受 `AssertUnwindSafe + catch_unwind + timeout` 三重包裹（默认 10s 超时）；`LifecycleGuard::shutdown` 也受超时保护（默认 5s）
- `NodeHandle::emit` 内部在节点未完成 `on_deploy` 之前调用会返回 `LifecycleNotReady` 错误——防止时序错乱

## 可选方案

### 方案 A: 维持现状（壳层代持）

- 优势：已经能跑；无需改 Trait 签名
- 劣势：
  - `src-tauri/src/lib.rs` 已经 3600+ 行，每加一种长连接节点就膨胀一次
  - Ring 分层被破坏，无法通过外部插件（未来的 WASM 模块）扩展新协议
  - 测试必须拖着整个 Tauri AppHandle
- 结论：随着协议节点增多，这条路走不下去

### 方案 B: 拆一个独立的 `TriggerNode` trait

```rust
pub trait TriggerNode: NodeTrait {
    async fn start(&self, handle: NodeHandle, shutdown: CancellationToken)
        -> Result<(), EngineError>;
}
```

- 优势：触发器节点语义独立，静态区分"源头"与"变换"
- 劣势：
  - 引入节点种类的**多态分裂**——Runner 要先 `Any::downcast` 看是否实现 `TriggerNode`，复杂度上升
  - 不能覆盖"纯变换节点也想做 warmup"的场景（例如 Rhai 节点可以在 `on_deploy` 预编译 AST）
  - Pin 系统、Capability Tag 等后续能力都要在两套 trait 上分别实现

### 方案 C: 默认空实现的 `on_deploy` + `LifecycleGuard` RAII 清理（已选）

- 优势：
  - **单一 trait，零破坏**：已有节点完全不需要改
  - **统一生命周期语义**：不管节点是触发器还是纯变换，Runner 调用序一致
  - 可选 warmup：任何节点都能在 `on_deploy` 做准备（预编译、建连池、加载模型权重）
  - 与 RAII `ConnectionGuard` 风格一致——`LifecycleGuard` 是它的节点级对应
- 劣势：
  - Trait 变"胖"一点（但只增加两个默认方法，相比 Pin 系统改动微小）
  - `NodeHandle` 打破了 `transform` 是纯函数的干净契约——但这是必要妥协，因为触发器本质上**不是**纯函数
- 结论：最佳平衡

### 方案 D: 用 Plugin 级别的 `on_ready` / `on_shutdown` 而不是节点级

- 优势：Plugin trait 里已有类似 hook 的设计空间
- 劣势：
  - 粒度完全错位——Plugin 代表"一批节点类型"，不是"某个部署的节点实例"
  - 节点实例是每次 deploy 动态创建的，Plugin 的 on_ready 只在进程启动时调用一次，没法管单个订阅连接
- 结论：不适用

## 后果

### 正面影响

- **壳层代码瘦身**：`src-tauri/src/lib.rs` 中约 600-800 行 MQTT/Timer/Serial 代持代码可以迁回引擎；壳层回归"IPC 桥接 + UI 集成"职责。
- **新长连接节点零壳层改动**：未来加 Kafka、WebSocket、OPC-UA 订阅等节点时，作者只需实现 `on_deploy` 建连 + `NodeHandle::emit` 推数据，不碰 Tauri 壳层。
- **Rhai / code 节点 warmup**：可以在 `on_deploy` 预编译 Rhai AST，摆脱"每次 new Engine" 的开销（顺带解决架构 Review 里列出的小问题之一）。
- **为 Pin 系统打基础**：`NodeLifecycleContext` 将来可以承载 `PinSchema` 校验结果、类型化资源等；先把生命周期骨架搭出来，后续能力可插拔。
- **测试性提升**：触发器节点可以用 `MockNodeHandle` 单测，不再依赖桌面壳层。

### 负面影响

- 核心 trait 接口变大，实现者的心智负担略增（即使有默认实现）
- Runner 的部署/撤销路径需要重构，引入 `LifecycleGuard` 容器、取消令牌、逆拓扑序清理——这部分必须要有详细的**超时 & 失败回滚**测试覆盖
- `NodeHandle::emit` 是一条新的"异步写 DataStore + 广播 ContextRef"路径，必须复用现有 Runner 的 `apply_output` 实现，否则会出现两份元数据合并逻辑

### 风险

- **风险 1：部署时序错乱**。订阅者 `on_deploy` 建连完成前就收到消息，`NodeHandle::emit` 可能打到未就绪的下游通道。
  - 缓解：Runner 在所有节点 `on_deploy` 完成**之后**才启动 transform 任务 & 对外开放 `dispatch`。`NodeHandle::emit` 内部在 ready 前缓冲或拒收。
- **风险 2：撤销竞态**。外部消息正好在 guard 清理中途到达。
  - 缓解：`CancellationToken` cancel 后，`NodeHandle::emit` 立即返回 `Cancelled`；订阅者在后台任务的 `tokio::select!` 第一分支监听 `token.cancelled()`，进入清理前先退订阅循环。
- **风险 3：回滚不干净**。`on_deploy` 对前 N 个节点成功、第 N+1 个失败时，前 N 个的资源必须释放。
  - 缓解：`LifecycleGuard` Drop 兜底 + 显式 `shutdown` 等待，双重保险；回滚按逆拓扑序执行。
- **风险 4：迁移期间的双轨运行**。MQTT 节点迁回引擎后，壳层旧代码需要一次性删除，否则同一份订阅会被双份建立。
  - 缓解：PR 必须"同一次合并完成迁移与删除"；不允许临时保留壳层路径作 fallback。

## 备注

- 本 ADR 仅定义**决策**与**接口形状**。具体实施拆任务见 `docs/superpowers/plans/2026-04-26-node-lifecycle-hooks.md`，包含：Ring 0 lifecycle 模块骨架、Runner 部署/撤销路径改造、Timer/Serial/MQTT 三类迁移、文档同步、E2E 矩阵。
