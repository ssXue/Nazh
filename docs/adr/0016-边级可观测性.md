# ADR-0016: 边级可观测性（EdgeTransmit / BackpressureDetected 事件）

- **状态**: 已接受
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-09 边级可观测性」；扩展 ADR-0004 统一执行事件模型；与 ADR-0014 边类型分离配合（不同边类型有不同观测维度）

## 背景

当前 `ExecutionEvent` 只跟踪**节点级别**的生命周期：`Started` / `Completed` / `Failed` / `Output` / `Finished`。对节点自身的诊断够用，但工业边缘的 DAG 规模上来后，一类关键问题无法回答：

1. **节点之间的等待时间有多长？**（payload 在 MPSC channel 里排队的时间）
2. **哪条边形成了瓶颈？**（下游处理慢 → channel 积压）
3. **payload 体积分布是什么样？**（某条边是否被大 JSON 载荷拖慢）
4. **背压在哪里产生？**（哪个节点的接收通道在溢出、触发丢弃策略）

这些问题用"节点级延迟"无法切分——上游 `Completed` 到下游 `Started` 的时间差里，channel 排队、Runner 调度、下游 warmup 都混在一起。

UE5 Blueprint 的调试器可以在连线上显示"本次触发流了哪些数据"——Nazh 当前做不到。FlowGram 画布的能力已经具备（线条高亮 + 数据预览），缺的是引擎侧的事件源头。

## 决策

> 我们决定扩展 `ExecutionEvent` 加入**边级事件**（`EdgeTransmit` 和 `BackpressureDetected`），由 Runner 在每次跨边传输和每次发现 channel 积压时发出。前端订阅这些事件，在画布上叠加一层"边热力图"——线条粗细表示流量、颜色表示延迟、闪烁警告表示背压。

### 拟议事件变体

```rust
// crates/core/src/event.rs
pub enum ExecutionEvent {
    // ... existing variants: Started / Completed / Failed / Output / Finished ...

    /// 边上传输一次数据
    EdgeTransmit {
        trace_id: Uuid,
        from_node: String,
        from_pin: String,
        to_node: String,
        to_pin: String,
        edge_kind: EdgeKind,       // ADR-0014
        emitted_at: DateTime<Utc>,
        received_at: DateTime<Utc>, // 下游收到的时间
        payload_bytes: usize,       // payload 序列化后字节数
        queue_depth: usize,         // 发送时 channel 里已有多少条
    },

    /// 背压告警：下游 channel 接近/达到容量上限
    BackpressureDetected {
        at_node: String,
        incoming_pin: String,
        channel_capacity: usize,
        channel_depth: usize,
        policy: BackpressurePolicy,    // Block / DropNewest / DropOldest / Sample
        dropped_since_last_report: u64, // 自上次报告以来丢弃的消息数
        detected_at: DateTime<Utc>,
    },
}

pub enum BackpressurePolicy {
    Block, DropNewest, DropOldest, Sample, Overflow,
}
```

### 发出时机

- **EdgeTransmit**：Runner 在 `mpsc::send().await` 完成后立即发出一条（`emitted_at` 从节点 `Completed` 继承，`received_at` 是接收端准备读取的时刻）
- **BackpressureDetected**：Runner 定期（或事件驱动）检查每条入站 channel 的深度；当深度 ≥ capacity × 80% 时发第一条，继续恶化每 1s 刷新一次；丢弃计数从"超过阈值"开始累计

### 采样策略（避免事件爆炸）

`EdgeTransmit` 在高频流中（10Hz 以上）会淹没事件通道。默认策略：

- **全采样模式**（调试用）：每条 transmit 都发事件
- **采样模式**（默认）：每边每 100ms 最多发一条汇总事件；汇总包含该窗口内的次数、平均延迟、平均字节数
- **关闭模式**（生产常驻）：只发 `BackpressureDetected`，不发 `EdgeTransmit`

模式由部署参数或前端"观测开关"控制。

## 可选方案

### 方案 A: 维持仅节点级事件（现状）

- 优势：事件总量可控
- 劣势：前述 4 个问题无解；调试大图只能靠猜

### 方案 B: 日志采集 + 外部分析

- Runner 写结构化日志，用户外挂 Prometheus / Loki 分析
- 优势：不改事件系统
- 劣势：
  - 失去实时性——Nazh 是桌面应用，画布上显示是核心能力
  - 工业边缘设备上跑完整日志栈不现实
  - 脱离画布的数字表格，调试体验远不如 UE5 Blueprint 式的可视化

### 方案 C: 扩展事件模型（已选）

- 优势：
  - 沿用现有 `ExecutionEvent` 通道，前端订阅逻辑零改动
  - 画布直接渲染"边热力"，调试体验类 UE5 但更工业化
  - 采样策略灵活，调试/生产切换方便
  - 与 ADR-0014 边类型配合：Reactive 边的订阅事件、Data 边的拉取事件都可以复用同一套
- 劣势：
  - 事件变体增加，`ExecutionEvent` 枚举膨胀
  - 高频场景下事件量巨大，采样与汇总逻辑必须严谨
  - 前端热力图渲染对大图性能有挑战

### 方案 D: 独立的 `TelemetryEvent` 通道

- 把观测事件与业务事件分通道
- 优势：业务事件纯净
- 劣势：
  - 前端要订阅两条事件流，时序对齐麻烦
  - 引入二次事件模型——ADR-0004 的统一事件契约被打破

## 后果

### 正面影响

- **调试体验对标 UE5**：画布上可见边粗细、颜色、闪烁——一眼看到瓶颈
- **背压可观测**：channel 深度、丢弃次数、策略效果实时可见，为未来的 `EdgeBackpressure`（RFC-0002 Phase 6）调优提供数据
- **性能回归可发现**：某个 PR 后发现某边延迟上升 → 快速定位
- **运营价值**：生产环境保留 `BackpressureDetected` 告警，能预警系统容量问题
- **是 ADR-0004 的自然延伸**：不重复造事件模型，保持前后端契约一致

### 负面影响

- `ExecutionEvent` 枚举膨胀（新增 2 个变体 + 子枚举）
- 高频采样模式下事件通道压力大——ts-rs 生成的前端类型处理也更重
- 前端热力图渲染在 >500 节点的大图上可能卡顿，需要虚拟化/抽稀

### 风险

- **风险 1：事件洪泛**——默认全采样会淹没前端
  - 缓解：默认"采样 + 汇总"模式；只在用户点击"高精度调试"时开启全采样
- **风险 2：时钟精度**——`emitted_at` / `received_at` 差值可能是负数（时钟抖动）
  - 缓解：统一用 `Instant::now()` 的单调计时，只在序列化前转成 DateTime；差值为负时记作 0
- **风险 3：`payload_bytes` 统计开销**——每次 transmit 都 serde 一次只为了测字节数成本高
  - 缓解：改为"采样测量 1/100"或"传输时已有序列化则顺便记录"；非关键指标不必精确
- **风险 4：边级事件与节点事件的因果对应**——UI 要把一次 trace 的边事件和节点事件串起来
  - 缓解：所有边事件强制携带 `trace_id`；前端按 trace 聚合

## 备注

- 本 ADR 建议放在 Phase 6（RFC-0002 中提到的 EventBus + EdgeBackpressure + ConcurrencyPolicy）窗口内实施——EdgeBackpressure 策略本身会产生新的观测需求，两者一起做效率最高。
- 实施阶段建议：先上 `EdgeTransmit` 的**汇总模式**（每 100ms 一条），验证事件量可控后再上背压检测；全采样模式可以作为调试开关延后。
- 与 OpenTelemetry 对接是更远的未来——如果项目后续引入 OTLP exporter，`EdgeTransmit` 天然映射为 span event。
