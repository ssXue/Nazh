# ADR-0007: ts-rs 前后端类型契约守卫

- **状态**: 已实施
- **日期**: 2026-04-09
- **决策者**: Niu Zhihong
- **关联**: 无

## 背景

### 测试覆盖现状

当前项目测试集中在 Rust 集成测试层（18 个测试），覆盖了 DAG 执行、流水线韧性、Rhai 脚本等核心路径。但存在三个结构性盲区：

| 层级 | 覆盖度 | 风险 |
|------|--------|------|
| Rust 单元测试（guard、template） | 高 | 低 |
| Rust 集成测试（pipeline、workflow） | 中高 | 中 |
| **前端测试** | **0%** | **极高** |
| **前后端 IPC 契约** | **0%** | **极高** |

### 核心风险场景

前后端通过 Tauri IPC 通信，共享以下数据结构：

**Tauri 命令（请求/响应）：**
- `deploy_workflow(ast: String)` → `DeployResponse`
- `dispatch_payload(payload: Value)` → `DispatchResponse`
- `undeploy_workflow()` → `UndeployResponse`
- `list_connections()` → `Vec<ConnectionRecord>`

**事件推送（`Window::emit`）：**
- `workflow://node-status` → `ExecutionEvent`
- `workflow://result` → `WorkflowContext`
- `workflow://deployed` → `DeployResponse`
- `workflow://undeployed` → `UndeployResponse`

**输入 AST（前端 → 后端）：**
- `WorkflowGraph`、`WorkflowNodeDefinition`、`WorkflowEdge`、`ConnectionDefinition`

当前这些类型在 Rust（`src/`）和 TypeScript（`web/src/types.ts`）中各自手动维护一份，没有任何机制保证同步。改了 Rust 字段 → `cargo test` 全绿 → 前端运行时静默失败。

### 已发现的潜在不一致

| Rust 类型 | TS 类型 | 差异 |
|-----------|---------|------|
| `ConnectionRecord`（snake_case 序列化） | `ConnectionRecord`（混用 snake_case） | serde 未加 `rename_all`，TS 侧 `in_use` / `last_borrowed_at` 是 snake_case，与其他 Response 类型的 camelCase 不一致 |
| `WorkflowNodeDefinition.node_type`（`#[serde(rename = "type")]`） | `WorkflowNodeDefinition.type` | 一致，但依赖 serde alias 隐式对齐 |
| `WorkflowEdge.from/to`（`#[serde(alias = "source")]`） | `WorkflowEdge.from/to` | 一致 |
| 无 `meta` 字段 | `WorkflowNodeDefinition.meta` | TS 侧多了 `meta.position`，Rust 用 `#[serde(default)]` 忽略 |

## 决策

> 我们决定采用 **ts-rs** 从 Rust struct 自动生成 TypeScript 类型定义，让 Rust 成为类型的唯一真相源（single source of truth）。

## 可选方案

### 方案 A: ts-rs（Rust struct → .ts 自动生成）

给 IPC 边界的 Rust struct 加 `#[derive(TS)]`，`cargo test` 时自动输出 `.ts` 文件到 `web/src/generated/`。

- 优势：零运行时开销，改字段 → 重新生成 → `tsc` 编译报错，一条龙
- 优势：只动 Rust 侧，前端 import 路径改一下即可
- 优势：社区活跃，支持 serde rename/alias、chrono、uuid
- 劣势：不生成带类型的 invoke 包装（需手动维护 IPC 调用签名）

### 方案 B: tauri-specta（Rust → 带类型的 Tauri invoke 包装）

不仅生成类型，还生成 `commands.deployWorkflow(ast)` 这样的类型安全调用函数。

- 优势：类型 + 调用签名一起生成，最强守卫
- 劣势：侵入性大，需要改 Tauri command 注册方式
- 劣势：与 Tauri v2 的兼容性需要验证
- 劣势：项目当前只有 4 个命令，收益有限

### 方案 C: insta 快照测试（序列化断言）

把关键结构体序列化成 JSON，与 `.snap` 文件比对。

- 优势：最轻量，几行代码
- 劣势：只能提醒人去改前端，不能阻止 `tsc` 编译
- 劣势：不生成任何前端代码

**选定方案 A**，理由：投入产出比最高。方案 B 可作为后续演进方向，方案 C 可补充使用。

## 实施计划

### 第一步：引入 ts-rs 依赖

```toml
# Cargo.toml [dependencies]
ts-rs = { version = "10", features = ["serde-compat", "chrono-impl", "uuid-impl"] }
```

### 第二步：标注 IPC 边界类型

需要加 `#[derive(TS)]` 和 `#[ts(export, export_to = "...")]` 的类型清单：

**引擎核心（`src/`）— 高优先级：**

| 文件 | 类型 | 说明 |
|------|------|------|
| `event.rs` | `ExecutionEvent` | 事件推送 payload |
| `context.rs` | `WorkflowContext` | 结果推送 payload + 输入 |
| `connection.rs` | `ConnectionDefinition` | AST 输入 |
| `connection.rs` | `ConnectionRecord` | `list_connections` 返回值 |
| `graph/types.rs` | `WorkflowGraph` | AST 顶层输入 |
| `graph/types.rs` | `WorkflowNodeDefinition` | AST 节点定义 |
| `graph/types.rs` | `WorkflowEdge` | AST 边定义 |

**Tauri 壳层（`src-tauri/`）— 高优先级：**

| 文件 | 类型 | 说明 |
|------|------|------|
| `lib.rs` | `DeployResponse` | 命令返回值 + 事件 payload |
| `lib.rs` | `DispatchResponse` | 命令返回值 |
| `lib.rs` | `UndeployResponse` | 命令返回值 + 事件 payload |

**注意**：Tauri 壳层的三个 Response struct 目前定义在 `src-tauri/src/lib.rs`，不在引擎 crate 中。有两种处理方式：
1. 把它们移到引擎 crate 的 `src/` 中（推荐，统一管理）
2. 在 `src-tauri/Cargo.toml` 也加 ts-rs 依赖

**节点配置（`src/nodes/`）— 中优先级：**

所有 `*NodeConfig` struct 也应标注，但可以第二批处理。

### 第三步：配置输出路径

在每个 `#[ts(export)]` 中指定输出到 `web/src/generated/`：

```rust
#[derive(TS)]
#[ts(export, export_to = "../../web/src/generated/")]
pub struct WorkflowGraph { ... }
```

或在 `Cargo.toml` 中全局配置：

```toml
[package.metadata.ts-rs]
output_directory = "web/src/generated"
```

### 第四步：更新前端 import

将 `web/src/types.ts` 中与 Rust 对应的 interface 替换为从 `generated/` 导入：

```typescript
// 之前
export interface ExecutionEvent { ... }

// 之后
export type { ExecutionEvent } from './generated/ExecutionEvent';
```

保留 `types.ts` 中前端独有的类型（`WorkflowRuntimeState`、`RuntimeLogEntry` 等）。

### 第五步：CI 集成

在 CI 流程中加入：
```bash
cargo test          # 触发 ts-rs 生成
npm --prefix web run build  # tsc 编译，类型不匹配会报错
```

### 需要注意的边界情况

1. **serde rename**：`ConnectionDefinition.kind` 用了 `#[serde(rename = "type")]`，ts-rs 的 `serde-compat` feature 会正确处理
2. **camelCase**：`DeployResponse` 等用了 `#[serde(rename_all = "camelCase")]`，ts-rs 会生成 camelCase 字段名
3. **ConnectionRecord 没有 rename_all**：当前 Rust 侧序列化为 snake_case（`in_use`），TS 侧也用 snake_case，保持一致。但需要确认这是有意为之还是遗漏
4. **Uuid → string**：ts-rs 的 `uuid-impl` 会将 `Uuid` 映射为 `string`
5. **DateTime<Utc> → string**：ts-rs 的 `chrono-impl` 会将 `DateTime<Utc>` 映射为 `string`
6. **serde_json::Value → any**：ts-rs 会将 `Value` 映射为 TypeScript 的某个 JSON 类型，需要验证是否与现有 `JsonValue` 兼容

## 后果

### 正面影响

- 改 Rust 字段 → `cargo test` 重新生成 TS → `tsc` 编译失败 → 开发者在 CI 阶段就能发现
- 消除手动维护两份类型定义的负担
- 为后续升级到 tauri-specta 打下基础

### 负面影响

- `cargo test` 会产生文件写入副作用（生成 .ts 文件）
- 引入新的编译期依赖

### 风险

- ts-rs 对某些复杂 serde 属性的支持可能不完善，需要逐个类型验证生成结果
- 前端 `WorkflowNodeDefinition.meta` 字段在 Rust 侧不存在，迁移时需要特殊处理

## 备注

- ts-rs 仓库：https://github.com/Aleph-Alpha/ts-rs
- tauri-specta 仓库：https://github.com/specta-rs/tauri-specta（后续演进备选）
- 前端独有类型（`WorkflowRuntimeState`、`WorkflowWindowStatus`、`RuntimeLogEntry`、`AppErrorRecord`、`WorkflowLogicBranch`、`SAMPLE_AST`、`SAMPLE_PAYLOAD`）不受影响，保留在 `types.ts` 中
