# ADR-0001: 使用 Tokio MPSC Channel 实现 DAG 节点调度

- **状态**: 已接受
- **日期**: 2025-01-01
- **决策者**: Niu Zhihong

## 背景

Nazh 引擎需要将用户在画布上设计的工作流 DAG 部署为可执行的异步流水线。
核心需求：

- 每个节点可能涉及 I/O 阻塞（协议通信、脚本执行），不能串行运行。
- 单个节点故障（超时、panic）不能拖垮其他节点或整个运行时。
- 需要支持扇出（一个节点向多个下游广播），即 DAG 而非线性链。
- 工业场景要求"防爆性"：运行时绝不 panic。

## 决策

> 我们决定每个节点运行在独立的 Tokio 任务中，节点之间通过 `tokio::sync::mpsc` 通道传递 `WorkflowContext`。

部署时（`deploy_workflow`）：
1. 为每个节点创建一个 MPSC 通道对（sender + receiver）。
2. 根据 DAG 边关系，将上游节点的 sender 列表指向下游节点的 receiver。
3. 每个节点 spawn 一个 Tokio 任务，循环接收上下文、执行、转发。

## 可选方案

### 方案 A: Actor 框架（如 Actix）

- 优势：成熟的消息传递抽象，内置监督策略
- 劣势：引入重量级依赖；Actix 的 API 与 Tauri 的异步运行时存在兼容性问题；对于当前规模过于复杂

### 方案 B: 共享内存 + 锁

- 优势：无需通道，延迟可能更低
- 劣势：违反项目"消息传递优于共享状态"的设计原则；死锁风险高；难以实现 panic 隔离

### 方案 C: Tokio MPSC Channel（已选）

- 优势：零外部依赖（Tokio 已是必选项）；天然支持背压（通道满时自动等待）；每个任务独立，panic 不传播
- 劣势：扇出需要为每个下游维护独立的 sender clone；大量节点时通道数量增长

## 后果

### 正面影响

- 节点之间完全解耦，一个节点挂了其他节点照常运行
- 天然支持 `catch_unwind` 做 panic 隔离
- 通道的缓冲区大小可按节点独立配置（`buffer` 字段）
- 事件流和结果流也使用同一通道模式，架构一致

### 负面影响

- 扇出时需要 `clone()` 上下文，大 payload 场景可能有内存开销
- 调试时任务之间的因果关系不如同步代码直观，需要依赖 trace_id 追踪

### 风险

- 如果下游节点消费速度远低于上游生产速度，通道缓冲区可能成为瓶颈
- 缓解措施：每个节点可配置 `buffer` 大小，后续可加入背压告警
