# ADR-0015: 反应式数据引脚（Reactive PinKind）

- **状态**: 已实施（Phase 1+2+3，2026-04-30）
- **日期**: 2026-04-24（提议）/ 2026-04-30（已实施）
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-08 反应式数据引脚」；依赖 ADR-0010 Pin 系统；扩展 ADR-0014 的 `PinKind`；与 ADR-0012 工作流变量 watch channel 协同；实施设计见 `docs/specs/2026-04-30-adr-0015-reactive-data-pin-design.md`

## 背景

工业边缘对**同一份数据**存在两种访问模式：

- **Push 模式**：高频采样场景（Modbus 10Hz 采样 → 每一帧都要进历史库）
- **Pull 模式**：仪表盘场景（前端每 1s 刷新一次最新值，不需要中间丢弃的帧）

当前 DAG 是纯 Push 模式，仪表盘场景要么自己 throttle（浪费计算）、要么前端轮询一个变量（失去实时性）。ADR-0014 引入的 Data 边提供了 Pull 能力，但它是**按需拉取**——不触发下游更新。

还缺一个能力：**订阅式推送**——数据变化时自动通知订阅者，类似 FRP（函数式响应式编程）的 signal/observable 模式。UE5 Blueprint 的 Event Dispatcher 是这个方向的简化版；Rx / SolidJS signals / Svelte runes 是更完整的对应物。

### 典型场景

1. **仪表盘订阅**：温度传感器采集 10Hz，仪表盘节点订阅"最新温度"——UI 只在变化时更新，不做轮询
2. **告警订阅**：多个告警接收端（短信、邮件、Bark、钉钉）订阅同一个告警源；新增一个接收端不改动告警生成节点
3. **变量变更触发**：`vars.setpoint` 被改动时，所有 PID 控制循环立即用新值，不用等下一个 tick
4. **AI 决策广播**：AI 节点每次输出决策，多个执行节点订阅决策 topic 并同步执行

## 决策

> 我们决定把反应式语义落在 `PinKind::Reactive` 上，而不是新增 `PinType::Reactive(T)` 或独立泛型 `ReactivePin<T>` 类型。Reactive 输出引脚复用 ADR-0014 的 `OutputCache` watch 槽位：上游输出时写缓存，同时按 Exec 路径推 `ContextRef` 唤醒下游；IPC 订阅通过 `OutputCache::subscribe(pin_id)` 获取 watch receiver，再由 Tauri 推送 `workflow://reactive-update/{workflow_id}/{node_id}/{pin_id}`。

### 最终类型形态

```rust
pub enum PinKind {
    Exec,
    Data,
    Reactive,
}
```

`PinType` 不新增 Reactive 包装。数据形状仍由 `PinType::{Json, Float, ...}` 表达；求值 / 推送 / 订阅语义由 `PinKind` 表达。

### 实施偏离原提案

原提案中的 `ReactivePin<T>` 泛型和 `tokio::sync::broadcast` 在实施评审中取消：

- 引擎数据面统一是 `serde_json::Value`，泛型类型在运行时没有收益。
- RFC-0002 Phase 6 已否决 broadcast 事件总线，原因是 `Lagged` 语义会静默丢值，不适合作为工业可靠性默认路径。
- `OutputCache` 在 ADR-0014 Phase 5 已改为 `watch` channel，可同时承担最新值存储与订阅通知。
- Reactive 是 Data + Exec 的组合语义，不是新的数据类型。

### PinKind 兼容矩阵

- Reactive 输出 → 可连 Reactive / Exec / Data 输入。
- Exec 输出 → 仅可连 Exec 输入。
- Data 输出 → 仅可连 Data 输入。
- Exec 或 Data 输出 → 不可连 Reactive 输入，因为 Reactive 输入期待“有缓存 + 有推送唤醒”。

### 与工作流变量的协同

`WorkflowVariables`（ADR-0012）的每个变量天然是 ReactivePin —— `vars.set("setpoint", 26.0)` 内部 emit，所有订阅该变量的节点立即被唤醒。

## 可选方案

### 方案 A: 仅保留 Push（现状）

- 优势：无新概念
- 劣势：仪表盘场景浪费计算、轮询变量失实时性

### 方案 B: 仅保留 Pull（ADR-0014 Data 边）

- 优势：简单清晰
- 劣势：
  - "变化时立即响应"的语义需要下游频繁轮询缓存——延迟不可控
  - FRP 风格的链式反应无法自然表达

### 方案 C: Push + Pull 双模，`PinKind` 三分（已选）

- 优势：
  - 覆盖高频采样（Exec）、按需读取（Data）、变化响应（Reactive）三种工业场景
  - 作为 ADR-0010 / ADR-0014 的自然延伸——`PinKind::Reactive` 是 `PinKind` 的第三种求值语义
  - 工作流变量通过 watch channel 获得订阅能力，后续可桥接到 Reactive 输入
- 劣势：
  - PinKind 变三种——部署期校验规则更复杂
  - Runner 需要维护 Data / Exec / Reactive 三分支 dispatch
  - watch 只保留最新值，不承诺完整历史；需要完整历史的链路仍应使用 Exec

### 方案 D: 外部事件总线（Redis Pub/Sub / NATS）

- 优势：跨进程
- 劣势：
  - 引入网络依赖
  - 与"单进程部署"前提冲突（ADR-0003 已确立）
  - 反应式能力应是引擎的一等公民，而不是外部胶水

## 后果

### 正面影响

- **仪表盘场景可直接订阅**：前端通过 IPC 建立到某个 Reactive 输出 pin 的 watch 订阅，变化时即刻得到推送——零轮询
- **告警分发自然扩展**：新增告警接收端只是新增订阅者；告警源节点无需改动
- **工作流变量自动升级**：变量变更自动触发下游重算——PID setpoint 改动立即生效
- **与 AI 决策链路天然契合**：AI 节点 emit 决策 → 多个执行节点订阅 → 零延迟同步
- **FRP 风格链式组合**：`sensor.temperature → debounce → average → dashboard` 可以全部用反应式引脚串起来

### 负面影响

- 边类型从 1 种变 3 种（Exec/Data/Reactive），心智模型复杂度最高的一次调整
- watch 的"只保留最新值"语义需要向用户清楚解释——需要完整历史时不能用 Reactive 代替 Exec
- 实现要仔细处理：订阅任务生命周期、订阅者不活跃时的资源回收、订阅者崩溃时的清理

### 风险

- **风险 1：实时性 vs 完整历史的权衡不清晰**
  - 缓解：文档里明确"Reactive 只表达最新值订阅，不保证完整历史；要完整历史请用 Exec"
- **风险 2：反应式风暴**——某变量被多个节点订阅，一次变更引发级联计算雪崩
  - 缓解：Runner 在反应式唤醒路径上施加背压；配合未来 `ConcurrencyPolicy`（RFC-0002 Phase 6）
- **风险 3：测试困难**——反应式流的测试需要事件时序断言
  - 缓解：核心层用 `OutputCache::subscribe` / `watch::Receiver::changed` 测试；工作流层用 Reactive 边集成测试覆盖唤醒路径
- **风险 4：比 UE5 走得更远反而难懂**
  - 缓解：文档强调"把它当作『订阅 + 最新值』的语义糖"；初版不提 FRP 术语

## 备注

- 本 ADR 强依赖 ADR-0010（Pin 系统）和 ADR-0014（边类型分离）先落地。
- 实施前置条件：先有 Data 边（ADR-0014）把 Pull 能力跑通，再加 Reactive 边。
- 对外 IPC：前端通过 `subscribe_reactive_pin(workflow_id, node_id, pin_id)` Tauri 命令订阅 OutputCache watch receiver，可以直接驱动仪表盘组件——这是 Nazh 区别于纯 UE5 Blueprint 模型的独特卖点。
- 长期愿景：所有节点的**每个输出引脚**默认都可订阅——反应式成为一等公民，不需要显式标记为 `Reactive(T)`。但这是第二阶段目标，本 ADR 仅走到"显式标记 + 边模式选择"。

## 实施记录（2026-04-30）

- **Phase 1**：`PinKind::Reactive` 加入 Ring 0；部署期兼容矩阵扩展；Runner dispatch 改为 Data / Exec / Reactive 三分支；Reactive 输出写 `OutputCache` 并推 `ContextRef`。
- **Phase 2**：`WorkflowVariables` 增加 watch channel；Tauri 新增 `subscribe_reactive_pin`；`ReactiveUpdatePayload` 通过 `tauri-bindings` 导出；事件通道为 `workflow://reactive-update/{workflow_id}/{node_id}/{pin_id}`。
- **Phase 3**：前端 `isKindCompatible` / pin schema / validator 扩展到三分支；Reactive 端口视觉与 `reactive-update` 事件解析完成；跨语言 fixture 覆盖 `PinKind` 三值矩阵。
