# ADR-0015: 反应式数据引脚（ReactivePin）

- **状态**: 提议中
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-08 反应式数据引脚」；依赖 ADR-0010 Pin 系统（是其子类）；与 ADR-0014 执行/数据边分离配合（Data 边的拉取 + 订阅 = Reactive）；与 ADR-0012 工作流变量协同（变量变更可 emit 反应式事件）

## 背景

工业边缘对**同一份数据**存在两种访问模式：

- **Push 模式**：高频采样场景（Modbus 10Hz 采样 → 每一帧都要进历史库）
- **Pull 模式**：仪表盘场景（前端每 1s 刷新一次最新值，不需要中间丢弃的帧）

当前 DAG 是纯 Push 模式，仪表盘场景要么自己 throttle（浪费计算）、要么前端轮询一个变量（失去实时性）。ADR-0014 引入的 Data 边提供了 Pull 能力，但它是**按需拉取**——不触发下游更新。

还缺一个能力：**订阅式推送**——数据变化时自动通知订阅者，类似 FRP（函数式响应式编程）的 signal/observable 模式。UE5 Blueprint 的 Event Dispatcher 是这个方向的简化版；Rx / SolidJS signals / Svelte runes 是更完整的对应物。

### 典型场景

1. **仪表盘订阅**：温度传感器采集 10Hz，仪表盘节点订阅"最新温度"——UI 只在变化时更新，不做轮询
2. **告警订阅**：多个告警接收端（短信、邮件、Bark、钉钉）订阅同一个告警源；新增一个接收端不改动告警生成节点
3. **变量变更触发**：`vars.setpoint` 被改动时，所有 PID 控制循环立即用新值，不用等下一个 tick
4. **AI 决策广播**：AI 节点每次输出决策，多个执行节点订阅决策 topic 并同步执行

## 决策

> 我们决定在 ADR-0010 Pin 系统的基础上引入**反应式引脚**（`ReactivePin`）——一种特殊的输出引脚类型，它同时暴露两种访问面：`current()` 读当前值（Pull），`subscribe()` 获取变更流（Push）。ReactivePin 是一等类型概念，和普通 `PinType::Json` / `PinType::Float` 并列；它的数据由上游节点 `emit` 时自动广播给所有订阅者，同时写入缓存槽供 Pull 访问。

### 拟议类型

```rust
// crates/core/src/reactive.rs
pub struct ReactivePin<T: Clone + Send + Sync + 'static> {
    current: Arc<ArcSwap<Option<(DateTime<Utc>, T)>>>,
    emitter: tokio::sync::broadcast::Sender<(DateTime<Utc>, T)>,
}

impl<T: Clone + Send + Sync + 'static> ReactivePin<T> {
    pub fn emit(&self, value: T);  // 上游节点调用
    pub fn current(&self) -> Option<(DateTime<Utc>, T)>;  // Pull
    pub fn subscribe(&self) -> tokio::sync::broadcast::Receiver<(DateTime<Utc>, T)>;  // Push
}

// PinType 枚举扩展（ADR-0010）
pub enum PinType {
    // ... existing variants ...
    Reactive(Box<PinType>),  // e.g. Reactive(Box::new(PinType::Float))
}
```

### 边模式扩展

配合 ADR-0014 的 `EdgeKind`：

```rust
pub enum EdgeKind {
    Exec,         // 现有：push，触发下游 transform
    Data,         // ADR-0014：拉取缓存
    Reactive,     // 本 ADR：订阅式推送，变化时触发下游
}
```

Reactive 边比 Data 边多一件事：订阅 broadcast，上游变化时自动唤醒下游。比 Exec 边少一件事：不保证每一次 emit 都触发一次下游 transform（broadcast 可能丢弃订阅者跟不上的值）。

### 与工作流变量的协同

`WorkflowVariables`（ADR-0012）的每个变量天然是 ReactivePin —— `vars.set("setpoint", 26.0)` 内部 emit，所有订阅该变量的节点立即被唤醒。

## 可选方案

### 方案 A: 仅保留 Push（现状）

- 优势：无新概念
- 劣势：仪表盘场景浪费计算、轮询变量失实时性

### 方案 B: 仅保留 Pull（ADR-0014 Data 边）

- 优势：简单清晰
- 劣势：
  - "变化时立即响应"的语义需要下游频繁轮询缓存——延迟不可控
  - FRP 风格的链式反应无法自然表达

### 方案 C: Push + Pull 双模，边类型三分（已选）

- 优势：
  - 覆盖高频采样（Exec）、按需读取（Data）、变化响应（Reactive）三种工业场景
  - 作为 ADR-0010 Pin 系统的自然延伸——`PinType::Reactive(T)` 就是"T 的反应式版本"
  - 工作流变量无缝升级为 Reactive——现有 `vars.set` 调用点自动获得订阅能力
- 劣势：
  - 边类型变三种——部署期校验规则更复杂
  - Runner 要多维护一条"订阅唤醒"的 loop
  - broadcast 语义下允许丢值（channel lag 行为），需要文档化清晰

### 方案 D: 外部事件总线（Redis Pub/Sub / NATS）

- 优势：跨进程
- 劣势：
  - 引入网络依赖
  - 与"单进程部署"前提冲突（ADR-0003 已确立）
  - 反应式能力应是引擎的一等公民，而不是外部胶水

## 后果

### 正面影响

- **仪表盘场景可直接订阅**：前端通过 IPC 建立到某个 ReactivePin 的订阅，变化时即刻得到推送——零轮询
- **告警分发自然扩展**：新增告警接收端只是新增订阅者；告警源节点无需改动
- **工作流变量自动升级**：变量变更自动触发下游重算——PID setpoint 改动立即生效
- **与 AI 决策链路天然契合**：AI 节点 emit 决策 → 多个执行节点订阅 → 零延迟同步
- **FRP 风格链式组合**：`sensor.temperature → debounce → average → dashboard` 可以全部用反应式引脚串起来

### 负面影响

- 边类型从 1 种变 3 种（Exec/Data/Reactive），心智模型复杂度最高的一次调整
- broadcast 的"慢订阅者丢值"语义需要向用户清楚解释——不是所有用户都熟悉 FRP 概念
- 实现要仔细处理：订阅者不活跃时的资源回收、broadcast lag、订阅者崩溃时的清理

### 风险

- **风险 1：实时性 vs 丢值的权衡不清晰**
  - 缓解：每个 ReactivePin 可配置 `broadcast_capacity`；文档里明确"Reactive 不保证每次变化都到达，要完整历史请用 Exec"
- **风险 2：反应式风暴**——某变量被多个节点订阅，一次变更引发级联计算雪崩
  - 缓解：Runner 在反应式唤醒路径上施加背压；配合未来 `ConcurrencyPolicy`（RFC-0002 Phase 6）
- **风险 3：测试困难**——反应式流的测试需要事件时序断言
  - 缓解：提供 `ReactivePin::collect_n(n)` 测试辅助；CI 用 `tokio::time::pause` 控制时钟
- **风险 4：比 UE5 走得更远反而难懂**
  - 缓解：文档强调"把它当作『订阅 + 最新值』的语义糖"；初版不提 FRP 术语

## 备注

- 本 ADR 强依赖 ADR-0010（Pin 系统）和 ADR-0014（边类型分离）先落地。
- 实施前置条件：先有 Data 边（ADR-0014）把 Pull 能力跑通，再加 Reactive 边。
- 对外 IPC：前端通过新增的 `subscribe_reactive_pin(node_id, pin_id)` Tauri 命令获得 broadcast receiver，可以直接驱动仪表盘组件——这是 Nazh 区别于纯 UE5 Blueprint 模型的独特卖点。
- 长期愿景：所有节点的**每个输出引脚**默认都可订阅——反应式成为一等公民，不需要显式标记为 `Reactive(T)`。但这是第二阶段目标，本 ADR 仅走到"显式标记 + 边模式选择"。
