# ADR-0010: 为节点引入 Pin 声明系统（类型化输入输出引脚）

- **状态**: 已实施
- **日期**: 2026-04-24（决策） / 2026-04-26（Phase 1 落地）
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-01 Pin 声明系统」；是后续 ADR-0012（工作流变量）、ADR-0013（子图）、ADR-0014（执行/数据边分离）、ADR-0015（反应式引脚）的共同基础；与 ADR-0009 生命周期钩子相容（Pin 可在 `on_deploy` 做校验）

## 背景

Nazh 当前的节点数据契约是最弱形式的约定：

- `NodeTrait::transform(trace_id, payload: Value) -> NodeExecution`
- `NodeDispatch::Route(Vec<String>)` 使用字符串名指定下游端口，但"端口"并不作为一等公民在节点元数据中声明
- 边的 schema 没有类型信息——任何输出都可以连到任何输入

这在 Phase 1-5 阶段是合理的简化，但随着使用场景扩展，限制显现：

1. **部署期无法校验连接合法性**：把一个输出 `String` 的节点连到期望 `Array` 的下游，当前要等运行时 `payload` 到达才能炸。Modbus 写入尤其受影响——寄存器类型错误可能会直接写坏设备。
2. **前端画布无法精准渲染端口**：FlowGram 侧目前所有节点都是"一进一出"的视觉抽象，多端口节点（`switch` 的 N 条分支、`mqttClient` 订阅模式下的多主题输出）没法正确呈现。
3. **AI 脚本/节点生成缺少类型锚点**：`ai_complete()` 生成 Rhai 代码时没法告诉 LLM"这个节点输入必然是 `{ value: f64, unit: String }`"，只能靠自然语言描述，准确率受限。
4. **子图（未来 ADR-0013）缺少接口定义**：要把一段 DAG 封装成子图节点，必须先有"子图对外的入口引脚 / 出口引脚"的概念。
5. **反应式能力（未来 ADR-0015）无依附载体**：`ReactivePin` 需要先有 `Pin` 这个概念。

## 决策

> 我们决定在 `NodeTrait` 上引入**引脚声明**（Pin Declaration），节点通过 `input_pins()` / `output_pins()` 静态声明自己期望消费和产出的数据形状。部署期用这些声明校验边的合法性，前端渲染端口与类型着色，AI 生成脚本时以引脚类型为上下文。Pin 系统**不改变**数据平面的传输形式（仍然是 `serde_json::Value`），只在 Trait 元数据与 DAG 校验层引入类型约束。

### 拟议 Trait 扩展

```rust
// crates/core/src/node.rs
#[async_trait]
pub trait NodeTrait: Send + Sync {
    fn id(&self) -> &str;
    fn kind(&self) -> &'static str;

    /// 静态声明节点的输入引脚。默认返回单引脚 `{ id: "in", type: Any }`。
    fn input_pins(&self) -> Vec<PinDefinition> {
        vec![PinDefinition::default_input()]
    }

    /// 静态声明节点的输出引脚。默认返回单引脚 `{ id: "out", type: Any }`。
    fn output_pins(&self) -> Vec<PinDefinition> {
        vec![PinDefinition::default_output()]
    }

    async fn transform(&self, trace_id: Uuid, payload: Value)
        -> Result<NodeExecution, EngineError>;

    // on_deploy / on_undeploy 见 ADR-0009
}

pub struct PinDefinition {
    pub id: String,              // 引脚稳定标识，例如 "in" / "temperature" / "error"
    pub label: String,           // 前端展示名
    pub pin_type: PinType,
    pub direction: PinDirection, // Input | Output
    pub required: bool,          // 输入：是否必须有上游；输出：是否必然触发
    pub description: Option<String>, // 给 AI / 文档用
}

pub enum PinType {
    Any,                        // 兼容所有
    Bool,
    Integer,
    Float,
    String,
    Json(Option<JsonSchema>),   // 可选附带 JSON Schema 做结构约束
    Binary,
    Array(Box<PinType>),
    Custom(&'static str),       // 命名类型，例如 "modbus-register"
}
```

### 部署期校验规则

DAG 中每条边 `from_node.output_pin → to_node.input_pin` 部署时执行：

- `to_pin.pin_type == Any` → 通过（Any 吃一切）
- `from_pin.pin_type == Any` → 通过（上游承诺能产出目标类型）
- 类型精确相等 → 通过
- `Array(Any)` 兼容 `Array(T)` → 通过
- `Json(None)` 兼容 `Json(schema)` → 通过
- 否则 → `EngineError::IncompatiblePinTypes { from, to, expected, actual }`

`Custom("modbus-register")` 必须精确匹配，给协议特定类型留逃生口。

## 可选方案

### 方案 A: 维持无类型（现状）

- 优势：零改动；最大灵活性
- 劣势：前面列出的 5 条限制持续存在；节点越多问题越大

### 方案 B: 运行时类型校验（节点自行检查 payload）

```rust
async fn transform(&self, ..., payload: Value) -> Result<...> {
    let temp: f64 = payload.get("temp").and_then(Value::as_f64)
        .ok_or(EngineError::type_mismatch("temp", "f64"))?;
    ...
}
```

- 优势：无需改 Trait；每个节点自证其责
- 劣势：错误延迟到运行时；部署期 DAG 合法性仍无从校验；AI 生成仍无类型锚点

### 方案 C: Pin 声明系统（已选）

- 优势：
  - 部署期校验 → 早失败
  - 前端端口渲染、类型着色、AI 生成上下文——三个痛点一起解决
  - 与后续 ADR 兼容（子图、反应式、执行/数据边）
  - `PinType::Any` 作为逃生阀，不强制老节点立即迁移
- 劣势：
  - Trait 变胖；每个节点要声明引脚
  - 默认 `Any` 入/出意味着新手上来可以先不写 pin，但真正受益的场景（类型校验）需要节点作者主动投入

### 方案 D: 通过 Rust 泛型做静态类型（`NodeTrait<I, O>`）

```rust
trait Node<I: DeserializeOwned, O: Serialize> {
    async fn transform(&self, input: I) -> Result<O>;
}
```

- 优势：编译期完全类型安全
- 劣势：
  - 节点无法装进同质 `Arc<dyn NodeTrait>` 容器（`dyn` 需擦除泛型）
  - 动态 DAG（运行时从 JSON 构建）无法享受这个编译期保证
  - Rhai / WASM 动态节点根本无法表达
- 结论：不适用

## 后果

### 正面影响

- **部署期 DAG 校验**：Modbus 写错寄存器类型、`switch` 分支接到错误类型下游等错误，在 `deploy_workflow` 阶段就被拒绝，不会拖到采集循环里炸设备
- **前端画布能渲染多端口节点**：`switch` 的 N 条分支、`if` 的 `true`/`false` 两端、未来子图的输入输出——都有一等公民表示
- **AI 自动化质量提升**：给 `ai_complete()` 的提示词可以携带节点的 pin schema，LLM 生成脚本更精准
- **节点能力的对外契约稳定**：作者一眼看到 `input_pins` 就知道自己契约是什么；reviewer 改 `transform` 时有"改动是否符合 pin 声明"的明确验证点
- **与 ts-rs 契约一脉相承**：`PinDefinition` 本身可 `#[derive(TS)]`，前端拿到节点类型列表的同时获得端口 schema

### 负面影响

- Trait 扩展：每个已有节点在迁移期间都要补充 `input_pins` / `output_pins`（默认实现可以先用 `Any` 保兼容）
- DAG 校验逻辑引入，部署路径复杂度增加
- `PinType` 枚举本身会随着节点生态扩展——每加一种 `Custom` 类型都要审视是否该提升为一等成员

### 风险

- **风险 1：类型系统过严 → 迁移痛**
  - 缓解：默认 `Any` + 渐进开启；存量节点保持 `Any`，新节点按需收紧
- **风险 2：类型系统过松 → 没价值**
  - 缓解：`Json(Some(schema))` 和 `Custom` 是"收紧通道"，协议类节点必须用具体类型声明
- **风险 3：JSON Schema 依赖带来 footprint**
  - 缓解：`Json(None)` 是默认、仅必要时开启 schema；把 schema crate 选择推迟到 ADR-0010 实施计划阶段

## 备注

- 本 ADR 仅定义**决策**与**接口形状**。实施计划见 `docs/plans/2026-04-26-pin-declaration-system.md`。
- 建议在 ADR-0009 生命周期钩子落地**之后**再启动 Pin 系统——避免两次核心 Trait 变更叠加带来的迁移风险。
- 实施顺序建议：先改 `NodeTrait` + `WorkflowGraph` 校验器，用 `Any` 默认让所有节点通过；再从 `switch` / `if` / `loop` 这类结构节点开始补具体 pin（它们最先受益）；协议 IO 节点随后。

**实施期间的决策偏离 ADR 草稿：**
1. **`PinType::Json` 不带 schema payload**——避免 Ring 0 引入 `schemars`/`jsonschema` 依赖；结构校验留待未来独立 ADR。
2. **`PinType::Custom(&'static str)` 改为 `Custom { name: String }`**——`&'static str` 走不通 serde JSON round-trip。
3. **`PinType` 用 `#[serde(tag = "kind", rename_all = "lowercase")]`**——比 ts-rs 默认 `{ Array: ... }` map 形式对前端友好，对递归 `Box<PinType>` 也友好。
4. **`EngineError::IncompatiblePinTypes` 合并 `from_node` + `from_pin` 为单一 `from: "node.pin"` 字段**——保持最大变体在 `clippy::result_large_err` 128 字节阈值内（结构化诊断信息无损）。

**Phase 2 范围（独立 PR）：**
- Phase 2：FlowGram 画布按 `output_pins()` 渲染多端口、类型着色、IPC `describe_node_pins` 命令
- Phase 2 **两层校验防御**（2026-04-26 范围扩充）：
  1. **UI 第一层（连接期拦截）**：FlowGram `onConnect` 钩子读取本地缓存的 pin schema，在用户拖边时即时拒绝不兼容连接。前端用 TS 实现的 `isCompatibleWith` 必须与 Rust `PinType::is_compatible_with` 严格一致——通过共享的兼容矩阵 fixture（`tests/fixtures/pin_compat_matrix.jsonc`，由 Phase 3 Task 5 Step 0 产出）作为单一真值源，前后端各自跑合约测试防止漂移。
  2. **后端 backstop（部署期拦截）**：保留 Phase 1 已落地的 `validate_pin_compatibility` 在 `deploy_workflow_with_ai` 阶段 0.5 的硬关卡。覆盖 UI 绕过场景：用户手改工程文件 / 导入第三方 JSON / 前端 bug / 缓存过期 / AI 生成的图直接 deploy。两层配合即 *defense in depth*——UI 层是 UX 加速，后端层是正确性保证，不可互相替代。
- JSON Schema：当未来 `Custom` 类型 ≥6 且 ≥2 个需要结构校验时，触发新 ADR 评估 `schemars` 引入

### Phase 4 落地记录（2026-04-27）

Phase 2 完成后列出的 4 个候选项中，**2 项已实施、2 项明确 defer**：

**已实施：**
- **Item 3：pin tooltip**（`796e6fb`）——分支节点（`if`/`switch`/`tryCatch`/`loop`）的输出端口加 HTML `title` 属性，hover 显示端口标签 + 类型 + required + description。新增 `formatPinType` / `getPortTooltip` helper 复用于其他需要展示 pin 信息的场景。
- **Item 4：AI 生成 prompt 携带 pin schema**（`f294de1`）——`script-generation.ts` 从 pin schema 缓存读取当前节点 + 上下游节点的 pin 摘要（如 `out: json` / `in: json (required)`），格式化进 prompt；`SYSTEM_PROMPT` 扩展类型语义解释，让 LLM 理解 `'json'` / `'array<T>'` / `'custom(name)'` 等类型对 payload 形态的约束。

**已 defer（带触发条件）：**
- **Item 1：协议节点端口着色**——FlowGram 默认端口由内部组件渲染，外部无法直接给 DOM 注入 `data-port-pin-type` 属性。技术路径需要把所有协议节点切换到 `useDynamicPort: true` 并自渲染端口 DOM——重构成本高，且 Phase 3 后协议节点都是 `Json/Any` 着色区分有限。**触发条件**：未来 Phase 4+ 引入 `Custom` 类型让协议节点端口产生颜色多样性时再做；或前端框架升级改变端口渲染机制时一并处理。
- **Item 2：`Custom` 类型 + 配套 row-formatter 节点**——`Custom("sql-row")` / `Custom("modbus-register")` 等命名类型必须**同时**引入"产生 Custom 输出的节点"才能避免"消费者孤岛"（参见 Phase 3 决策偏离记录第 1 条）。**触发条件**：当出现 ≥2 个真实场景需要协议级类型隔离（如 sqlWriter 拒绝接受非 row-formatter 输出）时启动新 ADR。

**两层防御之外的额外语义增强：**
- pin tooltip 让用户在画布层面就能看到端口的预期类型（之前要打开节点设置面板才能猜）
- AI prompt 携带 pin schema 让 LLM 生成的脚本与节点契约对齐——这是从"自由发挥"到"按 schema 生成"的实际升级

**Phase 4+ 候选项（持续观察，未列入近期路线）：**
- 协议节点端口 DOM 改造（依赖 useDynamicPort 重构）
- `Custom` 类型 + row-formatter 节点（依赖新 ADR）
- 自定义 tooltip 组件（替代 HTML title 属性，支持富文本 / 跳转链接）
- pin legend 面板（画布右侧或顶部展示当前图所有用到的 PinType + 颜色对照）
