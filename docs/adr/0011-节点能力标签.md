# ADR-0011: 为节点引入能力标签（NodeCapabilities Bitflags）

- **状态**: 已实施
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-04 节点能力标签」；与 ADR-0009 生命周期钩子、ADR-0010 Pin 系统独立但互补

## 背景

Nazh 当前把所有节点同质看待——都在 Tokio 任务里跑 `transform`，都受同一套超时/panic 保护。但实际节点特性差异很大：

| 节点 | 特性 |
|------|------|
| `if` / `switch` | 纯计算，无 I/O，微秒级延迟 |
| `code`（Rhai） | 纯计算 + 用户脚本，微秒到毫秒 |
| `httpClient` | 网络 I/O，百毫秒级，异步但可能阻塞 |
| `modbusRead` | 设备 I/O，可能阻塞 TCP 读取 |
| `sqlWriter` | 文件/数据库 I/O，同步阻塞（bundled rusqlite） |
| `mqttClient`（订阅） | 触发器，长连接，外部时钟驱动 |
| `timer` | 触发器，内部时钟驱动 |
| `loop` / `switch` | 多输出/分支路由 |

引擎目前对这些差异"视而不见"会带来几个问题：

1. **调度效率损失**：`sqlWriter` 类同步阻塞节点跑在普通 Tokio 任务里，理论上应该走 `spawn_blocking` 避免饿死 runtime
2. **纯计算节点无法被缓存/去重**：`if` 对同一输入永远得同一输出，Runner 可以做结果缓存，但需要知道"这是纯节点"
3. **前端画布缺少视觉分级**：触发器/计算/IO/分支四类节点在图上应该有不同的颜色/形状/图标，但当前前端要靠 `kind` 字符串硬编码判断
4. **监控指标无法分桶**：采集"网络 IO 节点的 p99 延迟"、"设备 IO 节点的失败率"需要标签支撑
5. **调度策略（如 ConcurrencyPolicy）无从下手**：后续要做"IO 节点并发上限 64、计算节点无上限"这类策略，没有标签就无法区分

## 决策

> 我们决定在 `NodeTrait` 上增加 `capabilities() -> NodeCapabilities` 方法，以 bitflags 形式声明节点的特性。Runner 依据标签做调度决策（`BLOCKING` 走 `spawn_blocking`、`PURE` 启用缓存等），前端按标签渲染视觉分组，监控按标签分桶。

### 拟议标签集

```rust
// crates/core/src/node.rs
bitflags::bitflags! {
    pub struct NodeCapabilities: u32 {
        /// 纯计算：相同输入必得相同输出、无副作用
        const PURE           = 0b0000_0001;
        /// 网络 I/O：HTTP / gRPC / MQTT / Kafka 等
        const NETWORK_IO     = 0b0000_0010;
        /// 文件/数据库 I/O：sqlite / 文件读写
        const FILE_IO        = 0b0000_0100;
        /// 设备 I/O：Modbus / 串口 / OPC-UA / GPIO
        const DEVICE_IO      = 0b0000_1000;
        /// 触发器：由外部事件/时钟驱动，不靠上游 payload
        const TRIGGER        = 0b0001_0000;
        /// 分支节点：按条件路由到一部分下游
        const BRANCHING      = 0b0010_0000;
        /// 多输出：一次 transform 产生多个 NodeOutput
        const MULTI_OUTPUT   = 0b0100_0000;
        /// 同步阻塞：transform 内部可能用阻塞 API，需 spawn_blocking
        const BLOCKING       = 0b1000_0000;
    }
}

#[async_trait]
pub trait NodeTrait: Send + Sync {
    // ...
    fn capabilities(&self) -> NodeCapabilities {
        NodeCapabilities::empty()
    }
}
```

### Runner 的利用方式（渐进）

- **第一阶段**：仅用于前端渲染和监控标签（零风险，纯展示）
- **第二阶段**：`BLOCKING` 节点自动走 `tokio::task::spawn_blocking`
- **第三阶段**：`PURE` 节点启用 `(input_hash) → cached_output` 缓存（需与 ADR-0014 数据边分离配合）
- **第四阶段**：配合未来 `ConcurrencyPolicy`，按标签类别设并发上限

## 可选方案

### 方案 A: 不加标签（现状）

- 优势：零改动
- 劣势：前述 5 条问题持续；调度策略永远做不精

### 方案 B: 用 Trait 方法而非 bitflags

```rust
fn is_pure(&self) -> bool { false }
fn is_blocking(&self) -> bool { false }
fn is_trigger(&self) -> bool { false }
// ...
```

- 优势：更 Rusty
- 劣势：
  - 方法数量线性增长；每加一个标签就改 Trait
  - 组合逻辑麻烦：`if node.is_pure() && !node.is_blocking() { ... }` 比 `caps.contains(PURE) && !caps.contains(BLOCKING)` 冗长
  - 无法一次性序列化/发给前端做批量展示

### 方案 C: 独立 bitflags（已选）

- 优势：
  - 组合自由，`NETWORK_IO | BLOCKING` 精准描述"阻塞式 HTTP 客户端"
  - `bitflags` crate 零成本抽象，序列化为 `u32` 对前端友好
  - 加标签只改 bitflags 定义，不动 Trait
- 劣势：
  - 标签的"语义定义"散落在注释里，容易漂移（必须在 ADR 里锁死）
  - 标签与实际行为可能脱节（`PURE` 声明了实际有副作用）——需要测试手段辅助验证

### 方案 D: 类型级标签（phantom types）

- 优势：编译期分类
- 劣势：`dyn NodeTrait` 无法承载 phantom；与动态 DAG 冲突（同 ADR-0010 方案 D）

## 后果

### 正面影响

- **调度效率可优化**：`sqlWriter` 走 `spawn_blocking` 后不阻塞 Tokio worker pool
- **纯节点缓存为后续铺路**：`if`/`switch` 逻辑节点在同输入时可复用结果（需 ADR-0014 数据边配合）
- **前端视觉分组立即可用**：按 `TRIGGER` 上方布局、`DEVICE_IO` 橙色边框等规则是纯 CSS 变更
- **监控分桶**：可观测性仪表盘里按 `NETWORK_IO` 聚合 p99，按 `DEVICE_IO` 聚合失败率
- **节点作者自证契约**：声明 `PURE` 就等于承诺"我不可能有副作用"，审查时多一道锁

### 负面影响

- 每个已有节点都要补 `capabilities()`——默认空集合可以过，但发挥价值需要作者投入一次性工作
- bitflags 语义必须在本 ADR 及未来版本里严格定义，否则会漂移

### 风险

- **标签撒谎**：作者错误地把有副作用节点标为 `PURE`，导致缓存失效路径产生脏数据
  - 缓解：写一套"`PURE` 节点随机 fuzzing 同输入验证输出恒等"的 CI 测试；review PR 时显式检查标签
- **组合爆炸**：未来标签数量超过 32 位
  - 缓解：如果真到那一步，迁移到 `u64` 或拆分为 `NodeCapabilities` + `NodeSchedulingHints` 两套 bitflags

## 备注

- 建议作为**轻量首发 ADR**先落地——它改动小、风险低、收益明确（尤其是前端视觉改进和 `spawn_blocking` 调度优化）。
- 可以与 ADR-0010 Pin 系统独立并行，二者没有耦合。
- 长期看，`PURE` + Pin 系统 + 数据边（ADR-0014）三者组合才能做真正的计算节点缓存；本 ADR 只是其中一块积木。

## 实施记录

**首发落地（2026-04-24）**：第一阶段（前端渲染 + 监控标签）已落地；第二至四阶段（`spawn_blocking` / `PURE` 缓存 / `ConcurrencyPolicy`）待后续 Runner 演进再启用。

**实施偏离**：上文"决策"段示意在 `NodeTrait` 上加 `fn capabilities(&self) -> NodeCapabilities` 方法。落地 code review 时发现 trait 方法零消费者（所有查询走注册表），11 个 override 都是类型级值的复读——遂移除，**只保留注册表中的类型级声明**。未来若需要实例级精化，推荐新增 `instance_capabilities(base) -> NodeCapabilities`，不要恢复旧方法。详见 `crates/core/AGENTS.md` 的"为什么 `NodeTrait` 没有 `capabilities()` 方法"。

决策本身**就此定版**——后续实施中出现的位含义细化、内置节点标签分配、修改 checklist 都属于"约定"，
