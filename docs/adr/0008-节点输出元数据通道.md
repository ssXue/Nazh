# ADR-0008: 节点执行元数据走事件通道，不污染业务 payload

- **状态**: 已接受
- **日期**: 2026-04-17
- **决策者**: Niu Zhihong
- **关联**: 实施提交 `b084a47`；对应内部提案「提案-02 Context 元数据分离」

## 背景

在 2026-04-16 的架构 Review 中，发现一条长期存在的坏味道：节点实现会往业务 payload 里塞带下划线前缀的执行元数据字段，例如：

- `_timer: {...}`（定时器节点的触发信息）
- `_http: {...}`（HTTP 客户端节点的请求/响应元数据）
- `_modbus: {...}`（Modbus 读取节点的寄存器地址与采样信息）
- `_connection: {...}`（连接节点的借出状态）
- `_loop: {...}`（循环节点的迭代上下文）
- `_error: {...}`（异常路由信息）

这带来几个具体问题：

1. **Rhai 脚本体验混乱**：脚本要处理业务数据，却需要从同一个对象中区分哪些是业务字段、哪些是"引擎私货"。一不小心就把 `_timer` 序列化回上游，或误以为业务字段。
2. **数据沿袭（data lineage）信息与业务数据混在一起**：审计日志需要区分"这条数据的业务内容"和"它流经了哪些节点、每步做了什么"。混在一起导致二者都残缺。
3. **前端展示耦合**：画布节点悬浮卡上要显示"最近一次执行的采样时间 / 连接 ID"等，现在必须从 payload 里摸索。
4. **节点作者契约不清**：什么样的键该加下划线、加哪些前缀、何时必须清理——全靠口口相传。

## 决策

> 我们决定**保持 `WorkflowContext` 极简**（仅三字段：`trace_id`、`timestamp`、`payload`），把节点执行元数据**完全挪出 payload、不进入数据通道**，改由 `NodeOutput.metadata` 字段返回，Runner 将其合并进 `ExecutionEvent::Completed` 事件并沿事件通道推送到前端。同时将 `NodeTrait::execute` 更名为 `transform`，以签名语义强化"节点是纯变换"这一契约。

### 最终形态

```rust
// crates/core/src/node.rs
pub struct NodeOutput {
    pub payload: Value,                  // 业务数据，写入 DataStore
    pub metadata: Map<String, Value>,    // 执行元数据，走事件通道
    pub dispatch: NodeDispatch,
}

#[async_trait]
pub trait NodeTrait: Send + Sync {
    fn id(&self) -> &str;
    fn kind(&self) -> &'static str;
    async fn transform(&self, trace_id: Uuid, payload: Value)
        -> Result<NodeExecution, EngineError>;
}
```

### 元数据键命名约定

- 使用**不带下划线前缀**的名称：`"timer"`、`"http"`、`"modbus"`、`"serial"`、`"sql_writer"`、`"debug_console"`、`"connection"`、`"bark"` 等。
- 一个节点返回的元数据键应反映该节点的"kind"（类型 ID），便于前端按节点类型分流渲染。
- 仅**路由语义**字段（`_loop`、`_error`）仍允许保留在 payload 中——它们参与下游分支决策，是业务语义的一部分。

## 可选方案

### 方案 A: 维持现状，节点继续往 payload 写下划线键

- 优势：改动量为零
- 劣势：上述 4 条痛点持续累积；节点越多污染越严重；和 AI 自动生成脚本的愿景冲突（脚本作者难以稳定推理 payload 形状）

### 方案 B: 在 `WorkflowContext` 中新增 `metadata: Map<String, Value>` 字段（Review 原提案）

```rust
pub struct WorkflowContext {
    pub trace_id: Uuid,
    pub timestamp: DateTime<Utc>,
    pub payload: Value,
    pub metadata: Map<String, Value>, // 新增
}
```

- 优势：元数据随数据包流动，下游节点可直接读到上游元数据
- 劣势：
  - 数据通道与事件通道职责仍然混合——元数据本质上是"审计/观测"信息，跟随业务数据流意义不大
  - `WorkflowContext` 在 IPC 边界（ts-rs 导出给前端）承载额外字段，前端类型复杂度上升
  - 下游"读上游 metadata 做决策"是反模式，鼓励了隐式耦合

### 方案 C: 独立 metadata 通道 + 事件聚合（已选）

- Runner 每完成一次 `transform`，把返回的 `metadata` 合并进 `ExecutionEvent::Completed`，前端订阅事件流获取
- 优势：
  - **数据平面纯净**：payload 只装业务数据，走 DataStore
  - **事件平面承担观测职责**：元数据、追踪、审计天然属于事件
  - `WorkflowContext` 保持三字段极简，IPC 契约稳定
  - 下游节点无法读到上游 metadata——这是特性不是缺陷：强制"业务决策只基于业务数据"
- 劣势：
  - 真的需要"上游元数据驱动下游逻辑"时（罕见）必须把该信息显式写回 payload，作者要有自觉
  - 需要同时改 Trait 签名、Runner、所有节点实现，一次性改动较大

### 方案 D: 单独拆一个 `ExecutionMetadata` 结构随 context 流转

- 类似方案 B 但更结构化（带 `source_node`、`node_path`、`annotations`）
- 优势：数据沿袭可追溯
- 劣势：仍然是"数据平面承担事件平面职责"；过度设计，当前观测需求事件通道已经够用

## 后果

### 正面影响

- **Rhai 脚本看到的 payload 就是业务数据**，不用再过滤下划线字段——直接受益的是正在开发的 AI 脚本生成能力（`ai_complete()` 返回结构化数据更可信）
- **`transform` 签名自带契约感**：作者一看就知道节点不该有隐式状态、不该读 DataStore
- **Control Plane / Data Plane 分离更彻底**：data 走 DataStore（ContextRef），metadata 走事件通道（broadcast），二者完全解耦
- **前端可以按节点类型精准渲染元数据**（每个 kind 有自己的 metadata 键空间）
- **审计与观测基础打牢**：后续做 edge-level 观测、分布式追踪都沿着事件通道自然延伸

### 负面影响

- 一次性改动面广：Trait 签名、Runner、所有节点、所有单测都要配合修改（已在 `b084a47` 完成）
- 节点作者如果确实需要下游读上游元数据，必须**显式**往 payload 里写——略增心智负担，但换来契约清晰
- 元数据只在 `Completed` 事件中可见，节点失败（`Failed` 事件）时收集到的元数据会略少——需要节点作者在失败路径上也调用 `with_metadata`

### 风险

- 如果未来出现"元数据参与下游执行决策"的合理场景（例如"根据上游连接的延迟值决定是否重试"），会发现元数据不在数据通道里，必须显式转写
- **缓解措施**：此类场景本应建模为业务数据（延迟值就是数据），不是元数据。审查节点 PR 时严格区分二者即可。
