# ADR-0004: 统一 Pipeline 和 Graph 的执行事件模型

- **状态**: 已接受
- **日期**: 2026-04-09
- **决策者**: Niu Zhihong
- **关联**: ADR-0001

## 背景

引擎目前有两套独立的执行事件类型：

- `WorkflowEvent`（4 个变体：`NodeStarted` / `NodeCompleted` / `NodeFailed` / `WorkflowOutput`）
- `PipelineEvent`（4 个变体：`StageStarted` / `StageCompleted` / `StageFailed` / `PipelineCompleted`）

两者的 Started / Completed / Failed 模式完全相同，只是字段名不同（`node_id` vs `stage`）。这导致：

1. `graph/runner.rs` 和 `pipeline/runner.rs` 各自维护独立的 `emit_event` / `emit_failure` 辅助函数，逻辑完全重复。
2. Tauri 前端需要分别注册两种事件的监听器，增加对接复杂度。
3. 新增执行模式（如未来的批处理流水线）需要再定义一套事件类型。

约束条件：

- 事件需要 `Serialize + Deserialize`，因为要通过 Tauri IPC 发往前端。
- 前端 TypeScript 类型需要同步更新。
- Graph 有"叶节点产出结果"的独有语义，Pipeline 有"整条流水线完成"的独有语义。

## 决策

> 我们决定用统一的 `ExecutionEvent` 替代 `WorkflowEvent` 和 `PipelineEvent`，因为两者的语义模型几乎一致，统一后可消除重复并简化前端对接。

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub enum ExecutionEvent {
    /// 阶段/节点开始执行。
    Started   { stage: String, trace_id: Uuid },
    /// 阶段/节点执行完成。
    Completed { stage: String, trace_id: Uuid },
    /// 阶段/节点执行失败。
    Failed    { stage: String, trace_id: Uuid, error: String },
    /// 叶节点产出最终结果（仅 Graph 模式下发出）。
    Output    { stage: String, trace_id: Uuid },
    /// 整条流水线执行完毕（仅 Pipeline 模式下发出）。
    Finished  { trace_id: Uuid },
}
```

迁移范围：

- `graph/types.rs`：删除 `WorkflowEvent`，改用 `ExecutionEvent`
- `pipeline/types.rs`：删除 `PipelineEvent`，改用 `ExecutionEvent`
- 两个 runner 的 `emit_event` / `emit_failure` 可提取为共享函数
- Tauri 桥接层（`src-tauri/src/lib.rs`）：更新事件监听
- 前端 TypeScript 类型（`web/src/types.ts`）：同步更新

## 可选方案

### 方案 A: 泛型事件 `ExecutionEvent<S>`

```rust
enum ExecutionEvent<S: Display> {
    Started { stage: S, trace_id: Uuid },
    // ...
}
```

- 优势：允许 Graph 和 Pipeline 用不同类型标识阶段
- 劣势：泛型传播到 Handle、Runner、前端接口，增加整体复杂度；实际上两边都用 `String` 标识

### 方案 B: 统一具体类型 `ExecutionEvent`（已选）

- 优势：零泛型传播；`stage: String` 足以标识节点/阶段；可直接序列化发往前端
- 劣势：`Output` 和 `Finished` 分属不同子系统的语义，需要文档说明适用场景

### 方案 C: 保持两套事件类型不变

- 优势：无迁移成本
- 劣势：重复持续累积；前端维护两套监听逻辑；每新增执行模式需要再定义事件类型

## 后果

### 正面影响

- 两个 runner 的 `emit_event` / `emit_failure` 可合并为共享函数
- Tauri 前端只需一个事件 listener，统一事件处理逻辑
- 未来新增执行模式自动复用同一事件类型

### 负面影响

- `WorkflowEvent` 和 `PipelineEvent` 的所有使用方需要迁移（两个 runner、两个 handle、Tauri 桥接层、前端 TypeScript 类型、集成测试）
- `Output` 变体在 Pipeline 场景中不会出现，`Finished` 在 Graph 场景中不会出现，可能困惑新开发者

### 风险

- Tauri 前端已在监听 `WorkflowEvent` 格式的事件名称。需要同步更新前端代码。
- 缓解措施：在同一 PR 中完成后端 + 前端迁移，避免中间态不一致。
