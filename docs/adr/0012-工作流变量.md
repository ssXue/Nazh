# ADR-0012: 工作流级共享变量（WorkflowVariables）

- **状态**: 已实施（Phase 1+2，2026-04-27）
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-05 工作流变量」；依赖 ADR-0009 生命周期钩子（变量在 `on_deploy` 初始化）；与 DataStore（ADR-0008 已提及）互补但定位不同

## 背景

Nazh 目前的数据流动严格沿 DAG 边传播——节点只能看到上游 `payload`。这在多数场景够用，但工业边缘有一类高频需求无法表达：

1. **PID 控制**：被控变量的 `setpoint`（期望值）由一个节点设定，多个控制循环（温度、湿度、压力）同时读取；setpoint 可能随时被运营人员修改
2. **累加器 / 计数器**：统计"本小时告警次数"、"已处理消息总数"——跨多次 trigger 维持状态
3. **模式开关**：全局"手动/自动模式"开关，影响多条支路的行为
4. **动态配置**：AI 决策节点下发"本轮采样频率"，多个 Modbus 根据此调整自己的轮询周期
5. **跨循环聚合**：窗口聚合（平均值、最大值）需要跨 trigger 累积历史数据

当前变通方案是"把状态塞进 payload 里绕一圈传给需要的节点"，但 DAG 不是一个图灵完备的数据流模型——没有回边，状态很难回到源头。另一个变通是让节点自己维护 `Arc<Mutex<...>>` 作为内部状态，但这属于节点私有、不在工作流层面可观测，也无法被 Rhai 脚本访问。

DataStore（ADR-0008 提及）也不解决问题：DataStore 是"按 DataId 索引的不可变消息体"仓库，供 ContextRef 指针引用，设计目标是减少 payload clone，不是共享可变状态。

## 决策

> 我们决定引入**工作流级变量存储** `WorkflowVariables`：每个已部署工作流持有一个 `DashMap<String, TypedVariable>`，生存周期与部署同步（部署时创建、撤销时销毁）。节点通过注入的句柄读写变量；Rhai 脚本通过 `vars.get("setpoint")` / `vars.set("counter", 42)` 访问；前端可在"运行时视图"实时查看变量当前值。

### 拟议类型

```rust
// 新增 crates/core/src/variables.rs

pub struct WorkflowVariables {
    inner: Arc<DashMap<String, TypedVariable>>,
}

pub struct TypedVariable {
    pub value: Value,
    pub variable_type: PinType,       // 复用 ADR-0010 的 PinType
    pub updated_at: DateTime<Utc>,
    pub updated_by: Option<String>,   // 最后写入的 node_id
}

impl WorkflowVariables {
    pub fn get(&self, key: &str) -> Option<TypedVariable>;
    pub fn set(&self, key: &str, value: Value, node_id: &str)
        -> Result<(), EngineError>;   // 类型不匹配则拒写
    pub fn compare_and_swap(&self, key: &str, expected: &Value, new: Value)
        -> Result<bool, EngineError>;
    pub fn snapshot(&self) -> HashMap<String, TypedVariable>; // 供前端读取
}
```

变量的**类型声明**作为工作流定义的一部分：

```json
{
  "nodes": [ ... ],
  "edges": [ ... ],
  "variables": {
    "setpoint":    { "type": "Float",   "initial": 25.0 },
    "mode":        { "type": "String",  "initial": "auto" },
    "alert_count": { "type": "Integer", "initial": 0 }
  }
}
```

### 传递到节点与 Rhai

```rust
// NodeLifecycleContext（ADR-0009）新增字段
pub struct NodeLifecycleContext {
    pub resources: SharedResources,
    pub handle: NodeHandle,
    pub shutdown: CancellationToken,
    pub variables: WorkflowVariables,  // 新增
}
```

Rhai 脚本通过 `scripting` crate 注入的全局对象访问：

```rhai
let current = vars.get("setpoint");
vars.set("alert_count", vars.get("alert_count") + 1);
```

## 可选方案

### 方案 A: 让节点自己持有 `Arc<Mutex<...>>` 内部状态

- 优势：改动最小
- 劣势：
  - 状态不可观测、不可持久化、不可跨节点共享
  - 每种需求（计数器/setpoint/模式）都要一个独立节点——生态爆炸
  - Rhai 脚本无法访问

### 方案 B: 复用 DataStore 作为共享可变状态

- 优势：不引入新概念
- 劣势：
  - DataStore 是"消息体仓库"，`DataId` 由 Runner 生成、生命周期随 ContextRef 引用计数
  - 没有"命名键"概念；没有类型约束；TTL GC 会误清掉常驻变量
  - 职责扭曲：控制平面（配置/状态）和数据平面（消息）混在一起，违反 ADR-0008 建立的分离契约

### 方案 C: 独立 `WorkflowVariables` 存储（已选）

- 优势：
  - 语义清晰：**控制平面的共享可变状态**，与 DataStore（数据平面）正交
  - 类型化：复用 ADR-0010 的 `PinType`，写入时类型不匹配被拒
  - 可观测：前端能实时看到所有变量当前值
  - Rhai 友好：一行代码访问
  - 生命周期与部署对齐：撤销工作流 = 释放变量
- 劣势：
  - 引入全局可变状态，理论上会制造"隐式耦合"（节点 A 改了变量影响节点 B）
  - 并发写入需要仔细设计（提供 `compare_and_swap` 基本原子语义）
  - 前端渲染变量面板是额外工作

### 方案 D: 接外部 KV 存储（Redis / etcd）

- 优势：持久化、跨进程
- 劣势：边缘设备多一跳网络开销；违反"单进程部署"前提；对当前需求过度设计

## 后果

### 正面影响

- **PID 控制、累加器、模式开关**等经典工业逻辑可在单 DAG 内表达，无需变通
- **运营可视化**：前端运行时视图可以显示"当前 setpoint = 25.3°C、alert_count = 142"——极大改善可观测性
- **Rhai 脚本表达力提升**：脚本不再是"无状态 payload 变换"，而是可以做有状态决策
- **配合 ADR-0009 生命周期钩子**：`on_deploy` 初始化变量初值、`on_undeploy` 做持久化快照（如果将来需要）
- **AI 决策自然落地**：AI 节点把决策结果写入 `vars.decision`，其他节点读取——AI 与确定性逻辑通过变量解耦

### 负面影响

- 引入"DAG 之外的信息通路"——调试时需要理解"这个值从哪里来"不再只是"沿边回溯上游"
- 变量面板的前端工作量不小
- 类型系统依赖 ADR-0010 Pin 的 `PinType`，两个 ADR 耦合度高

### 风险

- **风险 1：并发写入导致脏数据**
  - 缓解：`DashMap` 本身 thread-safe；提供 `compare_and_swap` 支持原子递增；复杂场景鼓励用 `Arc<Mutex<T>>` 模式
- **风险 2：变量成为"全局 god object"**
  - 缓解：部署时强制声明所有变量及类型；不允许运行时动态创建；review 时关注变量数量，超过 ~20 个视为架构 smell
- **风险 3：持久化需求出现后 schema 迁移痛苦**
  - 缓解：第一版**不持久化**变量，进程退出即清零；确实需要持久化再引入 ADR-00XX "变量持久化"专门讨论

## 备注

- 本 ADR 建议在 ADR-0010 Pin 系统落地**之后**实施——变量的类型系统直接复用 `PinType`。
- 前端"运行时视图的变量面板"可以并行于后端实施，只要事件协议（广播变量变更）先商定。
- 未来与 ADR-0015 反应式引脚有潜在协同：变量变更可以 emit 成 `VariableChanged` 事件，反应式节点订阅该事件触发重新计算。

### Phase 1 落地记录（2026-04-27）

#### 已落地范围

- **Ring 0（`crates/core/src/variables.rs`）**：`WorkflowVariables`（DashMap 后端）/ `TypedVariable` / `TypedVariableSnapshot` / `VariableDeclaration` + `pin_type_matches_value` helper + `WorkflowVariables::empty()` 构造器。三个 `EngineError` 变体：`UnknownVariable` / `VariableTypeMismatch` / `VariableInitialMismatch`。
- **Facade（`src/graph/variables_init.rs`）**：`build_workflow_variables(Option<&HashMap<...>>) -> Result<Arc<WorkflowVariables>, EngineError>` — 部署期校验 + 初始化，插入 `RuntimeResources` / `NodeLifecycleContext`。
- **`NodeLifecycleContext.variables: Arc<WorkflowVariables>`** 字段（`crates/core/src/lifecycle.rs`）。
- **`WorkflowGraph.variables: Option<HashMap<String, VariableDeclaration>>`** schema 字段（`src/graph/types.rs`）。
- **Rhai 绑定（`crates/scripting/src/lib.rs`）**：`ScriptNodeBase::new` 第 5 参 `variables: Option<Arc<WorkflowVariables>>`，注册 `vars.get` / `vars.set` / `vars.cas` 三个方法（`ScriptVars` 类型，`pub(crate)`）。
- **5 个 nodes-flow 节点工厂迁移**（`crates/nodes-flow/src/lib.rs`）：`if` / `switch` / `loop` / `tryCatch` / `code` 从 `SharedResources` 提取 `Arc<WorkflowVariables>` 传给 `ScriptNodeBase::new`。
- **IPC `snapshot_workflow_variables`**（`src-tauri/src/lib.rs`）+ `SnapshotWorkflowVariablesRequest` / `SnapshotWorkflowVariablesResponse` 类型（`crates/tauri-bindings/src/lib.rs`）+ ts-rs 导出；`WorkflowDeployment::resources()` 访问器（`src/graph/deploy.rs`）。

#### 实施期间的决策偏离

1. **`WorkflowGraph.variables` 改为 `Option<HashMap<...>>`**：原 plan 草稿为 `HashMap<String, VariableDeclaration>`，实施时改为 `Option<HashMap<...>>`——ts-rs 要求 `Option<T>` 才能在 TS 端生成 `?:` 可选字段。`build_workflow_variables` 接受 `Option<&HashMap>` 吞下这层 Option，调用方写法不受影响。
2. **`Custom` 类型变量 Phase 1 完全拒绝**：`from_declarations` 初值校验和 `set` / `compare_and_swap` 写入均拒绝 `PinType::Custom(_)` ——与 ADR-0010 Phase 4 deferred Item 2 共享触发条件，等 `Custom` 类型稳定后同步放开。
3. **持久化、变更事件广播、前端面板留 Phase 2**：独立 plan 启动，依赖 Phase 1 的 IPC + 类型契约。
4. **`compare_and_swap` 是唯一附加并发原语**：`fetch_add` 等可由用户 CAS 循环组合，按需扩展，不预留。
5. **`TypedVariable` 内部用 `chrono::DateTime<Utc>`，IPC 表示用 `TypedVariableSnapshot.updated_at: String`（RFC 3339）** + `#[serde(rename_all = "camelCase")]` ——避免前端处理时区差异，与现有 IPC 字段命名惯例对齐。
6. **`from_declarations` 与 `build_workflow_variables` 泛化为 `<S: BuildHasher>`**：满足 `clippy::implicit_hasher`，让公共 API 接受任意 hasher 的 `HashMap`。
7. **类型名 `Snapshot*` 替代 plan 草稿的 `List*`**（`SnapshotWorkflowVariablesRequest` / `SnapshotWorkflowVariablesResponse`）：`snapshot` 更精确表达"瞬时一致性读"语义。
8. **Rhai `ScriptVars` 设计偏离 plan 草稿**：弃用"zero-payload `Vars` shell + free fn `__vars_get/set/cas` fallback"路径，改为 `ScriptVars { binding: Arc<VarsBinding> }` 直接 `register_fn` 绑定 `&mut Self` 方法——代码更短，无 dead path。

#### Phase 2 候选项（独立 plan 启动）

- 前端变量面板（`web/src/components/RuntimeVariables/`）+ 实时刷新轮询 `snapshot_workflow_variables`
- `VariableChanged` 事件 + `Window::emit("workflow://variable-changed")` 广播（供面板订阅）
- IPC `set_workflow_variable`（带类型校验，外部 / 运营人员手动改 setpoint）

### Phase 2 落地记录（2026-04-27）

**已落地范围：**
- Ring 0：`ExecutionEvent::VariableChanged` 变体（含 RFC3339 时间戳 + updated_by）+ serde 影子双向同步
- `WorkflowVariables` 加 `OnceCell<EventSink>` 字段；`set` / `compare_and_swap` 在 `entry.value != new` 时 `try_send` 一条 `VariableChanged`（`drop(entry)` 在 `try_send` 之前避免持锁 IO）
- deploy 期 `workflow_variables.set_event_sender(workflow_id, event_tx.clone())` 注入；`workflow_id` 改为调用方传入（`deploy_workflow_with_ai` 加 `workflow_id: Option<String>` 第 5 参），engine 不再自己派生
- `WorkflowDeployment::resources()` 访问器加回（Phase 1 /simplify 删过），doc 注明 ADR-0012 Phase 2 集成测试与 IPC 共享访问需求
- Tauri shell drain loop 加 `VariableChanged` 分支：转发到 `Window::emit("workflow://variable-changed", VariableChangedPayload)`，**continue** 短路不再发到 `node-status`
- IPC `set_workflow_variable(workflow_id, name, value)`：走 `Arc<WorkflowVariables>::set` 路径，`updated_by = Some("ipc")`，类型不匹配 / 变量未声明通过 `Result<_, String>` 上抛，写入后立即读回 `TypedVariableSnapshot` 返回
- `TypedVariableSnapshot.updated_by` + `VariableChangedPayload.updated_by` 都补 `#[serde(skip_serializing_if = "Option::is_none")]`，wire 格式与 ts(optional) 契约对齐（None → key 缺失而非 null）
- 前端 `web/src/lib/workflow-variables.ts`：`setWorkflowVariable` / `snapshotWorkflowVariables` / `onWorkflowVariableChanged` 三个 IPC wrapper + 4 个 vitest 单测
- 前端 `RuntimeVariablesPanel.tsx`：变量列表（实时刷新）+ 类型化编辑表单（按 `PinType.kind` 分派输入解析）；集成进 `RuntimeDock` 作为新"变量" Tab；`activeWorkflowId` 通过 `StudioContentRouter → BoardWorkspace → RuntimeDock` 链路从 `engine.deployInfo` 流入

**实施期间的决策偏离 Phase 2 plan 草稿：**
1. `WorkflowVariables.event_sink` 用 `tokio::sync::OnceCell` 而非 plan 草拟的 `arc_swap::ArcSwapOption`——`OnceCell` 已是 workspace dep（Phase 1 触发器 lifecycle 用过同类原语），避免引入 `arc_swap`。语义"注入一次"与 ADR-0012 单次部署内 sink 不变的事实匹配。
2. **B 方案完整落地**：`set` 的"写就更新 updated_at"语义保留（对调用方零破坏），事件 emit 在值真正变化时才发生——审计与反应式两种语义并存。
3. workflow_id 来源**不**用 `WorkflowGraph.name` fallback，而是改为 `deploy_workflow_with_ai` 第 5 参由调用方传入——发现 src-tauri `derive_workflow_id(workflow_id, project_id, graph.name)` 会与 engine 内 fallback diverge，导致前端事件按 workflow_id 过滤失败。
4. 命令名最终是 `set_workflow_variable`（plan 文中早期命名讨论留下的 `Snapshot*` vs `List*` 命名争议在 Phase 1 final review 已解决，Phase 2 一致用 `Snapshot` / `Set` 系列）。
5. `VariableRow` 内加 `useEffect` 同步 draft——code review 找到的真实 UX bug：父组件接事件更新 `entry.value` 时 row 不重 mount，draft state 保留 mount 时旧值；用户开 edit 看到过期值。
6. `parseValueByPinType` 加空字符串守卫——`Number('') === 0` 会 silent 写 0 到 setpoint 类变量。
7. ADR-0012 Phase 1 的 `TypedVariableSnapshot.updated_by` 顺手加了 `skip_serializing_if`——Phase 2 IPC 一致性带回，wire 格式与 ts(optional) 契约对齐，前端 Tasks 5-6 直接受益。
8. 前端 `runtimeDock` 对象在 `StudioContentRouter.tsx` 而非 `App.tsx` 组装（plan 草稿 misidentify 了位置）。
