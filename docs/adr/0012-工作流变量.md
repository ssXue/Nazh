# ADR-0012: 工作流级共享变量（WorkflowVariables）

- **状态**: 提议中
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-05 工作流变量」；依赖 ADR-0009 生命周期钩子（变量在 `on_deploy` 初始化）；与 DataStore（ADR-0008 已提及）互补但定位不同

## 背景

Nazh 目前的数据流动严格沿 DAG 边传播——节点只能看到上游 `payload`。这在多数场景够用，但工业边缘有一类高频需求无法表达：

1. **PID 控制**：被控变量的 `setpoint`（期望值）由一个节点设定，多个控制循环（温度、湿度、压力）同时读取；setpoint 可能随时被运营人员修改
2. **累加器 / 计数器**：统计"本小时告警次数"、"已处理消息总数"——跨多次 trigger 维持状态
3. **模式开关**：全局"手动/自动模式"开关，影响多条支路的行为
4. **动态配置**：AI 决策节点下发"本轮采样频率"，多个 Modbus 根据此调整自己的轮询周期
5. **跨循环聚合**：窗口聚合（平均值、最大值）需要跨 trigger 累积历史数据

当前变通方案是"把状态塞进 payload 里绕一圈传给需要的节点"，但 DAG 不是一个图灵完备的数据流模型——没有回边，状态很难回到源头。另一个变通是让节点自己维护 `Arc<Mutex<...>>` 作为内部状态，但这属于节点私有、不在工作流层面可观测，也无法被 Rhai 脚本访问。

DataStore（ADR-0008 提及）也不解决问题：DataStore 是"按 DataId 索引的不可变消息体"仓库，供 ContextRef 指针引用，设计目标是减少 payload clone，不是共享可变状态。

## 决策

> 我们决定引入**工作流级变量存储** `WorkflowVariables`：每个已部署工作流持有一个 `DashMap<String, TypedVariable>`，生存周期与部署同步（部署时创建、撤销时销毁）。节点通过注入的句柄读写变量；Rhai 脚本通过 `vars.get("setpoint")` / `vars.set("counter", 42)` 访问；前端可在"运行时视图"实时查看变量当前值。

### 拟议类型

```rust
// 新增 crates/core/src/variables.rs

pub struct WorkflowVariables {
    inner: Arc<DashMap<String, TypedVariable>>,
}

pub struct TypedVariable {
    pub value: Value,
    pub variable_type: PinType,       // 复用 ADR-0010 的 PinType
    pub updated_at: DateTime<Utc>,
    pub updated_by: Option<String>,   // 最后写入的 node_id
}

impl WorkflowVariables {
    pub fn get(&self, key: &str) -> Option<TypedVariable>;
    pub fn set(&self, key: &str, value: Value, node_id: &str)
        -> Result<(), EngineError>;   // 类型不匹配则拒写
    pub fn compare_and_swap(&self, key: &str, expected: &Value, new: Value)
        -> Result<bool, EngineError>;
    pub fn snapshot(&self) -> HashMap<String, TypedVariable>; // 供前端读取
}
```

变量的**类型声明**作为工作流定义的一部分：

```json
{
  "nodes": [ ... ],
  "edges": [ ... ],
  "variables": {
    "setpoint":    { "type": "Float",   "initial": 25.0 },
    "mode":        { "type": "String",  "initial": "auto" },
    "alert_count": { "type": "Integer", "initial": 0 }
  }
}
```

### 传递到节点与 Rhai

```rust
// NodeLifecycleContext（ADR-0009）新增字段
pub struct NodeLifecycleContext {
    pub resources: SharedResources,
    pub handle: NodeHandle,
    pub shutdown: CancellationToken,
    pub variables: WorkflowVariables,  // 新增
}
```

Rhai 脚本通过 `scripting` crate 注入的全局对象访问：

```rhai
let current = vars.get("setpoint");
vars.set("alert_count", vars.get("alert_count") + 1);
```

## 可选方案

### 方案 A: 让节点自己持有 `Arc<Mutex<...>>` 内部状态

- 优势：改动最小
- 劣势：
  - 状态不可观测、不可持久化、不可跨节点共享
  - 每种需求（计数器/setpoint/模式）都要一个独立节点——生态爆炸
  - Rhai 脚本无法访问

### 方案 B: 复用 DataStore 作为共享可变状态

- 优势：不引入新概念
- 劣势：
  - DataStore 是"消息体仓库"，`DataId` 由 Runner 生成、生命周期随 ContextRef 引用计数
  - 没有"命名键"概念；没有类型约束；TTL GC 会误清掉常驻变量
  - 职责扭曲：控制平面（配置/状态）和数据平面（消息）混在一起，违反 ADR-0008 建立的分离契约

### 方案 C: 独立 `WorkflowVariables` 存储（已选）

- 优势：
  - 语义清晰：**控制平面的共享可变状态**，与 DataStore（数据平面）正交
  - 类型化：复用 ADR-0010 的 `PinType`，写入时类型不匹配被拒
  - 可观测：前端能实时看到所有变量当前值
  - Rhai 友好：一行代码访问
  - 生命周期与部署对齐：撤销工作流 = 释放变量
- 劣势：
  - 引入全局可变状态，理论上会制造"隐式耦合"（节点 A 改了变量影响节点 B）
  - 并发写入需要仔细设计（提供 `compare_and_swap` 基本原子语义）
  - 前端渲染变量面板是额外工作

### 方案 D: 接外部 KV 存储（Redis / etcd）

- 优势：持久化、跨进程
- 劣势：边缘设备多一跳网络开销；违反"单进程部署"前提；对当前需求过度设计

## 后果

### 正面影响

- **PID 控制、累加器、模式开关**等经典工业逻辑可在单 DAG 内表达，无需变通
- **运营可视化**：前端运行时视图可以显示"当前 setpoint = 25.3°C、alert_count = 142"——极大改善可观测性
- **Rhai 脚本表达力提升**：脚本不再是"无状态 payload 变换"，而是可以做有状态决策
- **配合 ADR-0009 生命周期钩子**：`on_deploy` 初始化变量初值、`on_undeploy` 做持久化快照（如果将来需要）
- **AI 决策自然落地**：AI 节点把决策结果写入 `vars.decision`，其他节点读取——AI 与确定性逻辑通过变量解耦

### 负面影响

- 引入"DAG 之外的信息通路"——调试时需要理解"这个值从哪里来"不再只是"沿边回溯上游"
- 变量面板的前端工作量不小
- 类型系统依赖 ADR-0010 Pin 的 `PinType`，两个 ADR 耦合度高

### 风险

- **风险 1：并发写入导致脏数据**
  - 缓解：`DashMap` 本身 thread-safe；提供 `compare_and_swap` 支持原子递增；复杂场景鼓励用 `Arc<Mutex<T>>` 模式
- **风险 2：变量成为"全局 god object"**
  - 缓解：部署时强制声明所有变量及类型；不允许运行时动态创建；review 时关注变量数量，超过 ~20 个视为架构 smell
- **风险 3：持久化需求出现后 schema 迁移痛苦**
  - 缓解：第一版**不持久化**变量，进程退出即清零；确实需要持久化再引入 ADR-00XX "变量持久化"专门讨论

## 备注

- 本 ADR 建议在 ADR-0010 Pin 系统落地**之后**实施——变量的类型系统直接复用 `PinType`。
- 前端"运行时视图的变量面板"可以并行于后端实施，只要事件协议（广播变量变更）先商定。
- 未来与 ADR-0015 反应式引脚有潜在协同：变量变更可以 emit 成 `VariableChanged` 事件，反应式节点订阅该事件触发重新计算。
