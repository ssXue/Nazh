# ADR-0019: AI 能力的依赖反转——`AiService` trait 上移到 Ring 0

- **状态**: 已实施（2026-04-26）
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 涉及 `ai`、`scripting`、`nodes-flow`；与 ADR-0009 节点生命周期钩子、ADR-0011 能力标签兼容

> **实施备注（2026-04-26）**：
> - `AiService` trait + `AiCompletionRequest` / `AiCompletionResponse` / `AiMessage` / `AiMessageRole` / `AiTokenUsage` / `AiGenerationParams` / `AiThinkingConfig` / `AiThinkingMode` / `AiReasoningEffort` / `StreamChunk` / `AiError` 全部上移到 `nazh-core::ai`（新增 `crates/core/src/ai.rs`）。
> - **trait 范围调整**：`test_connection(AiProviderDraft)` **未**进入 Ring 0 trait——`AiProviderDraft` 是壳层配置类型（仅用于 IPC `test_ai_provider`），把它拖进 Ring 0 会反向拖进配置 schema。落地时改为 `OpenAiCompatibleService` 的 inherent 方法，壳层把 `ai_service` 字段类型从 `Arc<dyn AiService>` 改为 `Arc<OpenAiCompatibleService>`，部署时通过普通 unsize 强制转换为 `Arc<dyn AiService>` 注入。
> - `AiTestResult` 仍住在 `ai` crate（仅壳层使用）；`AiConfigFile` / `AiProviderDraft` / `AiProviderUpsert` / `AiProviderView` / `AiSecretInput` / `AiAgentSettings` 等配置型同样留在 `ai` crate。
> - `scripting` 与 `nodes-flow` 的 `Cargo.toml` **删除** `ai` 依赖；现在两个 Ring 1 crate 只依赖 `nazh-core`。
> - `ai` crate 通过 `pub use` re-export Ring 0 类型，老调用点（`use ai::AiService` 等）仍可工作；`thiserror` / `tracing` / `uuid` 不再被 `ai` 直接使用，从 `Cargo.toml` 移除。
> - ts-rs 类型导出自动跟随：generated TS 仅有 docstring 字符变化，无 schema 改动。

## 背景

AI 能力通过以下依赖链进入工作流：

```
AiService impl (OpenAI 兼容客户端)
    └─ 定义于 crates/ai/src/service.rs

引用者：
    crates/scripting/src/lib.rs       — ScriptAiRuntime 让 Rhai 脚本能调 ai_complete()
    crates/nodes-flow/src/code_node.rs — CodeNodeAiConfig 让用户用 AI 生成 Rhai 脚本
    src-tauri/src/lib.rs              — AI 配置 IPC 命令
    src/graph/deploy.rs               — deploy_workflow_with_ai() 把 AiService 注入 RuntimeResources
```

### 当前依赖形态

```
ai
  ↑       ↑          ↑
scripting nodes-flow  (src-tauri, nazh-engine facade)
```

两个 Ring 1 crate（`scripting`、`nodes-flow`）**无条件依赖** `ai`。即使部署不用 AI：

- `AiService` trait 仍被 import
- `reqwest` + `futures-util` 等 `ai` 的间接依赖仍被编译
- `ai` 整个 crate 必须构建

### 痛点

1. **Ring 分层纯净性违反**：Ring 1 之间可以互相依赖 + 依赖 Ring 0，但 `ai` 是"协议客户端"性质——在分层图里它本应是叶子节点，不该被其他 Ring 1 依赖
2. **强耦合优于抽象**：`ScriptAiRuntime` 拿到的是**具体 `AiService` 实现**，而不是行为契约——替换实现要改 trait 对象；未来接入其他 AI provider（Claude API 原生、本地 ollama）路径不通
3. **边缘设备不能完全拆 AI**：即使用户不开 AI 功能，`code` 节点也得编译 `CodeNodeAiConfig`（虽然是 `Option`）
4. **依赖方向与模块定义矛盾**：`ai` 在当前 Cargo.toml 被描述为"公共层"，但它位于依赖图的**源头**（被依赖方），这和"公共层"通常位于底层的直觉相反

## 决策

> 我们决定把 `AiService` trait（抽象接口）上移到 **Ring 0（`nazh-core`）**，具体实现留在 `ai`。`scripting` 和 `nodes-flow` 只依赖 Ring 0 的 trait，通过依赖注入（`Arc<dyn AiService>`）在运行时接入实现。这样：
>
> - Ring 1 crate 不再编译任何 AI HTTP 客户端代码
> - 未来接入 Claude 原生 SDK / 本地模型时只需新增 `nazh-ai-native` 实现 `AiService`
> - 不用 AI 的部署可以完全不依赖 `ai`（facade crate 里让它成为可选）

### 拟议分层

```
Ring 0 nazh-core
  ├─ NodeTrait, Plugin, DataStore, ...
  └─ AiService trait (+ AiCompletionRequest / AiMessage 请求类型)  ← 新增

Ring 1 ai
  └─ OpenAiCompatibleClient: impl AiService
       (专注 HTTP 实现；依赖 reqwest / futures-util)

Ring 1 scripting / nodes-flow
  └─ 只依赖 Ring 0 的 AiService trait
     运行时通过 SharedResources 拿 Option<Arc<dyn AiService>>
```

### Trait 移交的具体形状

`ai` 当前的 `AiService`：

```rust
#[async_trait]
pub trait AiService: Send + Sync {
    async fn complete(&self, request: AiCompletionRequest)
        -> Result<AiResponse, AiError>;
    async fn stream(&self, request: AiCompletionRequest)
        -> Result<AiStream, AiError>;
}
```

上移策略：

- **`AiService` trait + 请求/响应类型 → Ring 0**（纯 trait + 数据结构，零协议依赖）
- **`AiError` → Ring 0** 或保留在 `ai`（有 `reqwest` 相关变体的话要脱钩——用 `String` 兜底或定义为"语义错误"类别）
- **OpenAI 客户端、Anthropic 客户端等实现 → `ai`**（或未来多个 crate）

### 注入路径不变

`RuntimeResources` 的 `Arc<dyn AiService>` 机制已经就位——只是 trait 的 crate 位置变了。`src-tauri` 仍负责实例化 `Arc<dyn AiService>` 注入。

## 可选方案

### 方案 A: 维持现状（scripting/nodes-flow 硬依赖 ai）

- 优势：零改动
- 劣势：分层债长期累积；换 AI provider 要改多个 crate

### 方案 B: Feature flag（`scripting` / `nodes-flow` 把 AI 功能放 feature 后面）

```toml
# crates/scripting/Cargo.toml
[features]
ai = ["dep:ai"]
```

- 优势：迁移量小，代码中 `#[cfg(feature = "ai")]` 包裹即可
- 劣势：
  - 仍然是**具体依赖**不是抽象——换 AI provider 还得改这两个 crate
  - feature 组合增加 CI 负担
  - `code` 节点的 `CodeNodeAiConfig` 类型会随 feature 消失，影响 JSON 向后兼容

### 方案 C: Trait 上移到 Ring 0（已选）

- 优势：
  - **真正的依赖反转**：Ring 1 只知道 trait，不知道具体实现
  - **换 provider 零侵入**：新 impl crate + 配置选择即可
  - **Ring 0 依赖图回归纯净**：ai 重新变成叶子节点
  - 与 `ConnectionManager` 设计哲学一致——通过接口抽象协议差异
- 劣势：
  - trait 定义在 Ring 0，Ring 0 文件略微膨胀（但 trait 本身轻）
  - 需要设计 Ring 0 版的 `AiError`——要剥离具体实现相关的错误分支
  - 迁移改动 10-15 个文件
  - `ai` crate 要重新自我定位为"OpenAI 兼容客户端实现"，而不是"AI 公共层"

### 方案 D: 干脆把 AI 代码都塞进 nazh-core

- 优势：最简单
- 劣势：严重违反"Ring 0 不含协议依赖"——reqwest 进 Ring 0 是底线失守

### 方案 E: 让 AI 成为独立 crate，但不上移 trait

```
crates/ai-trait/      （只有 trait）
crates/ai/  （impl）
```

- 优势：不动 Ring 0
- 劣势：无意义地多一个小 crate——trait 在 Ring 0 本就合适，这是多此一举

## 后果

### 正面影响

- **AI provider 可插拔**：未来加 Anthropic 原生 SDK、本地 Llama、阿里通义等只需新 crate 实现 trait
- **Ring 分层与文档对齐**：没有 Ring 1 crate 跨依赖 Ring 1 crate 这种扭曲
- **边缘部署可完全剥离 AI**：facade crate（`nazh-engine`）把 `ai` 设为 optional dep，embedded 部署零 AI 依赖
- **单元测试更容易 mock**：测试里直接 impl `AiService` 的 `MockAi`，不用 mock HTTP
- **与 ADR-0009 生命周期钩子天然协同**：未来 AI 节点的 `on_deploy` 钩子可以用注入的 `AiService` 做 warmup（预请求一次校验 token）

### 负面影响

- 需要设计"provider-neutral" 的 `AiError` —— 现有的 `AiError` 有 `Reqwest(reqwest::Error)` 等实现相关变体要脱钩
- 迁移期间两种引用并存（`nazh_core::AiService` vs `ai::AiService`）——要在一次 PR 内彻底完成切换
- `ai` 重新定位需要更新文档；未来读代码的人要理解"这个 crate 是 **OpenAI 兼容实现**，不是 AI 公共层"

### 风险

- **风险 1：`AiError` 抽象不够贴合具体实现**
  - 缓解：Ring 0 的 `AiError` 定义为"语义错误"（`AuthFailed` / `RateLimited` / `ServerError { status, body }` / `Network(String)`）；实现 crate 负责把 reqwest 错误映射过来
- **风险 2：流式 API 的 trait 对象对异步生态要求高**
  - 当前 `stream()` 返回的 `AiStream` 是 `impl Stream<Item = ...>` 类型——作为 trait 方法要 box 化（`Pin<Box<dyn Stream + Send>>`）
  - 缓解：box 化的性能开销可忽略；工业场景 AI 请求不在 hot path
- **风险 3：`CodeNodeAiConfig` 用到的类型迁移**
  - `AiGenerationParams`、`AiThinkingConfig` 等配置结构要跟着 trait 一起上移
  - 缓解：这些本来就是"用户配置" + "trait 入参"性质，适合在 Ring 0

## 备注

- 建议在 ADR-0017（IPC / ts-rs 迁出）之后实施——届时 Ring 0 已经经历一次瘦身，把 `AiService` 加回来的审视压力小。
- 实施顺序：(1) 在 `nazh-core` 中定义 trait + 请求/响应类型；(2) `ai` 实现 trait；(3) `scripting` / `nodes-flow` 改依赖；(4) 在 `nazh-engine` facade 中把 `ai` 转为 optional dep。
- 长期：未来 AI 能力可能扩展到"Embedding 向量检索"、"图像识别"、"TTS/ASR"——届时 Ring 0 可能需要拆分成 `AiService` / `EmbeddingService` / `VisionService` 等多个 trait。本 ADR 不预判，但 trait 拆分的前例由此建立。
