# ADR-0014: 分离执行边（Exec）与数据边（Data）

- **状态**: 提议中
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-07 执行/数据边分离」；依赖 ADR-0010 Pin 系统（边的类型由引脚决定）；配合 ADR-0011 能力标签（`PURE` 节点真正受益需此 ADR）；是本批次最"动根基"的 ADR

## 背景

Nazh 当前的 DAG 模型里，**每条边既是控制流又是数据流**：上游节点完成 `transform` → MPSC send → 下游收到消息 → 立即执行 `transform`。这是一个纯 **push 模型**，灵感来自消息管道。

这个模型在"每一跳都有意义"的 IO 管道场景下很自然，但遇到一类场景会别扭：

### 场景 1：纯计算节点（例如 `if`、`switch`）的结果应该被下游**按需拉取**

```
trigger → [readValue] → [computeAverage] → [shouldAlert?]
                               ↑
                               └──────── [otherNode]（另一路也想用 computeAverage 结果）
```

当前只能让 `computeAverage` 广播给所有需要它的下游，上游每次 trigger 都会推一次；如果 `otherNode` 和 `shouldAlert` 触发频率不同，就会做无效计算或者失去新鲜度。

### 场景 2：UE5 Blueprint 风格的 "pure 节点"

UE5 的 pure 节点（蓝色输出、无执行引脚）是按下游需要时才计算，避免做无用功。Nazh 想学习这个范式，但目前 DAG 只有一种边——push 边。

### 场景 3：跨触发时钟的数据读取

定时器节点 A 每 1s tick 一次，触发 `getLatestReading`；另一个定时器 B 每 5s tick 一次，需要读取"最近的 A 结果"。当前没有干净的办法，只能靠变量存储（ADR-0012）或者让 A 强行推给 B。

## 决策

> 我们决定为 DAG 边引入两种**枚举类型**：`EdgeKind::Exec`（执行边，当前语义不变）和 `EdgeKind::Data`（数据边，下游按需拉取上游缓存的最新输出）。节点的 `input_pins` 声明引脚类型时同时声明边模式（`required_edge_kind`），部署期校验。Runner 按边类型分两条路径执行：Exec 边触发下游 `transform`；Data 边仅更新上游节点的输出缓存，等下游被别的 Exec 边触发时才去读缓存。

### 拟议数据模型

```rust
// crates/core/src/graph.rs
pub enum EdgeKind {
    /// 当前语义：上游完成→push→下游触发 transform
    Exec,
    /// 新语义：上游完成→写缓存；下游在 Exec 触发时按需读取
    Data,
}

pub struct Edge {
    pub from_node: String,
    pub from_pin: String,
    pub to_node: String,
    pub to_pin: String,
    pub kind: EdgeKind,
}

// 上游节点内部新增输出缓存
pub struct OutputCache {
    /// 每个输出引脚一个缓存槽，使用 ArcSwap 做无锁更新
    slots: HashMap<String, Arc<ArcSwap<Option<CachedOutput>>>>,
}

pub struct CachedOutput {
    pub value: Value,
    pub produced_at: DateTime<Utc>,
    pub trace_id: Uuid,
}
```

### Runner 双模式

```
// 上游完成 transform 后：
for each output_pin:
    for each outgoing edge from this pin:
        match edge.kind:
            Exec  → MPSC send to downstream (current path)
            Data  → write to output cache (no send)

// 下游被 Exec 触发、开始执行时：
collect_data_inputs():
    for each Data edge into this node:
        read upstream's cached output slot
        (若槽为空：按下游配置决定 skip / error / default)
    compose payload from { exec_payload, data_inputs }
    call transform(payload)
```

### 引脚与边模式匹配

- `PinType::Any` 既可作为 Exec 端也可作为 Data 端
- 部署期校验：若下游 pin 声明 `required_edge_kind = Data` 而实际连接是 Exec 边，拒绝部署
- 默认行为：未显式指定 → Exec（向后兼容）

## 可选方案

### 方案 A: 维持单一 Push 模型（现状）

- 优势：模型简单、Runner 单一路径、易于推理
- 劣势：
  - 纯计算节点的缓存能力无处发挥
  - 跨时钟读取只能靠变量绕行
  - 无法实现 UE5 风格的 pure 节点

### 方案 B: 完全改成 Pull 模型

- 下游被触发时主动向上游请求数据
- 优势：按需计算，符合 FRP 思想
- 劣势：
  - 推翻当前所有节点的语义，迁移成本极高
  - 触发器节点（timer、MQTT subscribe）天然是 Push 的——Pull 模型反而拧巴
  - 实时 IO 流（每帧处理）在 Pull 模型下极其啰嗦

### 方案 C: 双模式混合（已选）

- 优势：
  - **向后兼容**：现有所有边默认 Exec，无需改动
  - **扩展自然**：按需声明 Data 边，获得缓存/跨时钟读取能力
  - **UE5 Blueprint 风格可达**：pure 节点的输出引脚连 Data 边给下游
  - 配合 ADR-0011 能力标签：`PURE` 节点的 Exec 边其实是冗余的，`PURE + only Data outgoing` 是理想模式
- 劣势：
  - Runner 复杂度显著增加——要维护双路径
  - 调试时用户需要理解两种边的语义差异
  - 缓存生命周期（何时清空、是否有 TTL）需要额外决策

### 方案 D: 显式的"缓存节点"作为中间件

- 在需要缓存的地方插入显式 `cache` 节点
- 优势：不改边模型
- 劣势：
  - DAG 视觉噪声大——每个需要缓存的地方多一个节点
  - 作者心智负担：要记得插入
  - 不解决"跨时钟按需读取"的本质问题

## 后果

### 正面影响

- **UE5 风格 pure 节点**：纯计算节点（`if`、`switch`、数学节点）可以真正发挥"按需执行、按需缓存"的效率
- **跨时钟数据读取干净**：定时器 B 可通过 Data 边读取定时器 A 的最新输出，无需变量
- **配合 ADR-0011 `PURE` 标签实现结果缓存**：`PURE` 节点 + Data 边下游 = 相同输入自动命中缓存
- **子图（ADR-0013）的内部边可声明为 Data**：使得子图内部的纯计算模式高效
- **节点生态分层更清晰**：Trigger → Exec 流水 → Data 拉取 → 终端节点——心智模型更丰富

### 负面影响

- **Runner 大幅改造**：从单一 push 循环变成"Exec 触发 + Data 拉取"双模式，代码复杂度显著增加
- **调试心智成本上升**：用户看到"数据边"不触发下游，需要理解为什么
- **缓存生命周期管理**：空槽读取、trace_id 不一致、TTL 策略都是新设计
- **前端需要区分两种边的视觉样式**（粗/细、实线/虚线）

### 风险

- **风险 1：缓存空槽**——下游被触发时，上游还没执行过，Data 边缓存是空的
  - 缓解：引脚声明时带 `default_value` 或 `block_until_ready` 策略；部署期校验至少一种兜底
- **风险 2：trace_id 不一致**——下游的 Exec 触发和 Data 读取可能来自不同的 trace
  - 缓解：`CachedOutput` 携带原始 `trace_id`，下游事件中同时记录多个来源 trace——观测层接受"一个 transform 可能关联多个 trace"
- **风险 3：循环依赖**——Data 边可能制造"节点 A 依赖 B 的缓存，B 依赖 A 的缓存"
  - 缓解：部署期的拓扑排序只考虑 Exec 边；Data 边不参与拓扑，但仍做环检测单独报错
- **风险 4：改造影响面广**——Runner、deploy、graph、前端渲染、事件系统全都要动
  - 缓解：本 ADR 建议放在 **ADR-0010 Pin 系统之后、较晚阶段**实施；分阶段：先支持 Data 边但不做 PURE 缓存；再做 PURE 缓存

## 备注

- 本 ADR 是本批次 7 条提案里**改动最大**的一条，建议放在执行顺序靠后。
- 实施前置条件：ADR-0010 Pin 系统必须已稳定落地（边的类型系统依赖它）；ADR-0011 能力标签最好也已存在（`PURE` 标签让缓存激进优化成为可能）。
- 未来可扩展：`EdgeKind::Reactive`（ADR-0015 反应式引脚）作为第三种边类型——在 Data 边的拉取基础上加"订阅"语义，上游变化自动通知下游。
