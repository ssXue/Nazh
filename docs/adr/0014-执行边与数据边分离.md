# ADR-0014: 引脚求值语义二分（PinKind: Exec / Data）

- **状态**: 已实施 Phase 1
- **日期**: 2026-04-24（提议）／ 2026-04-28（重写为引脚二分 + Phase 1 落地）
- **决策者**: Niu Zhihong
- **关联**:
  - 取代本 ADR 原"边二分（EdgeKind）"方案——详细对比与重写理由见
    `docs/superpowers/specs/2026-04-28-pin-kind-exec-data-design.md`
  - 扩展 ADR-0010 Pin 系统（增加 `PinKind` 维度，与 `PinType` 正交）
  - 与 ADR-0011 `NodeCapabilities` 正交（引脚级语义 vs 节点级语义）
  - 为 ADR-0015「反应式数据引脚」铺垫（反应式作为 `PinKind::Reactive` 第三值）
  - 不与 ADR-0012「工作流变量」冲突（变量 = 具名跨工作流共享；引脚二分 = 工作流内按需读最新值）

## 背景

Nazh 当前 DAG 模型里，每条边既是控制流又是数据流——上游 transform 完成 → MPSC send → 下游收到立即 transform。这是一个纯 push 模型，在"每一跳都有意义"的 IO 管道下很自然，但有三类场景反复硌脚：

1. **跨时钟读取**：1 秒主控环的滑动平均，5 分钟摘要环要"最新平均值"——目前只能用变量绕行（setVariable / getVariable 桥接节点污染画布）
2. **派生计算 fan-out**：同一个 Modbus 寄存器读结果，被告警判断 / 仪表盘显示 / 历史采样三个不同节奏消费者使用——push 模型导致重复 IO 或中转节点
3. **UE5 风格组合表达式**：派生字段（华氏温度、距上次告警分钟数、设备序列号）想形成独立"表达式树"挂在主链 Data 输入上，而非串入主流水

## 决策

> 我们决定把"求值语义（push 还是 pull）"放在**引脚**上而非**边**上。`PinDefinition`
> 新增 `kind: PinKind` 字段，与 `PinType`（数据形状）正交。边的语义由两端引脚的
> PinKind 自动派生（Exec 引脚连 Exec 引脚 = push 边；Data 引脚连 Data 引脚 = pull 边）。
> 部署期校验拒绝跨 Kind 连接。

## 可选方案

设计阶段对比了三种放置语义层级的方案（详见 spec 第一章）：

### 方案 A: 边二分（原 ADR-0014）

- **优势**：边类型清晰显式
- **劣势**：同引脚出来两条线行为不同、引擎维度膨胀、UE5 没有此范式

### 方案 B: 节点二分（DATA_SOURCE / EFFECT capability）

- **优势**：贴 SCADA / CQRS 工业范式
- **劣势**：节点级声明无法表达"同节点既给主路径推、又被旁路拉"

### 方案 C: 引脚二分（PinKind: Exec / Data）— 已选

- **优势**：UE5 Blueprint 经过验证、与 ADR-0010 PinType 同层级正交、表达力最强
- **劣势**：引脚维度需要扩展、用户首次需理解形状二分（但形状差异显著，比"两种边"低得多的认知负担）

## 后果

### 正面影响

- 跨时钟读取无需变量桥接节点
- Fan-out 场景设备 IOPS 降为 1× / 触发频率
- 派生表达式与主控逻辑视觉分离
- 后续 ADR-0013 / ADR-0015 / Phase 6 EventBus 直接消费 PinKind 不必重发明语义维度

### 负面影响

- Runner 引入双路径分支（Exec push + Data 写 cache）
- 调试心智需要理解"两种引脚形状"
- 缓存生命周期（空槽兜底、TTL 等）需要后续 Phase 决策

### 风险

- **缓存空槽**：下游被触发时上游还没产生值——Phase 4 通过引脚级 default_value / block_until_ready 策略缓解
- **trace_id 不一致**：一次 transform 可能关联多个上游 trace——CachedOutput 携带 trace_id，观测层接受多 trace 关联
- **Data 边环依赖**：A 拉 B 缓存，B 拉 A 缓存——部署期 detect_data_edge_cycle 拒绝（Phase 1 已实施）

- 本 ADR 是本批次 7 条提案里**改动最大**的一条。Phase 1 通过严格的"骨架先行 + 业务零侵入"策略把改动 contained 在引擎核心，14 类生产节点不需要任何代码修改
