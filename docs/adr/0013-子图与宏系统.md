# ADR-0013: 子图节点与宏展开（SubgraphNode）

- **状态**: 已实施（子图核心，2026-04-28）
- **日期**: 2026-04-24
- **决策者**: Niu Zhihong
- **关联**: 对应内部提案「提案-06 子图 / 宏系统」；依赖 ADR-0010 Pin 系统（子图需要对外的输入/输出引脚定义）；与 ADR-0009 生命周期钩子互相兼容

## 背景

Nazh 目前的工作流是**扁平 DAG**：所有节点都在同一层，没有封装/嵌套机制。这在工作流规模小时不是问题，但一旦用户在多个项目中反复使用相同模式，问题开始浮现：

典型重复模式（来自当前节点库可组合的场景）：
- **Modbus 采集三件套**：`modbusRead` → `if`（值合法性校验） → `sqlWriter`
- **告警推送链**：`if`（阈值判断） → `httpClient`（调用告警 API） → `barkPush`（手机通知）
- **定时上报管道**：`timer` → `httpClient`（查询接口） → `code`（数据清洗） → `sqlWriter`

每个新工程都要手工重建这些模式——节点拖拽、连线、参数填写。这带来：

1. **重复劳动**：相同拓扑结构复制 N 遍
2. **不一致风险**：每次"差不多"重建，容易漏连一条边或填错阈值
3. **升级难**：所有副本都是独立节点，模式本身升级（例如告警链加一层脱敏）要逐个改
4. **不支持组织级别的"最佳实践"沉淀**：无法把"公司标准的采集管道"封装成可分发资产

## 决策

> 我们决定引入**子图节点**（`SubgraphNode`）作为一等公民的工作流组合单元。子图对外表现为单个节点（有自己的输入引脚、输出引脚、配置），内部包裹一个 `WorkflowGraph`。**部署时采用"内联展开（inline expansion）"策略**：子图在部署期被解糖成扁平 DAG，节点 ID 加前缀 `<subgraph_id>/<inner_node_id>`，边同理。运行时的 Runner 不知道子图的存在，保持单一执行模型。

### 拟议数据模型

```rust
// crates/core/src/subgraph.rs
pub struct SubgraphDefinition {
    pub id: String,
    pub name: String,
    pub version: String,
    pub input_pins: Vec<PinDefinition>,   // 子图对外输入
    pub output_pins: Vec<PinDefinition>,  // 子图对外输出
    pub parameters: Vec<ParameterDefinition>,  // 实例化时需要填的参数（如 MQTT host）
    pub inner: WorkflowGraphDefinition,   // 内部 DAG（可嵌套子图）
}

pub struct SubgraphNode {
    pub id: String,                    // 子图实例 id
    pub definition_ref: String,        // 指向 SubgraphDefinition
    pub parameter_bindings: Map<String, Value>,  // 实例填的参数
}
```

### 部署期展开规则

```
输入 DAG：
    trigger → [subgraph("collect-modbus") ] → sqlWriter

SubgraphDefinition("collect-modbus").inner：
    in → modbusRead → validate → out

展开后的扁平 DAG：
    trigger
       → collect-modbus/modbusRead
       → collect-modbus/validate
       → sqlWriter

参数替换：SubgraphNode 的 parameter_bindings
    { "mb_host": "192.168.1.10", "register": 40001 }
    在展开时被注入到内部 modbusRead 节点的 config 中
```

### 输入/输出引脚映射

- 子图内部使用特殊节点 `SubgraphInputNode` / `SubgraphOutputNode` 作为与外部的接驳点
- 展开时，外部边 `upstream → subgraph.input_pin_X` 被重写为 `upstream → <subgraph_id>/<input_node_for_X>`
- 出口同理

### 嵌套上限

允许子图嵌套子图，但**最大深度限制为 8 层**（防止循环引用或无意递归）——展开算法循环检测 + 深度计数。

## 可选方案

### 方案 A: 人工复制 + 模板文件

- 优势：不改引擎
- 劣势：
  - 每个模板副本都是独立节点，升级时 N 处改动
  - 前端"模板库"功能其实是复制粘贴 + 改名，不是真正的复用
  - 无法形成"组织级节点库"的技术基础

### 方案 B: 宏展开（部署期 inline，已选）

- 优势：
  - Runner 零改动——它依然只看到扁平 DAG
  - 子图可用标准 DAG 工具调试（节点ID前缀清晰）
  - 参数化友好：部署时替换
- 劣势：
  - 展开后节点数量膨胀——大子图、深嵌套场景下 DAG 节点数可能很大
  - 调试时需要从 `subA/subB/modbus` 这种路径反推子图来源（需要前端支持"折叠显示"）

### 方案 C: 运行时独立子 DAG（不展开）

- 子图作为运行时的**独立 DAG 实例**，父 DAG 通过特殊通道与子图 DAG 通信
- 优势：节点总数不膨胀；子图可独立启停/热更新
- 劣势：
  - Runner 必须认识"嵌套 DAG"概念——复杂度显著上升
  - 事件流需要"子图内事件是否要冒泡到父图"的策略
  - 资源管理复杂：ConnectionManager 是父图共享还是子图独立

### 方案 D: 函数式抽象（把子图当成 "宏调用" 存在节点库里）

- 子图本身不作为一等公民存在，而是**编辑器层面**的宏：用户拖一个"宏"，编辑器自动把它展开为若干节点后保存
- 优势：零后端改动
- 劣势：
  - 展开结果成为用户的"普通节点"——失去"这是一个封装"的语义
  - 子图升级时，旧展开结果不会跟进
  - 不具备版本管理

## 后果

### 正面影响

- **模式复用正式化**：公司/个人可以沉淀"标准采集模板库"、"标准告警链模板"等
- **Runner 零改动**：展开在部署期完成，运行时仍是扁平 DAG，保持单一执行模型的简洁
- **子图版本化**：`SubgraphDefinition` 自带 `version` 字段，工程可明确引用"告警链 v2"
- **参数化实例**：同一模式用不同参数部署多份（例如对 10 台 Modbus 设备跑同一采集模板）
- **为"AI 生成子图"铺路**：AI 可以生成并注册新子图，用户直接引用

### 负面影响

- 部署期展开逻辑增加复杂度（循环检测、ID 前缀、参数替换）
- 前端必须支持"子图折叠视图"才能在大图上有意义——UI 工作量不小
- 前端节点库面板需要区分"原子节点"和"子图节点"

### 风险

- **风险 1：循环引用**（子图 A 引用子图 B，B 引用 A）
  - 缓解：展开算法做循环检测 + 深度上限（8）
- **风险 2：子图对外引脚类型随版本变动导致破坏性升级**
  - 缓解：子图 `version` 字段作为强制语义化；前端在引用过时版本时给警告
- **风险 3：调试体验差**——用户看到错误发生在 `modbus-alerts/inner/modbusRead/validate`，难以追溯
  - 缓解：前端显示错误时高亮包含关系：`告警链 › 采集子图 › validate`；日志保留前缀路径

## 备注

- 本 ADR 依赖 ADR-0010 Pin 系统——子图对外的引脚定义必须是类型化的，否则"标准模板"的契约无从表达。建议 Pin 系统先行。
- 实施阶段建议：先支持**单层子图**（不嵌套）+ 无参数化；验证概念后再开启嵌套和参数化。
- 与 ADR-0012 工作流变量的关系：子图内部是否共享外部工作流的变量？**建议默认隔离**（子图有自己的变量作用域），需要共享时通过参数显式传递——避免全局 god object 在嵌套中失控。
