# ADR-0013: 子图节点（基于 FlowGram 子画布插件）

- **状态**: 已实施（子图核心）
- **日期**: 2026-04-24（初稿），2026-04-27（修订），2026-04-28（实施）
- **决策者**: Niu Zhihong
- **修订者**: ssXue <529576446@qq.com>（2026-04-27：基于 `@flowgram.ai/free-container-plugin` 实际 API 重构数据模型与展开策略）
- **关联**: 对应内部提案「提案-06 子图 / 宏系统」；依赖 ADR-0010 Pin 系统；与 ADR-0009 生命周期钩子兼容；基于 FlowGram 子画布插件实现

## 背景

Nazh 工作流是**扁平 DAG**，无封装/嵌套机制。工业场景中大量重复拓扑模式：

- **Modbus 采集三件套**：`modbusRead` → `if`（值合法性校验） → `sqlWriter`
- **告警推送链**：`if`（阈值判断） → `httpClient` → `barkPush`
- **定时上报管道**：`timer` → `httpClient` → `code` → `sqlWriter`

每个工程手工重建，导致：重复劳动、不一致风险、模式升级困难、组织级最佳实践无法沉淀。

### 已有技术基础

`@flowgram.ai/free-container-plugin`（v0.4.19）已安装注册（`useFlowgramEditorProps.ts:183`），提供：

| 能力 | 说明 |
|------|------|
| 数据模型 | `WorkflowNodeJSON.blocks[]`（子节点）+ `edges[]`（内部连线），递归树形 |
| 容器声明 | `meta.isContainer: true` |
| 渲染组件 | `SubCanvasRender` / `SubCanvasBackground` / `SubCanvasBorder` / `useNodeSize` |
| 拖入/拖出 | `NodeIntoContainerService`，拖入时 `clearInvalidLines` 移除跨容器连线 |
| 容器端口 | `defaultPorts`（input/output），但与内部子节点**无自动映射** |

官方文档：https://flowgram.ai/zh/guide/free-layout/sub-canvas

当前 `flowgram.ts` 已处理 `blocks`/`edges` 的 merge/normalize（`toFlowgramWorkflowJson` 保留），但 `toNazhWorkflowGraph` **完全忽略容器内部结构**——只取顶层 `nodes` 和 `edges`。

## 决策

> 我们决定引入**子图节点**作为一等工作流组合单元，**基于 FlowGram 子画布插件的原生容器模型**。子图对外表现为单个节点（带输入/输出引脚），内部包裹 `blocks[]` + `edges[]` 子拓扑。**部署时在前端内联展开**：`toNazhWorkflowGraph` 递归展平 `blocks`，拍平为扁平 DAG。Runner 零改动，保持单一执行模型。

核心设计主张：

1. **数据模型对齐 FlowGram** — 不自创 struct，直接用 `WorkflowNodeJSON` 的 `blocks`/`edges` 递归结构
2. **桥接节点补齐插件 gap** — FlowGram 容器端口与内部子节点之间无自动映射，用 `subgraphInput`/`subgraphOutput` 特殊节点桥接
3. **前端展平，Rust 不感知** — 展开逻辑在 `flowgram.ts`，Runner 始终处理扁平 DAG
4. **变量隔离约定** — 子图内部变量使用 `<subgraph-id>::` 命名空间；自动作用域改写依赖后续变量系统扩展
5. **模板库独立于核心 ADR** — 跨工程复用是分发层，不污染运行时数据模型

### 前端节点注册

```typescript
// 子图节点 — 容器
const SubgraphNodeRegistry = {
  type: 'subgraph',
  meta: {
    isContainer: true,
    size: { width: 560, height: 400 },
    padding: () => ({ top: 150, bottom: 100, left: 100, right: 100 }),
    defaultPorts: [
      { type: 'input' },
      { type: 'output' },
    ],
    selectable(node, mousePos) {
      if (!mousePos) return true;
      const transform = node.getData(FlowNodeTransformData);
      return !transform.bounds.contains(mousePos.x, mousePos.y);
    },
  },
  formMeta: {
    render: () => (
      <div>
        {/* 子图标题、参数配置面板 */}
        <SubCanvasRender />
      </div>
    ),
  },
};
```

### 桥接节点

FlowGram 容器的 `defaultPorts` 与内部子节点之间**没有连线通道**——`clearInvalidLines` 在拖入/拖出时切断跨容器连线。这是插件的固有行为，不是我们要绕过，而是要**在上面建立约定**。

引入两个特殊内部节点作为容器边界与内部子拓扑的接驳点：

```typescript
// subgraphInput — 接收容器 input port 数据，转发给内部子节点
const SubgraphInputRegistry = {
  type: 'subgraphInput',
  meta: {
    inputDisable: true,
    defaultPorts: [{ type: 'output' }],
  },
};

// subgraphOutput — 收集内部子节点输出，映射到容器 output port
const SubgraphOutputRegistry = {
  type: 'subgraphOutput',
  meta: {
    outputDisable: true,
    defaultPorts: [{ type: 'input' }],
  },
};
```

桥接节点的本质：**容器 `defaultPorts` 的内部代理**。前端展开后它们会成为扁平 DAG 中的普通节点；Rust 侧以 `PassthroughNode` 原样广播 payload，不做副作用。

### JSON 数据结构

```jsonc
{
  "id": "collect-modbus-1",
  "type": "subgraph",
  "meta": { "isContainer": true, "position": { "x": 400, "y": 200 } },
  "data": {
    "label": "Modbus 采集",
    "definitionRef": "collect-modbus@1.0.0",  // 可选：模板库引用
    "parameterBindings": {                      // 可选：参数化实例
      "mb_host": "192.168.1.10",
      "register": 40001
    }
  },
  "blocks": [
    { "id": "sg-in",   "type": "subgraphInput",  "meta": { ... } },
    { "id": "mb-read", "type": "modbusRead",      "data": { "config": { "host": "{{mb_host}}" } } },
    { "id": "sg-out",  "type": "subgraphOutput", "meta": { ... } }
  ],
  "edges": [
    { "sourceNodeID": "sg-in",   "targetNodeID": "mb-read" },
    { "sourceNodeID": "mb-read", "targetNodeID": "sg-out" }
  ]
}
```

### 展开规则

```
外部 DAG：
  timer ──→ [subgraph "collect-modbus-1"] ──→ sqlWriter
                 │                        │
              input:0                  output:0

子图内部 blocks + edges：
  sg-in ──→ mb-read ──→ sg-out

────────── 前端展开后（传给 Rust）──────────

  timer ──→ collect-modbus-1/sg-in ──→ collect-modbus-1/mb-read ──→ collect-modbus-1/sg-out ──→ sqlWriter

规则：
  1. 外部边 timer → subgraph.input:0    →  timer → <subgraph-id>/sg-in
  2. 外部边 subgraph.output:0 → sqlWriter  →  <subgraph-id>/sg-out → sqlWriter
  3. 内部节点 ID 加前缀 <subgraph-id>/<inner-node-id>
  4. 内部边保持
  5. 桥接节点在 Rust 侧为 passthrough（payload 直传）
  6. config 中的 {{paramName}} 被 parameterBindings 替换
```

### 展开实现（前端 `flowgram.ts`）

```typescript
function flattenSubgraphs(
  flowgramGraph: FlowgramWorkflowJSON,
  depth = 0,
): { nodes: WorkflowNodeDefinition[]; edges: EdgeDefinition[] } {
  if (depth > 8) throw new Error('子图嵌套超过 8 层上限');

  const flatNodes: WorkflowNodeDefinition[] = [];
  const flatEdges: EdgeDefinition[] = [];

  for (const node of flowgramGraph.nodes) {
    if (isContainerNode(node)) {
      const prefix = node.id;
      const params = node.data?.parameterBindings ?? {};

      // 递归展开
      const inner = flattenSubgraphs(
        { nodes: node.blocks ?? [], edges: node.edges ?? [] },
        depth + 1,
      );

      // 内部节点加前缀 + 参数替换
      for (const n of inner.nodes) {
        n.id = `${prefix}/${n.id}`;
        applyParameterBindings(n, params);
        flatNodes.push(n);
      }

      // 重写外部边：容器 port ↔ 桥接节点
      rewriteExternalEdges(flatEdges, node, inner);
    } else {
      flatNodes.push(convertNode(node));
    }
  }

  return { nodes: flatNodes, edges: flatEdges };
}
```

### Rust 侧影响

Runner 零改动。展开后 Runner 收到的是标准扁平 DAG：
- 桥接节点注册为 `PassthroughNode`（`transform` 直接返回 payload）
- 内部节点 ID 带 `/` 前缀，不影响 `NodeRegistry` 查找（按 `type` 查找，不按 `id`）
- 事件流中的 `trace_id` 和 `node_id` 保留完整路径，前端按 `/` 拆解显示

### 参数化

`config` 中用 `{{paramName}}` 占位，展开时替换：

```jsonc
// 子图定义中
{ "config": { "host": "{{mb_host}}", "register": "{{register}}" } }

// 展开时 parameterBindings = { "mb_host": "192.168.1.10", "register": 40001 }
// 结果
{ "config": { "host": "192.168.1.10", "register": 40001 } }
```

未绑定的参数保留原值（不替换），允许部分参数化。

### 变量作用域

子图内部变量约定使用 `<subgraph-id>::` 前缀命名空间（如 `collect-modbus-1::counter`），与父图 `WorkflowVariables` 隔离。

数据进出子图**只通过 pin**，不通过变量。这保证：
- 子图是纯函数式组合单元（输入 → 处理 → 输出）
- 嵌套子图不会产生变量命名冲突
- 部署期可静态分析子图的输入/输出类型契约

2026-04-28 的子图核心实施只落地展平、桥接、嵌套和参数绑定；变量命名空间仍是作者约定，自动校验 / 重写需在 ADR-0012 后续 scope 中单独设计。

### 嵌套上限

最大深度 **8 层**。`flattenSubgraphs` 递归时计数，超限报错。同时检测循环引用（ID 路径中出现相同 `definitionRef`）。

### 模板库（分发层，不属于本 ADR）

跨工程复用通过独立机制：
- 模板库存储子图定义（name / version / input_pins / output_pins / parameters / inner topology）
- 拖入时前端实例化为容器节点（填充 `blocks[]`）
- 实例化后是独立内联数据——模板升级不自动同步
- 存储方案（本地文件 / 项目库 / 远程）单独 ADR

## 可选方案

### 方案 A: 人工复制 + 模板文件

- 优势：零改动
- 劣势：
  - 每个副本独立，升级 N 处改动
  - 不是真正复用，无法形成组织级节点库

### 方案 B: 前端内联展开（FlowGram 子画布，已选）

- 优势：
  - Runner 零改动，单一执行模型不变
  - 复用已安装的 `free-container-plugin`，增量改动最小
  - 数据模型与 FlowGram 原生一致，自动持久化和渲染
  - 节点 ID 前缀清晰，可用标准 DAG 工具调试
- 劣势：
  - 展开后节点数膨胀（大子图 / 深嵌套）
  - 桥接节点是额外概念
  - 调试需从路径反推子图来源（需前端折叠显示）

### 方案 C: 运行时独立子 DAG（不展开）

- 子图作为运行时独立 DAG 实例，父/子 DAG 通过通道通信
- 优势：节点不膨胀；可独立启停/热更新
- 劣势：
  - Runner 必须认识嵌套 DAG——复杂度显著上升
  - 事件冒泡策略、ConnectionManager 共享/独立、资源生命周期全部复杂化
  - **不符合 FlowGram 子画布插件的内联容器哲学**

### 方案 D: 纯编辑器宏（拖入即展开）

- 编辑器层面拖入后立即展开为普通节点
- 优势：零后端改动
- 劣势：
  - 失去封装语义
  - 升级不跟进，不具备版本管理
  - 本质是方案 A 的自动化变体

## 后果

### 正面影响

- 模式复用正式化——可沉淀标准采集模板库、告警链模板
- Runner 零改动——前端展平，单一执行模型不变
- 复用已有基础设施——`free-container-plugin` 已安装，`flowgram.ts` 已处理 `blocks`/`edges`
- 参数化实例——同一模板不同参数部署多份
- 为 AI 生成子图铺路

### 负面影响

- `flowgram.ts` 增加递归展开逻辑（循环检测 + ID 前缀 + 参数替换 + 桥接映射）
- 前端需支持子图折叠视图和节点库分类（原子节点 vs 子图节点）
- 桥接节点是额外概念，需用户理解

### 风险

| 风险 | 缓解 |
|------|------|
| 循环引用（A → B → A） | 递归深度上限 8 + ID 路径循环检测 |
| 桥接节点数量不匹配容器 defaultPorts | 部署期校验：`subgraphInput` 数量 = 容器 input port 数 |
| 调试路径不直观（`collect-modbus-1/mb-read`） | 前端按 `/` 拆解高亮：`Modbus 采集 › mb-read` |
| 模板升级不自动同步已部署实例 | 加载工程时对比 `definitionRef` 版本号，提示可选升级 |

## 备注

- 依赖 ADR-0010 Pin 系统（已实施）——子图对外引脚必须是类型化的
- 实施建议：先**单层子图 + 无参数化**验证概念，再开启嵌套和参数化
- 变量隔离与 ADR-0012 一致：数据通过 pin 进出子图，变量 `<subgraph-id>::` 命名空间隔离
- FlowGram 官方子画布文档：https://flowgram.ai/zh/guide/free-layout/sub-canvas
- FlowGram 容器插件 API：`SubCanvasRender`、`SubCanvasBackground`、`SubCanvasBorder`、`SubCanvasTips`、`useNodeSize`、`useSyncNodeRenderSize`、`ContainerUtils`、`NodeIntoContainerService`
- FlowGram 容器节点关键 meta：`isContainer`、`size`、`padding`、`selectable`、`defaultPorts`、`subCanvas`
- 已实施范围（2026-04-28）：`nodes-flow` 注册 `subgraphInput` / `subgraphOutput` passthrough 节点；前端注册 `subgraph` 容器和桥接节点；`flowgram.ts` 在部署前递归展平 `blocks[]` / `edges[]`，支持 8 层嵌套、参数绑定替换和外部边桥接重写；AI 编排器通过节点能力目录自动读取 `NodeDefinition` / 默认 config 键 / 运行时注册表 / pin schema，并支持 `upsert_subgraph` 操作创建子图容器。
- Deferred：模板库、桥接节点数量与容器 port 数量的显式部署期校验、变量命名空间自动化。
