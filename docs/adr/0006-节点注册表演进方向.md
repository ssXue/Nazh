# ADR-0006: 节点工厂演进方向——从手写 match 到注册表

- **状态**: 已接受
- **日期**: 2026-04-09
- **决策者**: Niu Zhihong

## 背景

当前节点工厂 `graph/instantiate.rs` 使用手写 `match` 将 `node_type` 字符串映射到具体节点构造函数：

```rust
fn instantiate_node(def, conn_mgr) -> Result<Arc<dyn NodeTrait>> {
    match def.node_type.as_str() {
        "native" | "native/log" | "log" => { ... }
        "rhai" | "code" | "code/rhai" => { ... }
        "timer" => { ... }
        // ...11 个分支，每分支 ~8 行
        other => Err(UnsupportedNodeType(other)),
    }
}
```

现有 11 个节点类型。按路线图，后续将增加：

- Modbus Write、MQTT Pub/Sub、OPC-UA Read/Write（协议驱动阶段）
- AI Copilot 节点（脚本生成阶段）
- Aggregate / Window / Buffer 等数据处理节点

预计节点类型将增长到 20–30 个。当前的手写 match 模式面临：

1. 每新增一个节点需要修改 `instantiate.rs` 的 match 块——违反开闭原则。
2. match 块已触发 `#[allow(clippy::too_many_lines)]`，可读性下降。
3. 节点类型别名（`"native/log"` / `"log"`）散布在 match 分支中，无法被外部工具发现。

## 决策

> 我们决定**分两步演进**节点注册机制：近期通过提取辅助函数减少单分支样板；中期在节点数量达到 ~20 时引入注册表 crate 实现自动注册。

### 近期（本次重构）

提取 `resolve_description` 辅助函数，将每个 match 分支从 ~10 行降至 ~6 行。工厂函数整体行数从 ~175 行降至 ~120 行，暂时移除 `#[allow(clippy::too_many_lines)]`。

### 中期（节点数达到 ~20 时）

引入 `inventory` 或 `linkme` crate，让每个节点在定义时自注册：

```rust
// 每个节点文件中：
inventory::submit! {
    NodeRegistration::new("timer", &["timer"], |def, conn| {
        let config: TimerNodeConfig = parse_config(def)?;
        Ok(Arc::new(TimerNode::new(def.id.clone(), config, ...)))
    })
}

// instantiate.rs 中：
fn instantiate_node(def, conn) -> Result<Arc<dyn NodeTrait>> {
    let registry: Vec<&NodeRegistration> = inventory::iter::<NodeRegistration>().collect();
    for reg in &registry {
        if reg.matches(&def.node_type) {
            return (reg.factory)(def, conn);
        }
    }
    Err(UnsupportedNodeType(def.node_type.clone()))
}
```

这样新增节点只需在节点文件中添加注册宏，无需修改工厂函数。

## 可选方案

### 方案 A: 维持手写 match 不变

- 优势：最简单，无新依赖，IDE 跳转直接
- 劣势：match 块线性增长；每个新节点都要改工厂文件

### 方案 B: HashMap 注册表（手动注册）

```rust
let mut registry = HashMap::new();
registry.insert("timer", |def, conn| { ... });
```

- 优势：无新依赖
- 劣势：仍然需要一个集中注册点，只是从 match 变成 HashMap 初始化，没有根本改善

### 方案 C: `inventory` / `linkme` 自动注册（中期目标）

- 优势：真正的开闭原则——新节点无需修改任何现有文件
- 劣势：引入新依赖；编译期收集机制在某些平台（WASM）有限制

### 方案 D: Trait 方法 + `#[typetag]`

- 优势：基于 Rust trait 系统，最 "Rusty"
- 劣势：`typetag` 需要 `dyn Trait` 的序列化支持，与当前 JSON config 解析模式不兼容

## 后果

### 正面影响

- 近期：代码更整洁，每个分支样板减少
- 中期：新增节点类型从"改两个文件"变为"改一个文件"
- 节点类型别名集中管理，可被工具（如前端画布节点面板）自动发现

### 负面影响

- 近期改动最小，无显著负面影响
- 中期引入 `inventory` crate 增加一个编译依赖

### 风险

- 如果项目最终节点数量维持在 15 以下，注册表模式可能是过度设计
- 缓解措施：此 ADR 设定明确的触发条件（~20 个节点类型），在此之前不引入注册表
