# ADR-0002: 选择 Rhai 作为嵌入式脚本引擎

- **状态**: 已接受
- **日期**: 2025-01-01
- **决策者**: Niu Zhihong

## 背景

Nazh 的"动态脚本节点"需要让用户在前端编写业务逻辑代码，由引擎在沙箱中执行。
核心需求：

- 用户提交的代码不可信，必须有沙箱隔离。
- 必须能限制执行时间/步数，防止死循环拖死运行时。
- 需要能读写 JSON payload（`serde_json::Value`）。
- 语法应接近 JavaScript，降低工业用户学习成本。
- 不依赖外部运行时（V8、WASM 等），保持部署轻量。

## 决策

> 我们决定使用 Rhai 作为嵌入式脚本引擎，通过 `max_operations` 限制执行步数。

## 可选方案

### 方案 A: 嵌入 V8（via rusty_v8 / deno_core）

- 优势：完整的 JavaScript 支持，生态丰富
- 劣势：编译时间极长（V8 构建 > 30 分钟）；二进制体积膨胀 ~20MB；运行时内存占用高；与 Tokio 集成复杂

### 方案 B: WASM 沙箱（wasmtime / wasmer）

- 优势：强隔离，支持多语言编译为 WASM
- 劣势：用户需要额外的编译步骤（不能直接写脚本）；调试体验差；JSON 序列化需要跨 WASM 边界

### 方案 C: Rhai（已选）

- 优势：纯 Rust 实现，编译快；内置 `max_operations` 步数限制；语法类 JS，学习成本低；原生支持 `serde` JSON 互转；库体积 < 1MB
- 劣势：性能不如原生 Rust 或 V8 JIT；生态较小，第三方库少；不支持 `async`（脚本内不能做异步 I/O）

## 后果

### 正面影响

- 零外部依赖，Rhai 作为纯 Rust crate 直接编译
- `max_operations` 天然防止死循环，无需额外的看门狗线程
- JSON payload 通过 `to_dynamic` / `from_dynamic` 无缝进出 Rhai 环境
- 脚本在构造节点时编译（`Engine::compile`），运行时无编译开销

### 负面影响

- 脚本内无法做异步 I/O（如 HTTP 请求），这类操作必须由 NativeNode 完成
- Rhai 解释执行，性能约为原生 Rust 的 1/10 ~ 1/50

### 风险

- 如果未来用户脚本复杂度大幅增长，可能需要引入 WASM 作为补充
- 缓解措施：NodeTrait 抽象不绑定具体脚本引擎，后续可新增 WasmNode 类型
