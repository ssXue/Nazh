# ADR-0003: 前后端通信使用 Tauri IPC 而非 HTTP

- **状态**: 已接受
- **日期**: 2025-01-01
- **决策者**: Niu Zhihong

## 背景

Nazh 是一个桌面应用，需要 React 前端与 Rust 后端之间的通信机制。
核心需求：

- 低延迟：节点状态事件需要实时推送给前端做画布高亮。
- 双向通信：前端调用后端（部署工作流），后端主动推送事件给前端。
- 类型安全：参数和返回值应有明确的结构定义。
- 部署简单：边缘计算场景下不应依赖网络栈。

## 决策

> 我们决定使用 Tauri v2 的 `invoke`（前端→后端）和 `Window::emit`（后端→前端）作为唯一通信机制，不引入 HTTP/gRPC 服务。

## 可选方案

### 方案 A: 内嵌 HTTP 服务（Axum / Actix-Web）

- 优势：标准 REST/WebSocket 协议，前端可用 fetch 调用；方便外部工具集成
- 劣势：额外的端口占用和网络栈开销；在边缘设备上可能有防火墙限制；需要处理 CORS 等问题

### 方案 B: gRPC

- 优势：强类型、高性能、双向流
- 劣势：依赖 protobuf 编译工具链；前端需要 gRPC-Web 或代理层；对桌面应用过于重量级

### 方案 C: Tauri IPC（已选）

- 优势：进程内调用，零网络开销；`invoke` 自动处理序列化/反序列化；`emit` 原生支持事件推送；与 Tauri 窗口生命周期集成
- 劣势：绑定 Tauri 框架，前端单独运行时 IPC 不可用；不适合多客户端场景

## 后果

### 正面影响

- 前后端通信延迟 < 1ms（进程内），满足实时高亮需求
- 无端口冲突，无网络依赖，部署零配置
- `#[tauri::command]` 宏自动生成类型安全的 IPC 桥接

### 负面影响

- 前端脱离 Tauri 运行时（如纯浏览器开发）无法调用后端
- 缓解措施：`lib/tauri.ts` 中的 `hasTauriRuntime()` 检测函数做了优雅降级

### 风险

- 如果未来需要支持 Web 版（非桌面），需要补充 HTTP API 层
- 缓解措施：Tauri 命令函数内部调用 `nazh_engine` 的纯函数，引擎逻辑不绑定 Tauri
