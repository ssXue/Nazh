# 蓝图偏离项分析

> **状态：** 提议中
> **日期：** 2026-04-30
> **对照基准：** `docs/blueprints/ai_native_industrial_edge_workflow_review.md`（以下简称"蓝图文档"）
> **目的：** 逐项对照蓝图文档与当前项目（main 分支 7dd0d02）的实现状态，标记偏离项与偏离程度，为后续路线修正提供输入。

---

## 偏离项清单

### 偏离-01：无三段式 DSL（Device / Capability / Workflow）

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §7 三段式 DSL：`device.yaml` + `capability.yaml` + `workflow.yaml`，覆盖设备语义、能力抽象、业务状态机 |
| **当前状态** | 无自定义 DSL parser。工作流以 FlowGram JSON AST 透传部署。设备/能力建模完全缺失。Pin 声明系统（ADR-0010）部分覆盖信号/端口描述，但远未到 `DeviceSpec` / `CapabilitySpec` / `SignalSpec` 层级 |
| **偏离程度** | **大** |
| **影响** | 蓝图定位"DSL 是产品核心资产"，当前核心资产缺失 |
| **提议** | 提议中 |

---

### 偏离-02：无 Safety Compiler / 安全校验层

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §6.4 Safety Compiler 为"核心护城河"（P0），需校验 DSL 语法/Schema/单位一致性/量程边界/读写权限/协议映射/状态机完整性/危险动作审批/机械互锁/故障 fallback/资源占用/循环触发风险 等 17 项 |
| **当前状态** | 仅有 DAG 拓扑校验（Kahn cycle check）+ Pin 类型兼容矩阵（ADR-0010 Phase 3） |
| **偏离程度** | **大** |
| **影响** | 校验能力远低于蓝图最低要求，生成的工作流不可验证安全性 |
| **提议** | 提议中 |

---

### 偏离-03：无 edge-daemon / 边缘独立运行时

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §5/§10 推荐三可执行体：`desktop-tauri` / `edge-daemon` / `edge-cli`，daemon 负责现场确定性执行、断网自治、远程部署包接收 |
| **当前状态** | 仅 `desktop-tauri`（`nazh-desktop`），一切运行时逻辑嵌在 Tauri shell 进程内。无 daemon、无 CLI、无远程部署、无断网自治 |
| **偏离程度** | **大** |
| **影响** | 架构只有 1/3 可执行体，无法脱离桌面端独立运行 |
| **提议** | 提议中 |

---

### 偏离-04：无设备语义层 / 能力抽象层

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §6.1 设备语义层：将说明书/点表/寄存器表转换为结构化设备模型（`DeviceSpec`/`SignalSpec`/`AlarmSpec`）；§6.2 能力抽象层：将底层操作封装为受约束的设备能力（`CapabilitySpec`，含前置条件/副作用/fallback/安全等级） |
| **当前状态** | 节点直接绑定协议连接（`ConnectionManager` → `ConnectionGuard`），寄存器/信号级别元数据不在系统内。无 `DeviceSpec` / `CapabilitySpec` 类型 |
| **偏离程度** | **大** |
| **影响** | 工业语义理解层未启动，AI 无法生成结构化设备/能力模型 |
| **提议** | 提议中 |

---

| 维度 | 说明 |

| 维度 | 说明 |

### 偏离-09：AI 应用层深度不足

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §14 AI 负责：说明书抽取 → 设备建模 → 能力建模 → 工作流生成 → 测试用例 → 诊断辅助。AI 输出必须结构化（含 uncertainties/warnings） |
| **当前状态** | AI 仅覆盖脚本生成（`ai_complete()`）+ Copilot 流式补全。`AiService` trait + OpenAI 客户端管道就位。无 PDF 说明书解析、无设备模型生成、无能力建模、无测试用例生成、无结构化输出约束（uncertainties/warnings 字段） |
| **偏离程度** | **中** |
| **影响** | AI 管道在但应用层极浅，"AI 原生"定位尚未体现 |
| **提议** | 提议中 |

---

### 偏离-10：观测/审计层维度不足

| 维度 | 说明 |
|------|------|
| **蓝图要求** | §6.6 每次设备数据变化/条件触发/状态转移/动作调用/安全拒绝/人工审批必须可追踪。支持运行日志/工作流 trace/设备读写记录/报警记录/安全拒绝记录/审批记录/数据回放/仿真对比/AI 辅助诊断 |
| **当前状态** | `ExecutionEvent` + ADR-0016 边级可观测性（部分实施）覆盖了运行日志/trace/基本度量。无安全拒绝记录、无人工审批记录、无数据回放、无仿真对比 |
| **偏离程度** | **中小** |
| **影响** | 框架在但审计维度不足，工业可追溯性要求未满足 |
| **提议** | 提议中 |

## 对齐项（当前做得好的部分）

以下领域当前实现**达到或超出**蓝图基线：

| 领域 | 对齐说明 |
|------|----------|
| **技术栈** | Rust + Tauri + React 完全对齐蓝图 §8 |
| **Ring 分层** | Ring 0 / Ring 1 + feature gate + plugin 系统，比蓝图 §9 建议的 workspace 拆法更成熟 |
| **AI 安全边界** | `AiService` trait + Rhai 沙箱 + `max_operations` 落地"AI 不直接控制设备"原则（蓝图 §4/§14） |
| **RAII 资源管理** | `ConnectionGuard` / `LifecycleGuard` 超出蓝图预期 |
| **Pin 类型系统** | ADR-0010 兼容矩阵部分覆盖蓝图"单位校验/量程边界"方向 |
| **Panic 隔离** | `AssertUnwindSafe + catch_unwind + timeout` 比蓝图要求的更严格 |
| **IPC 类型契约** | `ts-rs` 自动生成确保 IPC 边界类型安全 |
| **协议支持** | Modbus / MQTT / Serial / HTTP / SQL / Bark 已落地，覆盖蓝图第一阶段协议要求 |
| **插件注册** | `NodeTrait` + `Plugin::register` + `standard_registry()` 实现蓝图 §5"核心无硬编码节点" |
| **工作流变量** | ADR-0012 Phase 1+2 实现蓝图"配置与数据分离"的 control plane |

---

## 总结

当前项目在**运行时引擎质量**（Ring 分层、插件系统、panic 隔离、类型安全、资源管理）上超出蓝图基线，但在**产品定义层**存在显著差距：

- **4 项大偏离**（三段式 DSL、Safety Compiler、edge-daemon、设备/能力建模）——构成蓝图"核心资产"和"核心护城河"
- **4 项中偏离**（仿真器、持久化、流处理、通信层）——影响产品完整度
- **2 项中小偏离**（AI 深度、审计维度）——框架在但深度不足

引擎造得好，但"从设备说明书和业务描述编译安全可执行的工业边缘工作流"这一核心产品闭环尚未启动。
