# RFC-0002: 分层内核与插件架构

- **状态**: 已接受
- **日期**: 2026-04-16
- **作者**: Niu Zhihong
- **关联**: 扩展 [RFC-0001](0001-节点插件化.md)（NodeRegistry 阶段一已实现），本 RFC 在此基础上定义完整的系统级架构

## 摘要

将 Nazh 引擎从单体 crate 重构为**分层内核（Layered Kernel）**架构。Ring 0 内核只包含 14 个不可替换的原语（DAG 调度、数据面、事件总线等），所有具体节点、脚本引擎、连接管理器均通过统一的 `Plugin` trait 以 Ring 1（编译期）或 Ring 2（WASM 运行时）方式接入。同时引入 **DataStore 控制面/数据面分离**，彻底解决大 payload 场景下的 clone 放大和背压级联问题。

## 动机

### 当前架构的问题

Nazh 引擎目前是一个单体 crate（`nazh_engine`），所有模块编译在一起。虽然内部耦合度已经较低（NodeRegistry 工厂模式、graph 不直接依赖 nodes），但存在以下根本性限制：

1. **无法扩展**：添加新节点类型必须修改引擎 crate，重新编译。用户无法提供自定义协议驱动。
2. **无法裁剪**：边缘设备不需要 SQLite 或 HTTP 客户端，但必须编译整个 `reqwest` 和 `rusqlite` 依赖树。
3. **内核边界模糊**："什么属于引擎核心"没有明确定义，导致核心代码直接依赖 `SharedConnectionManager` 等具体实现。
4. **性能瓶颈**：
   - 节点串行执行（每节点一次只处理一条消息）
   - `WorkflowContext` 在每条边上完整 clone（大 payload × 多下游 = 内存爆炸）
   - 事件通道满时静默丢弃
5. **竞态条件**：
   - Panic 导致连接永久锁死（`with_connection` 的 borrow/release 无 RAII 保护）
   - Deploy 操作的 TOCTOU 竞态（移除旧工作流和插入新工作流分两次加锁）

### 设计目标

- 定义清晰的内核边界，使核心代码零协议依赖
- 通过 Plugin trait 支持编译期和运行时两种扩展方式
- 通过架构原语（而非补丁）系统性消除性能瓶颈和竞态条件
- 保持单进程部署形态，不引入微服务/进程隔离的复杂性

## 详细设计

### 1. 总体架构：分层内核

#### 为什么不用微内核

真正的微内核（每个节点运行在独立 WASM 沙箱或子进程中）被排除，理由如下：

- **性能**：跨 WASM/进程的 serde 序列化-反序列化每次 ~10-50μs。工业边缘的采集周期可能是 10ms，I/O 开销占 0.5-5%，不可忽略。
- **隔离已有**：Rust 的所有权系统消除了数据竞态，`catch_unwind` 实现了 panic 隔离，`tokio::time::timeout` 实现了超时隔离。进程级隔离解决的问题已由语言特性覆盖。
- **部署复杂度**：边缘设备上管理多进程是运维灾难。一个静态链接的二进制是最可靠的部署形态。

#### 为什么不维持单体

单体 crate 的问题是**编译时耦合**：添加一个 Modbus 驱动需要重新编译整个引擎，不需要 SQLite 的设备也必须拉入 `rusqlite`。这在部署目标增多时会成为严重的工程瓶颈。

#### 选定方案：分层单体（Layered Monolith）

```
┌─────────────────── 单进程 ─────────────────────────────────┐
│                                                            │
│  ┌──────────────────────────────────┐                      │
│  │         nazh-core (Ring 0)       │  ← 内核              │
│  │  14 个原语（见下文）              │  零协议依赖           │
│  └──────────────────┬───────────────┘                      │
│                     │ Rust trait 边界（零开销抽象）          │
│  ┌──────────────────┴───────────────┐                      │
│  │       Ring 1: 编译期插件          │                      │
│  │  nazh-nodes-flow / nazh-nodes-io │  ← feature flag 可选 │
│  │  nazh-scripting / nazh-connections│                      │
│  └──────────────────┬───────────────┘                      │
│                     │ NodeRegistry::register()              │
│  ┌──────────────────┴───────────────┐                      │
│  │       Ring 2: 运行时插件          │                      │
│  │  WASM 模块 / Rhai 脚本文件        │  ← 热加载           │
│  └──────────────────────────────────┘                      │
└────────────────────────────────────────────────────────────┘
```

**关键约束**：`nazh-core` 的 `Cargo.toml` 中不可出现 `rhai`、`reqwest`、`rusqlite`、`serialport` 中的任何一个。如果出现了，说明内核边界画错了。

### 2. Ring 0 内核：14 个原语

"原语"的判断标准：**移除它意味着这不再是 Nazh。**

#### 2.1 执行面

**① NodeTrait** — 节点契约

```rust
#[async_trait]
pub trait NodeTrait: Send + Sync {
    fn id(&self) -> &str;
    fn kind(&self) -> &'static str;
    fn ai_description(&self) -> &str;

    /// 声明输入/输出引脚（类型、方向、是否必须）
    fn pins(&self) -> Vec<PinDefinition> { vec![] }

    /// 声明并发策略（默认串行）
    fn concurrency(&self) -> ConcurrencyPolicy { ConcurrencyPolicy::Sequential }

    /// 节点部署后、首次执行前调用
    async fn on_deploy(&self) -> Result<(), EngineError> { Ok(()) }

    /// 处理一个上下文引用，从 DataStore 读写数据
    async fn execute(
        &self,
        ctx_ref: ContextRef,
        store: &dyn DataStore,
    ) -> Result<NodeExecution, EngineError>;

    /// 节点取消部署时调用
    async fn on_undeploy(&self) -> Result<(), EngineError> { Ok(()) }
}
```

对比当前设计的变更：
- 新增 `pins()`：节点声明自己的输入/输出引脚和类型，支持部署时类型校验和前端端口渲染。
- 新增 `concurrency()`：节点声明自己的并发安全性，runner 据此选择执行模式。
- 新增 `on_deploy()` / `on_undeploy()`：支持需要持久连接（MQTT 订阅、WebSocket）的节点。
- `execute` 签名变更：接收 `ContextRef`（轻量引用）+ `DataStore`（数据面），而非完整的 `WorkflowContext`。

**② NodeRegistry** — 节点工厂（保持现有设计，不变）

**③ Plugin / PluginHost** — 插件生命周期

```rust
pub trait Plugin: Send + Sync + 'static {
    fn manifest(&self) -> PluginManifest;
    fn register(&self, ctx: &mut PluginContext);
    fn on_ready(&self) -> Result<(), EngineError> { Ok(()) }
    fn on_shutdown(&self) -> Result<(), EngineError> { Ok(()) }
}

pub struct PluginManifest {
    pub id: String,                       // "nazh.modbus"
    pub name: String,
    pub version: semver::Version,
    pub api_version: u32,
    pub contributions: Vec<Contribution>,
}

pub enum Contribution {
    NodeTypes(Vec<NodeTypeDeclaration>),
    ResourceDriver { protocol: String, description: String },
    ScriptEngine { language: String },
}

pub struct PluginContext<'a> {
    pub nodes: &'a mut NodeRegistry,
    pub resources: &'a mut ResourceRegistry,
    pub script_engines: &'a mut ScriptEngineRegistry,
    pub api_version: u32,
}
```

设计决策：
- Plugin 粒度是"一组相关能力"而非"单个节点"。一个 Modbus 插件可同时贡献 modbusRead 节点 + modbusWrite 节点 + modbus-tcp 连接驱动。
- `PluginContext` 是内核暴露给插件的受控操作界面，插件不能直接操作内核数据结构（类似 OS syscall 接口）。
- 编译期插件（Ring 1）和 WASM 插件（Ring 2）都实现同一个 `Plugin` trait，对内核透明。

#### 2.2 调度面

**④ DAG Topology** — 拓扑校验 + Kahn 排序（保持现有设计）

**⑤ Runner** — 节点执行循环

变更：支持 `ConcurrencyPolicy`。

```rust
pub enum ConcurrencyPolicy {
    /// 严格串行：一次一条。用于硬件 I/O、有状态节点。
    Sequential,
    /// 并发执行：最多 N 条消息同时处理。用于无状态计算、HTTP 请求。
    Parallel(usize),
}
```

Runner 根据策略选择执行模式：
- `Sequential`：当前 `while recv → execute → send` 循环（不变）。
- `Parallel(n)`：使用 `tokio::sync::Semaphore` 限流，每条消息 spawn 独立 task。

**⑥ EdgeBackpressure** — 边级背压策略

```rust
pub enum EdgeBackpressure {
    /// 阻塞上游直到下游有空间（当前默认行为）
    Block,
    /// 丢弃最新消息（当前 trigger lane 的行为）
    DropNewest,
    /// 丢弃最旧消息（仅保留最新值，适合仪表盘）
    DropOldest,
    /// 每 N 条取 1 条（降采样，适合高频采集 → 低频存储）
    Sample { every_n: usize },
    /// 满时路由到指定端口（溢出旁路）
    Overflow { port_id: String },
}
```

设计决策：
- 背压策略挂在**边**上而非**节点**上。同一个节点的不同下游边可以有不同策略：告警链路用 Block（不可丢）、仪表盘用 DropOldest（只要最新）、存储用 Sample（降采样）。
- 当前的 Trigger/Manual lane 背压策略移入内核的 EdgeBackpressure，Shell 层不再单独实现。

#### 2.3 数据面

**⑦ DataStore trait** — 数据存储契约

```rust
pub trait DataStore: Send + Sync + 'static {
    /// 写入一份数据，声明有多少消费者需要读取
    fn write(&self, payload: Value, consumers: usize) -> Result<DataId, EngineError>;
    /// 零拷贝读取（返回共享引用）
    fn read(&self, id: &DataId) -> Result<Arc<Value>, EngineError>;
    /// 读取可修改副本（Copy-on-Write：只在修改时 clone）
    fn read_mut(&self, id: &DataId) -> Result<Value, EngineError>;
    /// 消费完成，引用计数 -1。归零时自动释放
    fn release(&self, id: &DataId);
}
```

**⑧ ContextRef** — 控制面信号

```rust
pub struct ContextRef {
    pub trace_id: Uuid,
    pub timestamp: DateTime<Utc>,
    pub data_id: DataId,
    pub source_node: Option<String>,
}
// 大小 ≈ 64 bytes（对比当前 WorkflowContext 含完整 payload 可达数 KB-MB）
```

**⑨ ArenaDataStore** — 默认内存实现

基于 `DashMap<DataId, DataEntry>` + `Arc<Value>` + 原子引用计数。

设计决策与取舍：

| 决策 | 选择 | 排除方案 | 理由 |
|------|------|----------|------|
| 默认后端 | 内存（DashMap + Arc） | SQLite / sled | 内存比 SQLite 快 2000 倍。在途数据丢失可接受（传感器持续产生新数据）。磁盘写入消耗 flash 寿命。 |
| 是否提供磁盘后端 | Ring 1 可选插件 | 不提供 | 大 payload（视觉检测图片）或合规场景需要溢出到磁盘，通过 TieredDataStore 插件实现。 |
| 内存管理 | 引用计数 + 内存上限 + TTL GC | 纯引用计数 | 引用计数处理正常路径；内存上限防 OOM；TTL GC 兜底 panic 导致的引用泄漏。 |
| 扇出共享 | 同一 DataId 多消费者 | 每下游 clone | N 个下游共享同一份 Arc<Value>，零拷贝。通过 consumers 引用计数管理生命周期。 |

性能对比（1KB payload，5 节点 DAG，100 msg/s）：

| 后端 | 每消息开销 | 100 msg/s CPU |
|------|-----------|--------------|
| DashMap + Arc | ~350 ns | 0.004% |
| sled | ~75 μs | 0.75% |
| SQLite (WAL) | ~750 μs | 7.5% |

三种访问模式：

- **只读**（if/switch/debugConsole）：`store.read(id)` 返回 `Arc<Value>`，~1ns（原子引用计数 +1），零拷贝。
- **变换**（Rhai 脚本、数据注入）：`store.read_mut(id)` 真正 clone payload → 修改 → `store.write(new_payload)` → `store.release(old_id)`。
- **扇出**（1 输出 → N 下游）：`store.write(payload, N)` 设置 consumers=N，N 个下游各自 `read()` 获取 Arc 共享引用，处理完调 `release()`。最后一个 release 触发释放。

数据生命周期：

```
write(payload, consumers=3)
   → DataEntry { payload: Arc::new(payload), remaining: 3 }

downstream A: read() → Arc::clone (refcount 不变，DashMap entry 保持)
downstream A: release() → remaining: 3→2

downstream B: read() → Arc::clone
downstream B: release() → remaining: 2→1

downstream C: read() → Arc::clone
downstream C: release() → remaining: 1→0 → 从 DashMap 移除 → Arc refcount→0 → 内存释放
```

#### 2.4 资源面

**⑩ ResourceProvider trait** — 外部资源契约

```rust
pub trait ResourceProvider: Send + Sync + 'static {
    type Guard: ResourceGuard + Send;
    async fn acquire(&self, id: &str) -> Result<Self::Guard, EngineError>;
    fn snapshot(&self, id: &str) -> Option<ResourceSnapshot>;
}
```

设计决策：
- Ring 0 只定义 trait，不规定锁策略。Ring 1 的 ConnectionManager 可以选择 Mutex（简单场景）、DashMap 分片（高并发）或 Actor 模型（无锁）。
- 取代当前 `deploy_workflow` 签名中直接依赖的 `SharedConnectionManager` 具体类型。

**⑪ ResourceGuard (RAII)** — 自动释放保证

```rust
pub trait ResourceGuard: Send {
    fn id(&self) -> &str;
    fn metadata(&self) -> &Value;
    fn mark_success(&mut self);
    fn mark_failure(&mut self, reason: &str);
    // Drop 实现：如果未调用 mark_success/mark_failure，
    // 视为异常退出（panic），自动 release + 记录失败。
}
```

设计决策：用 RAII Drop 替代手动 borrow/release 配对。

当前问题：`with_connection()` 中如果闭包 panic，`release_lease()` 不会执行，连接 `in_use` 永久为 true（代码注释已承认此问题）。

解法：ResourceGuard 的 Drop 实现在任何退出路径（正常返回、错误返回、panic 展开）都会执行释放。Rust 所有权系统在编译期保证 Guard 一定会被 drop。这不是一个 bug fix，而是**让这类 bug 在类型层面不可能存在**。

#### 2.5 可观测面

**⑫ EventBus (broadcast)** — 生命周期事件广播

```rust
pub struct EventBus {
    sender: broadcast::Sender<ExecutionEvent>,
}
```

替代当前的 `mpsc::Sender<ExecutionEvent>`。

| 对比 | 当前 MPSC | 改后 Broadcast |
|------|----------|---------------|
| 消费者数量 | 1 个 | N 个（独立） |
| 慢消费者影响 | 阻塞生产者或静默丢弃 | 只影响自己（收到 `Lagged` 错误） |
| 多用途 | 不行 | 前端 UI / JSONL 日志 / 指标聚合各自独立订阅 |

### 3. 插件系统

#### 3.1 三种加载机制

**Ring 1：编译期插件（Rust Crate）**

用于标准库节点和官方协议驱动。通过 `impl Plugin` 向 NodeRegistry 注册节点工厂。门面 crate `nazh-engine` 通过 Cargo feature flag 控制各 Ring 1 插件的编译。

```toml
# nazh-engine/Cargo.toml
[features]
default = ["flow", "io", "scripting", "connections"]
flow = ["dep:nazh-nodes-flow"]
io = ["dep:nazh-nodes-io"]
scripting = ["dep:nazh-scripting"]
connections = ["dep:nazh-connections"]
```

**Ring 2：WASM 插件**

用于用户自定义节点。通过 WIT (WebAssembly Interface Type) 定义宿主-插件 ABI。宿主提供的 host 函数：`log`、`acquire-resource`、`release-resource`、`get-variable`、`set-variable`。插件导出：`list-node-types`、`execute`。

WASM 插件的安全模型是**声明式权限**：插件在清单中声明所需能力（网络、文件系统、特定资源 ID），宿主在运行时强制执行。

**Ring 2 变体：Rhai 脚本插件**

当前 `RhaiNode` 的延伸。一个 `.rhai` 文件 + `nazh-plugin.toml` 清单文件 = 一个可发现的脚本插件。

#### 3.2 插件依赖关系

```
nazh-core（Ring 0 内核）
    ↑
nazh-scripting（Rhai 引擎，impl ScriptEngine）
    ↑
nazh-nodes-flow（if/switch/loop/tryCatch，依赖 Rhai 做脚本求值）

nazh-core
    ↑
nazh-connections（ConnectionManager，impl ResourceProvider）
    ↑ （可选依赖）
nazh-nodes-io（native/timer/serial/modbus/http/sql/debug）

nazh-core
    ↑
nazh-pipeline（线性流水线，独立于 DAG）
```

两条依赖链互相独立：Flow→Scripting 和 IO→Connections。

### 5. 性能与竞态的架构解法

以下问题通过上述架构原语在**设计层面消除**，而非通过代码补丁修复：

| 问题 | 根因 | 解决它的原语 | 原理 |
|------|------|-------------|------|
| 节点串行执行瓶颈 | runner 只有一种执行模式 | `NodeTrait::concurrency()` | 节点声明自己的并发策略，runner 自动适配 |
| 背压级联到根节点 | DAG 内部的边只有 Block 策略 | `EdgeBackpressure` | 每条边独立策略（Block/Drop/Sample/Overflow） |
| 大 payload clone 放大 | Channel 搬运完整数据 | `DataStore` + `ContextRef` | Channel 只传 64B 引用，数据零拷贝共享 |
| 事件静默丢弃 | MPSC 单消费者，满则丢 | `EventBus` (broadcast) | 多消费者独立速度，慢消费者不阻塞快消费者 |
| 连接池高并发锁竞争 | ConnectionManager 锁策略固定 | `ResourceProvider` trait | Ring 1 可自由选择锁策略（Mutex/分片/Actor） |
| Panic 连接永久锁死 | 手动 borrow/release 无 RAII | `ResourceGuard` (Drop) | 类型系统保证释放，panic 路径也执行 Drop |
| Deploy 并发 TOCTOU | 两次加锁之间有窗口 | Shell 层状态机 | WorkflowPhase 枚举 + 单锁原子转换 |

## 缺点与风险

### 已知缺点

- **Workspace 拆分增加构建复杂度**：从 1 个 crate 变为 8 个，CI 配置和依赖管理更复杂。
- **DataStore 引入间接层**：节点不再直接操作 payload，需要通过 `store.read()` / `store.write()`，代码略微冗长。
- **WASM 插件的序列化开销**：跨 WASM 边界传递 payload 仍需 JSON 序列化（~10-50μs/次），但 DataStore 可以缓解（只传 data_id，宿主代读代写）。

### 风险

- **过度抽象**：如果实际上只有 1 种 ResourceProvider 实现，trait 抽象就是多余的复杂度。缓解措施：初期只实现 ArenaDataStore 和 ConnectionManager，不过度设计。
- **Plugin API 稳定性**：一旦外部 WASM 插件依赖了 API，变更代价极高。缓解措施：Ring 2 通过 `api_version` 字段做版本兼容，初期不急于发布 WASM 支持。
- **DataStore GC 延迟**：TTL GC 可能在内存紧张时不够及时。缓解措施：内存上限硬顶 + 写入时检查，满时拒绝写入并触发 EdgeBackpressure。

## 替代方案

### 方案 A: 真微内核

每个节点运行在独立 WASM 沙箱或子进程中，通过 IPC 通信。

已排除。理由：
- 性能不可接受（每次节点调用 10-50μs IPC 开销）
- 部署复杂（边缘设备管理多进程/WASM runtime）
- Rust 已提供足够的故障隔离

### 方案 B: 维持单体 + Feature Flag

不拆分 crate，只用 `cfg(feature)` 控制编译。

已排除。理由：
- Feature flag 不能表达 "用户自定义节点" 的需求
- 单 crate 内的模块边界太容易被打破（一次 `use crate::xxx` 就破坏了解耦）
- Workspace 的 crate 边界是编译器强制的，比约定更可靠

### 方案 C: Channel 直接传 Arc<Value>

不引入 DataStore，改为 Channel 传递 `Arc<Value>` 而非 `Value`。

```rust
pub struct WorkflowContext {
    pub trace_id: Uuid,
    pub timestamp: DateTime<Utc>,
    pub payload: Arc<Value>,
}
```

已排除。理由：
- 解决了 clone 放大（Arc::clone 是引用计数 +1），但失去了：
  - 按 ID 查找历史数据的能力（调试、回放）
  - 内存上限控制（无法统计总内存占用）
  - 磁盘溢出的扩展点（没有 DataStore trait）
  - WASM 插件的数据代理能力
- DataStore 方案包含了 Arc 方案的所有收益，且提供了更多扩展能力，额外开销仅为一次 DashMap lookup（~50ns）

## 未解决的问题

以下细节留待实现阶段决定：

1. **Pin 类型系统的具体类型枚举**：`PinType` 应该包含哪些类型？是否需要 `Binary`（原始字节）和 `Custom(String)`（扩展类型）？
2. **WASM WIT 接口的确切定义**：host 函数集合和 guest 导出函数集合的最终版本。
3. **EdgeBackpressure::Sample 的实现位置**：是在 runner 中实现，还是在一个独立的 "edge processor" 中间层实现？
4. **DataStore 的内存上限默认值**：256MB？可配置？还是按 DAG 动态计算？
5. **Plugin manifest 的 TOML schema 细节**：WASM 插件的 `nazh-plugin.toml` 的完整字段定义。
6. **NodeTrait::execute 的签名是否需要 ResourceProvider 参数**：是显式传入还是通过 PluginContext 注入到节点构造时？

## 备注

- 本 RFC 的讨论背景参见 ADR-0001（Tokio MPSC DAG 调度）—— DataStore 方案是 ADR-0001 "负面影响"中提到的"大 payload clone 开销"的架构级解法。
- 本 RFC 参考了 Unreal Engine 5 Blueprint 系统（Pin 类型系统、插件模块化）、Linux 内核（分层架构、Ring 模型）、Kafka（Control Plane / Data Plane 分离）的设计思想。
