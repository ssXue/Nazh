# RFC-0001: 节点插件化

- **状态**: 实施中
- **日期**: 2026-04-15
- **作者**: Niu Zhihong

## 摘要

将 Nazh 引擎的节点体系从硬编码 `match` 改为注册式架构，使引擎核心不依赖任何具体节点类型。所有内置节点变为"标准库"（预注册），外部能力通过 Sidecar 进程插件接入，用户脚本通过 Rhai 脚本包复用——三种来源共享同一个 `NodeRegistry` 入口。

## 动机

当前每新增一种节点类型，需要改动 5 个文件（节点实现 → `nodes/mod.rs` → `lib.rs` → `instantiate.rs` match → 前端类型映射）。核心引擎与具体节点实现紧耦合。

更重要的是，部分能力扩展（如 OpenCV 视觉检测、MQTT 协议驱动、OPC UA 接入）无法编译进引擎：
- 引擎 `unsafe_code = "forbid"`，而 OpenCV / 动态库加载 (FFI) 都需要 `unsafe`
- 不同部署环境需要的节点集合不同，不应强制所有环境都编译全部依赖
- 外部贡献者应能在不改动引擎源码的情况下扩展节点

## 详细设计

### 阶段一：NodeRegistry（已完成 ✅）

将硬编码的工厂 `match` 替换为运行时注册表。

#### 接口设计

```rust
/// 节点工厂函数签名
type FactoryFn = dyn Fn(&WorkflowNodeDefinition, SharedConnectionManager)
    -> Result<Arc<dyn NodeTrait>, EngineError> + Send + Sync;

/// 节点注册表
pub struct NodeRegistry {
    factories: HashMap<String, Arc<FactoryFn>>,
}

impl NodeRegistry {
    pub fn new() -> Self;
    pub fn with_standard_nodes() -> Self;           // 预注册所有内置节点
    pub fn register<F>(&mut self, name, factory);   // 注册工厂
    pub fn alias(&mut self, alias, canonical);       // 添加别名
    pub fn create(&self, def, cm) -> Result<Arc<dyn NodeTrait>>;
    pub fn registered_types(&self) -> Vec<&str>;
}
```

### 阶段二：SidecarNode（待实现）

引入通用的 `SidecarNode`，通过子进程 + JSON stdio 协议接入外部能力。

#### 接口设计

```rust
/// Sidecar 节点配置（写在工作流 AST 中）
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SidecarNodeConfig {
    /// 可执行文件路径（相对于 plugins/ 目录或绝对路径）
    pub command: String,
    /// 启动参数
    #[serde(default)]
    pub args: Vec<String>,
    /// 单次执行超时（毫秒）
    #[serde(default = "default_sidecar_timeout")]
    pub timeout_ms: u64,
    /// 进程生命周期策略
    #[serde(default)]
    pub lifecycle: SidecarLifecycle,
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub enum SidecarLifecycle {
    /// 每次执行启动新进程（最安全，适合不稳定的外部工具）
    #[default]
    PerExecution,
    /// 长驻进程，工作流生命周期内复用（性能更好）
    Persistent,
}
```

#### 通信协议

采用 **JSON-line over stdio**（最简单、跨语言、无依赖）：

```
Engine → Sidecar (stdin):
{"trace_id":"...","payload":{...}}

Sidecar → Engine (stdout):
{"payload":{...}}

Sidecar → Engine (stdout, 错误):
{"error":"描述信息"}
```

每条消息为一行 JSON，以 `\n` 分隔。

#### 使用示例

```json
{
  "detect_defect": {
    "type": "sidecar",
    "config": {
      "command": "./plugins/opencv-worker",
      "args": ["--model", "defect-v3"],
      "timeout_ms": 5000,
      "lifecycle": "persistent"
    },
    "ai_description": "使用 OpenCV 检测产品外观缺陷"
  }
}
```

#### 实现要点

- SidecarNode 实现 `NodeTrait`，在 `execute()` 中：
  1. 启动（或复用）子进程
  2. 将 `WorkflowContext.payload` 写入 stdin
  3. 从 stdout 读回结果（带超时）
  4. 子进程崩溃/超时 → 返回 `EngineError`，不 panic
- 长驻进程需要健康检查：若 stdout 关闭，标记为不可用并重启
- 子进程的 stderr 收集为日志，通过 `ExecutionEvent` 上报

#### 注册方式

```rust
// 引擎启动时自动注册
registry.register("sidecar", |def, _cm| {
    let config: SidecarNodeConfig = def.parse_config()?;
    Ok(Arc::new(SidecarNode::new(def.id.clone(), config, ...)?))
});
```

### 阶段三：插件发现与声明（待设计）

让用户通过配置文件声明 Sidecar 插件，引擎启动时自动扫描注册。

#### 初步方案

```toml
# plugins/opencv-worker/plugin.toml
[plugin]
name = "opencv-worker"
version = "0.1.0"
node_type = "opencv/detect"
command = "./opencv-worker"
description = "基于 OpenCV 的产品缺陷检测"

[plugin.config_schema]
# JSON Schema 约束，前端可据此渲染配置面板
model = { type = "string", required = true }
confidence = { type = "number", default = 0.8 }
```

引擎扫描 `plugins/*/plugin.toml`，为每个插件注册对应的 `SidecarNode` 工厂。

## 缺点与风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| Sidecar 进程 IPC 延迟（~500μs/次） | 不适合高频热路径 | 流程原语保持进程内执行，仅 I/O 能力走 Sidecar |
| 子进程崩溃 | 节点执行失败 | SidecarNode 负责重启 + 错误上报，不影响其他节点 |
| 插件安全性 | 恶意插件可能读写文件系统 | 插件目录白名单 + 运行时权限声明（未来可引入 seccomp/AppArmor） |
| 注册表是 `&mut self` | 运行时不能动态注册 | 当前阶段足够；若需运行时注册，改为 `RwLock<HashMap>` |

## 替代方案

| 方案 | 排除理由 |
|------|---------|
| 动态库 `.so` 插件 | 引擎 `unsafe_code = "forbid"`，无法使用 `libloading` |
| WASM 插件 | 无法访问摄像头/串口等硬件，图像处理性能差 |
| 嵌入 Python 解释器 | 增加巨大依赖，与 Rhai 沙箱职责重叠 |
| gRPC 替代 stdio | 增加依赖（tonic/protobuf），对简单场景过度设计 |

## 未解决的问题

- Sidecar 进程的资源限制（内存、CPU）如何实施？是否需要 cgroup 集成？
- 长驻 Sidecar 进程的优雅关闭协议（SIGTERM → 等待 → SIGKILL）的超时策略？
- 插件声明中的 `config_schema` 如何与前端画布的配置面板对接？
- 是否需要插件版本管理和依赖解析？
