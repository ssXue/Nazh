# RFC-0005: AI 调用前移到前端

- **状态**: 提案中
- **日期**: 2026-05-12
- **作者**: ssxue
- **关联**: 取代 ADR-0019 的 trait-in-Ring-0 路线；涉及 `crates/ai/`、`crates/core/src/ai.rs`、`crates/scripting/`、`crates/nodes-flow/`、`src-tauri/src/commands/ai.rs`、`web/src/`

## 摘要

将所有 AI HTTP 调用从 Rust 引擎层移到前端（React / TypeScript），让 AI 调用发生在"上下文所在、输出所往"的地方。Rust 引擎不再持有 `AiService` trait 和 HTTP 客户端；废弃 Rhai 脚本中的 `ai_complete()` 自由函数，改为通过专用 AI 推理节点（`aiInfer`）将 AI 能力固化为 DAG 中的显式一等公民。

## 动机

### 上下文错位

当前 AI 调用链为：

```
前端（拥有对话历史、画布状态、编辑器选区）
  → Tauri invoke 序列化
  → Rust AiService HTTP 调用
  → Tauri emit 反序列化
  → 前端渲染
```

Rust 在这条链里是纯转发层——它既不产生 AI prompt 的上下文，也不消费 AI 的输出。IPC 序列化/反序列化在对话历史较长时带来不必要的延迟和内存开销。

### 单进程架构的事实

Nazh 是 Tauri v2 桌面单进程应用。不存在 server 部署、headless agent、远程 runner 等场景。前端 webview 和 Rust 进程同生共死——"前端没打开所以需要后端 AI 能力"的前提不成立。

### 多 provider 的现实

当前 `crates/ai/` 只有一个实现（`OpenAiCompatibleService`，基于 `async-openai`）。每新增一个 provider（Anthropic 原生、Google Gemini、本地 Ollama）都需要在 Rust 侧新增 crate + 实现 `AiService` trait，维护成本随 provider 数量线性增长。

前端生态已有成熟的 AI SDK（Vercel AI SDK、`@ai-sdk/anthropic`、`@ai-sdk/openai` 等），多 provider 支持开箱即用，provider 适配逻辑由社区维护。

### Ring 0 膨胀

ADR-0019 将 `AiService` trait + 14 个关联类型（`AiCompletionRequest`、`AiMessage`、`StreamChunk`、`AiToolDefinition`、`AiToolCall`、`AiToolResult`、`AiError` 等）上移到 Ring 0（`nazh-core::ai`），占据约 370 行。这些类型本质上是"AI 协议建模"，与 Ring 0 的"引擎内核抽象"定位（`NodeTrait`、`DataStore`、`ExecutionEvent`）无关。

### 引擎侧 AI 的实际用途

| 场景 | 上下文来源 | 输出去向 | Rust 是否不可替代 |
|------|-----------|---------|:---:|
| Copilot 对话 | 前端会话历史 | 前端聊天面板 | 否 |
| 脚本生成（code 节点） | 前端编辑器 | 前端代码编辑器 | 否 |
| 工作流 AI 编排 | 前端画布状态 | 前端画布 diff | 否 |
| 设备 PDF 提取 | 前端触发，文件通过 IPC 可读 | 前端 DSL 编辑器 | 否 |
| 能力生成 | 前端触发，YAML 通过 IPC 可读 | 前端资产编辑器 | 否 |
| Rhai `ai_complete()` | 工作流运行时 | 脚本返回值 | 否——设计缺陷，见下方分析 |
| Embedding | 暂无实际使用 | — | — |

**Rhai `ai_complete()` 应当直接移除**。理由：

1. **破坏确定性**：相同脚本输入因 AI 输出不同而结果不同，无法复现、无法测试
2. **隐式 I/O 依赖**：`code` 节点在 DAG 拓扑中表现为纯计算节点，但运行时偷偷发起 HTTP 调用——部署时无法校验、拓扑无法反映
3. **无界延迟**：AI 调用耗时不可控，但脚本节点没有超时声明机制，阻塞整个执行路径
4. **违背设计原则**：AGENTS.md 设计原则 3（"设备语义高于协议适配"）同样适用于 AI——AI 能力应通过显式节点暴露，而非脚本中的自由函数
5. **与 `NodeTrait` 契约矛盾**：`transform(trace_id, payload) → NodeExecution` 是确定性数据变换，不应包含不可控的外部服务调用
6. **干扰 AI 决策**：保留此 API 会给 AI agent（Copilot、代码生成）释放错误信号，诱导其在生成的脚本中调用 `ai_complete()`，制造不可靠的工作流

正确的做法是引入专用 AI 推理节点（如 `aiInfer`），像 `capabilityCall` 封装设备能力一样封装 AI 能力——有明确的输入/输出 pin、provider 配置、超时声明，在 DAG 拓扑中可见、可校验、可监控。本 RFC 同步移除 `ai_complete()`；`aiInfer` 节点的设计与实现作为独立后续工作。

## 需求与约束

### 硬性需求

1. 所有 AI provider 的 HTTP 调用在前端发起，Rust 引擎不持有 HTTP 客户端
2. 前端直调 AI 时的 streaming 延迟不高于当前的 Rust→Tauri emit 路径
3. API key 不明文暴露在 webview 的全局 JS 空间

### 约束

1. 不引入 `@tauri-apps/plugin-http` 以外的网络层依赖——CORS 通过 Tauri fetch 插件绕过
2. 前端 AI 模块的类型导出仍通过 `ts-rs` 与 Rust 侧类型对齐（请求/响应类型保留在 Rust，仅删除 HTTP 客户端实现）
3. ADR-0019 的 `AiService` trait 从 Ring 0 移除，相关类型迁移到合适位置
4. `crates/ai/` crate 最终删除或保留为最小配置管理层

### 非目标

- 不改变工作流执行引擎的核心架构（`NodeTrait`、`DataStore`、`ExecutionEvent` 不变）
- 不改变 Tauri IPC 的整体通信模式（仅改变 AI 调用方向）
- 不引入 AI agent 框架（LangChain、CrewAI 等）

## 设计

### 整体架构变更

```
变更前：
  前端 → invoke("copilot_complete") → Rust AiService → HTTP → Provider
  前端 ← emit("copilot://stream/{id}") ← Rust SSE ← Provider
  Rhai ai_complete() → Rust AiService → HTTP → Provider

变更后：
  前端 → fetch(Provider API) → Provider
  前端 ← SSE Response ← Provider
  ai_complete() → 移除（后续由 aiInfer 节点替代）
```

### Phase 1：前端 AI 基础设施

引入 Vercel AI SDK（`ai` + `@ai-sdk/openai` + `@ai-sdk/anthropic` 等 provider 包）作为前端 AI 调用的统一入口。

#### 1.1 Provider 注册表

```typescript
// web/src/ai/providers.ts

interface AiProviderAdapter {
  id: string;
  name: string;
  createLanguageModel(config: AiProviderConfig): LanguageModel;
  testConnection(config: AiProviderConfig): Promise<boolean>;
}

// 基于 ai-sdk 的 provider 工厂
const registry: Map<string, AiProviderAdapter> = new Map([
  ["openai", { ... createOpenAI, ... }],
  ["anthropic", { ... createAnthropic, ... }],
  ["deepseek", { ... createOpenAICompatible, ... }],
  ["moonshot", { ... createOpenAICompatible, ... }],
  ["ollama", { ... createOpenAICompatible, ... }],
]);
```

#### 1.2 API Key 安全

> **2026-05-16 更新**：本 RFC 早期写法把 `AiProviderSecretRecord` 描述为加密存储。当前实现和决策均为本机明文配置 + 最小传播面，不接入 OS keychain / 自建加密 vault。根 `AGENTS.md` 是最终安全边界说明。

API key 存储在 Tauri shell 层的 `AiProviderSecretRecord`，前端配置读取只拿到 `AiProviderView.has_api_key`。完整 key 只能通过 `load_ai_api_key(provider_id)` 按需读取。

读取后立即用于构造 provider 实例，不暴露到 `window` 或全局 store。

#### 1.3 Streaming 替换

当前 streaming 链路：

```
invoke("copilot_complete_stream") → Rust → async-openai SSE → emit("copilot://stream/{id}") → 前端 listener
```

替换为 ai-sdk 原生 streaming：

```typescript
import { streamText } from "ai";

const result = streamText({
  model: provider.createLanguageModel(config),
  messages,
  ...params,
});

for await (const chunk of result.textStream) {
  // 直接更新 React state，无 IPC 中转
  updateUi(chunk);
}
```

#### 1.4 Copilot 迁移

`copilot-stream.ts` 当前基于 Tauri event listener 拼装 stream chunk。迁移后改为直接消费 ai-sdk 的 `textStream` / `fullStream`，删除 Tauri event 中间层。

### 未来方向：`aiInfer` 专用节点

当前 Rhai 脚本中的 `ai_complete()` 存在设计缺陷（破坏确定性、隐式 I/O、无界延迟），正确的做法是引入专用 AI 推理节点（如 `aiInfer`），像 `capabilityCall` 封装设备能力一样封装 AI 能力。该节点应有明确的输入/输出 pin（`prompt` → `result` / `error`）、provider 配置、超时声明，在 DAG 拓扑中可见可校验。执行时通过 IPC 回调前端代为调用 AI provider。

本 RFC 不实施此节点，留作独立后续工作。当前 `ai_complete()` 在 AI 调用前移后仍继续工作（仍由 Rust 侧现有的 `AiService` 实现，直至 `aiInfer` 节点就绪后统一废弃）。

- [ ] 迁移 `extract_device_from_pdf`（Rust 保留 PDF 解析，AI 调用移前端）
- [ ] 迁移 `extract_device_proposal` / `extract_device_proposal_stream`
- [ ] 迁移 `generate_capabilities_from_device_cmd`
- [ ] 迁移 `generate_pin_schema`

### Phase 3：Rust 侧清理

#### 3.1 `crates/ai/` 重定位

- 删除 `OpenAiCompatibleService`（HTTP 客户端实现）
- 删除 `ScriptAiRuntime` 及 `ai_complete()` 函数
- 删除 `async-openai`、`reqwest`（AI 专用）、`futures-util`（AI 专用）依赖
- 保留配置型类型（`AiConfigFile`、`AiProviderDraft`、`AiProviderView` 等）——这些仍由 Tauri shell 管理（本机明文配置、provider 配置 CRUD）
- `AiSecretInput` / 明文传播面收敛逻辑留在 Rust（安全边界）

#### 3.2 Ring 0 清理

从 `crates/core/src/ai.rs` 移除：

- `AiService` trait（不再有 Rust 侧实现）
- `AiEmbeddingRequest` / `AiEmbeddingResponse`（暂无使用）
- `StreamChunk`（流式类型改为前端内部类型）

保留并迁移到 `crates/tauri-bindings/`：

- `AiCompletionRequest` / `AiCompletionResponse`
- `AiMessage` / `AiMessageRole` / `AiToolDefinition` / `AiToolCall` / `AiToolResult`
- `AiGenerationParams` / `AiThinkingConfig` / `AiThinkingMode` / `AiReasoningEffort`
- `AiError`（简化为 IPC 错误传递用）
- `AiTokenUsage`

这些类型仍然是 `#[ts(export)]` 的——前端通过生成的 TS 类型与 Rust 保持对齐。

#### 3.3 Tauri 命令调整

| 命令 | 处置 |

| `copilot_complete` | 删除，前端直调 |

| `load_ai_config` / `save_ai_config` | 保留，配置管理仍在 Rust |

## 备选方案考虑

### 方案 A：前端全权（本提案）

所有交互式 AI（Copilot / 脚本生成 / 设备提取 / 能力生成）的 HTTP 调用移到前端。移除 `ai_complete()`，后续由 `aiInfer` 节点替代。

- 优势：上下文对齐；多 provider 开箱即用；Ring 0 完全瘦身；移除破坏确定性的 API；streaming 延迟降低
- 劣势：引入前端 AI SDK 依赖；已有使用 `ai_complete()` 的脚本需要迁移；`aiInfer` 节点未就绪前工作流中无 AI 推理能力

### 方案 B：维持现状（ADR-0019 路线）

保持 `AiService` trait 在 Ring 0，`crates/ai/` 作为实现层，保留 `ai_complete()`。

- 优势：零迁移成本
- 劣势：Rust 是纯转发层；多 provider 维护成本高；`ai_complete()` 继续破坏工作流确定性并干扰 AI 决策

### 方案 C：混合模式——交互式 AI 前移，`ai_complete()` 保留 Rust

Copilot / 脚本生成等走前端；`ai_complete()` 保留 Rust HTTP 客户端实现。

- 优势：无需 IPC 回调；`ai_complete()` 延迟最低
- 劣势：AI 逻辑分裂到两层；provider 配置双重维护；`ai_complete()` 的确定性问题未解决且继续干扰 AI 决策

### 方案 D：引入 sidecar 进程处理 AI

AI 调用放在独立的 sidecar 进程（Node.js / Bun），Rust 和前端都通过 IPC 与之通信。

- 优势：AI 逻辑完全解耦；可用完整的 Node.js AI 生态
- 劣势：架构复杂度大幅增加（进程管理、通信、错误恢复）；对桌面应用过度设计

## 风险与未知

### 风险

1. **前端 AI SDK 的 browser 兼容性**
   - ai-sdk 主要面向 Node.js，browser 使用可能有坑（如 `ReadableStream` API 差异）
   - 缓解：Tauri webview 基于 WebKit/Chromium，现代 Web API 支持良好；`@tauri-apps/plugin-http` 提供 fetch polyfill

2. **API key 暴露面扩大**
   - key 虽不在 `window` 上，但 webview 进程内存中可访问
   - 缓解：桌面单用户场景，风险等同于当前 Rust 进程持有 key；Tauri 的安全模型在桌面场景下足够

3. **ai-sdk 供应商锁定**
   - 引入 ai-sdk 后 provider 适配逻辑依赖其生态
   - 缓解：ai-sdk 的 provider 接口是标准化的（`LanguageModelV1`），且是开源项目；必要时可替换为裸 `fetch`

4. **`aiInfer` 节点未就绪前工作流中无 AI 推理能力**
   - 移除 `ai_complete()` 后，在 `aiInfer` 节点实现之前，工作流无法在工作流运行时调用 AI
   - 缓解：`aiInfer` 作为独立后续工作尽快跟进；此空窗期对当前用户影响有限（`ai_complete()` 目前使用率极低）

### 未解决的问题

1. **ai-sdk 的选择**：是否必须使用 Vercel AI SDK，还是用裸 `fetch` + 自建 provider 适配？前者开箱即用但引入依赖，后者更轻但维护成本更高。建议 Phase 1 初期用 ai-sdk 验证可行性，如果 bundle size 或兼容性有问题再降级为裸 `fetch`。

2. **会话持久化的归属**：Copilot 会话历史当前在 Rust 侧管理（SQLite）。AI 调用前移后，会话持久化是否也跟着移到前端（IndexedDB）？还是保留 Rust 管理？建议保留 Rust 管理（已有 SQLite 实现，且与 store crate 的持久化策略一致）。

3. **`test_ai_provider` 的处置**：测试 provider 连通性可以在前端直调（发一个最小请求），也可以保留 Rust 端的 `test_connection`。前者更简洁，后者在配置阶段多一层保障。建议前端直调，失败时返回具体错误。
