# RFC-0004：三段式 DSL——Device / Capability / Workflow

> **状态：** 提议中
> **日期：** 2026-04-30
> **决策者：** ssXue
> **关联：** 偏离-01（三段式 DSL）、偏离-02（Safety Compiler）、偏离-04（设备语义 / 能力抽象）、ADR-0010（Pin 声明系统）、ADR-0013（子图与宏）、ADR-0014（Pin 求值语义二分）、ADR-0019（AI 依赖反转）、RFC-0002（分层内核）、蓝图文档 §6.1 / §6.2 / §6.4 / §7

---

> **后续决策落点（2026-04-30）：** ADR-0021 复评后把本 RFC 中"文本通道 / 画布通道"的探索措辞收敛为"AI 编排控制台生成 DSL，编译后导入画布；画布页是唯一编辑/部署真值源"。本 RFC 仍记录 DSL 设计空间，产品入口与部署汇合以 ADR-0021 为准。

## 动机

蓝图文档 §7 将三段式 DSL 定义为"产品核心资产"：

1. **Device DSL**——设备 / 信号 / 协议 / 寄存器 / 数据转换的结构化描述
2. **Capability DSL**——将底层操作封装为受约束的设备能力（前置条件 / 副作用 / fallback / 安全等级）
3. **Workflow DSL**——业务状态机 / 触发条件 / 动作序列 / 异常处理的声明式描述

当前项目无自定义 DSL。工作流以 FlowGram JSON AST 透传部署，设备/能力建模完全缺失。Pin 声明系统（ADR-0010）部分覆盖信号/端口描述，但远未到 `DeviceSpec` / `CapabilitySpec` / `SignalSpec` 层级。

**问题**：没有设备模型和能力抽象，AI 无法从说明书生成结构化工作流；没有声明式状态机，工作流全靠画布拖拽——无法版本化、无法静态校验、无法离线仿真。蓝图定位"DSL 是核心资产"，当前核心资产缺失。

**目标**：设计一套与现有 Ring 0/1 架构兼容的三段式 DSL，使其成为引擎的"编译输入"——经过 Safety Compiler 校验后，编译为可部署的 DAG。

---

## 需求与约束

### 硬需求

1. **声明式**——三种 DSL 均为声明式（YAML / TOML / JSON），不含命令式逻辑
2. **可编译**——Workflow DSL 经 Safety Compiler 校验后，编译为引擎可部署的 DAG（`WorkflowGraph` + 节点注册）
3. **类型安全**——单位、量程、信号类型在编译期校验，不推迟到运行时
4. **Ring 纯净性**——DSL parser / compiler 作为 Ring 1 或独立 crate，不进入 `crates/core/`
5. **双通道输入**——保留 FlowGram JSON AST 作为"画布通道"的输入；DSL 作为"文本通道"的输入，两条路径最终编译到同一个 DAG schema
6. **AI 可生成**——DSL 语法足够规则化，AI 可从说明书 / 点表 / 自然语言描述生成

### 可协商

- DSL 具体格式（YAML vs TOML vs 自定义语法）
- Capability DSL 与 Device DSL 是否合并为同一文件
- Workflow DSL 的状态机是否复用现有 `loop` / `switch` 节点，还是引入新节点类型
- 编译输出的中间表示（IR）是否暴露为独立 crate

### 非目标（本 RFC 不覆盖）

- Safety Compiler 的 17 项校验规则细节（另开 ADR）
- AI 生成管道（说明书 → DeviceSpec → CapabilitySpec）的具体 prompt / chain 设计
- 仿真器 / 回放（偏离-05）
- edge-daemon 独立运行时（偏离-03）

## 设计

### 整体架构

```
                    ┌─────────────────────┐
                    │   AI / 人工编辑器     │
                    └─────────┬───────────┘
                              │ 生成 / 编辑
                    ┌─────────▼───────────┐
                    │  三段式 DSL 文件      │
                    │  device.yaml         │
                    │  capability.yaml     │
                    │  workflow.yaml       │
                    └─────────┬───────────┘
                              │ 解析
                    ┌─────────▼───────────┐
                    │  DSL Compiler        │  ← Ring 1 新 crate
                    │  (parser + checker)  │
                    └─────────┬───────────┘
                              │ 校验 + 编译
                    ┌─────────▼───────────┐
                    │  Safety Compiler     │  ← Ring 1（后续 ADR）
                    └─────────┬───────────┘
                              │ 通过 → WorkflowGraph
                    ┌─────────▼───────────┐
                    │  现有 DAG 部署管道     │
                    │  (deploy_workflow)   │
                    └─────────────────────┘

    画布通道（现有）：FlowGram JSON AST → 直接解析为 WorkflowGraph
    文本通道（新增）：DSL → Compiler → Safety → WorkflowGraph
```

两条通道共享同一个 DAG schema（`WorkflowGraph`），部署管道不需要知道输入来自哪条通道。

### 7.1 Device DSL

描述设备实体、信号、协议连接和数据转换。

```yaml
# device.yaml
device:
  id: hydraulic_press_1
  type: hydraulic_press
  manufacturer: "某液压"
  model: YP-320T
  connection:
    type: modbus-tcp          # 复用 ConnectionManager 的协议名
    id: press_modbus          # 引用连接配置中的 connection_id
    unit: 1

signals:
  pressure:
    type: analog_input        # 模拟量输入
    unit: MPa
    range: [0, 35]
    source:
      register: 40001         # Modbus 寄存器地址
      access: read
      data_type: float32
    scale: "raw * 35 / 65535" # 可选缩放表达式

  position:
    type: analog_input
    unit: mm
    range: [0, 150]
    source:
      register: 40003
      access: read
      data_type: float32

  servo_ready:
    type: digital_input
    source:
      register: 40100
      access: read
      bit: 0

  emergency_stop:
    type: digital_input
    source:
      register: 40100
      access: read
      bit: 1

  target_position:
    type: analog_output
    unit: mm
    range: [0, 150]
    source:
      register: 40010
      access: write
      data_type: float32

alarms:
  over_pressure:
    condition: "pressure > 34"
    severity: critical
    action: emergency_stop
```

**关键类型映射**（Rust 侧）：

```rust
/// Device DSL 编译后的结构化设备模型
struct DeviceSpec {
    id: String,
    device_type: String,
    connection: ConnectionRef,          // 引用 ConnectionManager 中的连接
    signals: Vec<SignalSpec>,
    alarms: Vec<AlarmSpec>,
}

struct SignalSpec {
    id: String,
    signal_type: SignalType,            // AnalogInput / AnalogOutput / DigitalInput / DigitalOutput
    unit: Option<String>,
    range: Option<(f64, f64)>,
    source: SignalSource,               // Register / Topic / SerialCommand
    scale: Option<String>,              // Rhai 表达式
}

enum SignalType {
    AnalogInput,
    AnalogOutput,
    DigitalInput,
    DigitalOutput,
}

struct AlarmSpec {
    id: String,
    condition: String,                  // Rhai 表达式
    severity: AlarmSeverity,
    action: Option<String>,
}
```

### 7.2 Capability DSL

将底层寄存器/信号操作封装为受约束的设备能力。AI 和 Workflow DSL 不直接操作寄存器，而是调用能力。

```yaml
# capability.yaml
capability:
  id: hydraulic_axis.move_to
  device: hydraulic_press_1
  description: "控制液压轴移动到指定位置"

  inputs:
    position:
      unit: mm
      range: [0, 150]
      required: true

  outputs:
    - position_reached: bool

  preconditions:
    - "servo_ready == true"
    - "emergency_stop == false"
    - "pressure < 32"

  effects:
    - "axis_state = moving"

  implementation:
    type: modbus-write               # 底层实现方式
    register: 40010                  # 写入 target_position 寄存器
    value: "${position}"             # 参数替换

  fallback:
    - hydraulic_axis.stop

  safety:
    level: high                      # high / medium / low
    requires_approval: false
    max_execution_time: 30s
```

**关键类型映射**：

```rust
struct CapabilitySpec {
    id: String,
    device_id: String,
    inputs: Vec<CapabilityParam>,
    outputs: Vec<CapabilityOutput>,
    preconditions: Vec<String>,         // Rhai 表达式列表
    effects: Vec<String>,
    implementation: CapabilityImpl,     // ModbusWrite / MqttPublish / SerialCommand / Script
    fallback: Vec<String>,              // 引用其他 capability id
    safety: SafetyConstraints,
}

struct CapabilityParam {
    id: String,
    unit: Option<String>,
    range: Option<(f64, f64)>,
    required: bool,
}

struct SafetyConstraints {
    level: SafetyLevel,                 // High / Medium / Low
    requires_approval: bool,
    max_execution_time: Option<Duration>,
}
```

### 7.3 Workflow DSL

描述业务状态机——状态、转移条件、触发动作、异常处理。

```yaml
# workflow.yaml
workflow:
  id: auto_pressing_cycle
  description: "自动压装循环"
  version: "1.0.0"

  devices:
    - hydraulic_press_1               # 引用 device.yaml

  variables:
    target_pressure: 25.0             # MPa
    hold_time: 5.0                    # seconds
    approach_position: 100.0          # mm

  states:
    idle:
      entry: []
      exit: []

    approaching:
      entry:
        - capability: hydraulic_axis.move_to
          args:
            position: "${approach_position}"

    pressing:
      entry:
        - capability: hydraulic_axis.apply_pressure
          args:
            target: "${target_pressure}"

    holding:
      entry:
        - capability: hydraulic_axis.hold_pressure
          args:
            target: "${target_pressure}"
            duration: "${hold_time}"

    returning:
      entry:
        - capability: hydraulic_axis.move_to
          args:
            position: 0.0

    fault:
      entry:
        - capability: hydraulic_axis.stop
        - action: alarm.raise
          args:
            message: "压装循环异常停机"

  transitions:
    - from: idle
      to: approaching
      when: "start_button == true"

    - from: approaching
      to: pressing
      when: "position >= approach_position"

    - from: pressing
      to: holding
      when: "pressure >= target_pressure"

    - from: holding
      to: returning
      when: "hold_timer >= hold_time"

    - from: returning
      to: idle
      when: "position <= 1.0"

    - from: "*"
      to: fault
      when: "pressure > 34"
      priority: 100

  timeout:
    pressing: 60s
    holding: 30s
    on_timeout: fault
```

**编译目标**：Workflow DSL 编译后生成一个 `WorkflowGraph`，其中：
- 每个 `state` 映射为一个子图（复用 ADR-0013 子图机制）
- 每个 `transition` 映射为条件判断节点（复用 `if` / `switch` 节点 + Pin 类型系统 ADR-0010）
- 每个 `capability` 调用映射为设备操作节点（新节点类型或脚本节点调用）
- `variables` 映射为 `WorkflowVariables`（复用 ADR-0012）

### Crate 归属

```
crates/
  dsl-core/            # Ring 1 — DeviceSpec / CapabilitySpec / WorkflowSpec 类型 + YAML parser
                      #   依赖: serde, serde_yaml, nazh_core (仅类型引用)
  dsl-compiler/        # Ring 1 — WorkflowSpec → WorkflowGraph 编译器
                      #   依赖: dsl-core, nazh-engine facade (或 src/graph/ 模块)
```

**Ring 纯净性保证**：
- `crates/core/` 不引入 `serde_yaml`、不引入 DSL 类型
- `DeviceSpec` / `CapabilitySpec` / `SignalSpec` 等类型定义在 `dsl-core`，不进 `crates/core/`
- 编译器输出 `WorkflowGraph`（现有类型），部署管道零修改

## 与现有系统的关系

| 现有系统 | DSL 如何复用 / 扩展 |
|----------|---------------------|
| ADR-0010 Pin 声明 | `SignalSpec` 的 `signal_type` / `unit` / `range` 与 Pin 类型系统对齐；编译器生成节点时自动填充 Pin 声明 |
| ADR-0012 工作流变量 | Workflow DSL 的 `variables` 直接编译为 `WorkflowVariables` 初始值 |
| ADR-0013 子图 | 每个 `state` 编译为子图，`transitions` 编译为子图间路由 |
| ADR-0014 PinKind | capability 调用节点区分 Exec pin（触发执行）和 Data pin（参数传入） |
| ADR-0019 AiService | AI 从说明书生成 DSL 文件，复用现有 `AiService` trait |
| ConnectionManager | Device DSL 的 `connection` 字段引用 `ConnectionManager` 中的连接 ID |
| FlowGram JSON AST | 保持不变；新增"文本通道"作为并行的编译输入 |

## 备选方案考虑

### 方案 A：三段式 YAML DSL（本提案）

三份独立 YAML 文件，各有专属 schema，编译器分阶段处理。

- **优势**：与蓝图 §7 完全对齐；关注点分离清晰；设备模型可跨工作流复用；AI 生成粒度自然（先生成设备，再生能力，最后组装工作流）
- **劣势**：三份文件增加管理复杂度；跨文件引用需要校验；初期学习成本

### 方案 B：单一统一 DSL

将 Device / Capability / Workflow 合并为一个 `.nazh` 文件，类似 Dockerfile。

- **优势**：单文件自包含；分发简单
- **劣势**：设备模型无法跨工作流复用；文件膨胀后难以维护；与蓝图三段式定位不符

### 方案 C：基于现有 FlowGram AST 扩展元数据

不引入新 DSL，而是在 FlowGram JSON AST 上增加 `device` / `capability` 元数据节点。

- **优势**：零新增 crate；画布编辑器可直接操作
- **劣势**：AST 不是人可读/可写的；AI 生成 JSON AST 比 YAML 困难；无法做声明式状态机；与蓝图定位冲突

### 方案 D：Rhai 脚本作为 DSL

用 Rhai 脚本描述设备和工作流，复用现有脚本引擎。

- **优势**：已有运行时支持；表达式能力强
- **劣势**：Rhai 是命令式语言，不是声明式 DSL——无法做静态校验（蓝图 §6.4 的 Safety Compiler 前提是声明式输入）；失去了"可编译、可校验"的核心价值

**推荐方案 A**。理由：蓝图明确要求三段式分离；AI 生成管道需要分阶段输出（设备 → 能力 → 工作流）；声明式是 Safety Compiler 的前提。

- [ ] 单元测试覆盖：合法 / 非法 DSL 文件解析
- [ ] 依赖：`nazh_core`（仅 Ring 0 类型引用）+ `serde` + `serde_yaml`

### Phase 1：Device DSL 编译器（~5 天）

- [ ] `DeviceSpec` → 设备注册到 `ConnectionManager` 的连接引用
- [ ] `SignalSpec` → Pin 声明自动生成（对接 ADR-0010）
- [ ] 设备目录管理（多设备加载、ID 唯一性校验）

- [ ] 集成测试：加载 `capability.yaml` → 验证参数校验 / 前置条件求值

### Phase 4：AI 生成管道对接（~5 天，依赖 Phase 1-3）

- [ ] 结构化输出约束（`uncertainties` / `warnings` 字段）

- [ ] 单位一致性校验

## 风险与未知

1. **Workflow DSL → WorkflowGraph 编译复杂度**：状态机到 DAG 的映射不简单——状态转移可能产生复杂拓扑。需要确定"一个状态 = 一个子图"是否足够，还是需要更细粒度的映射
2. **与 FlowGram 画布的双向同步**：DSL 编辑 → 画布更新 / 画布编辑 → DSL 生成，双向同步容易出错。初期可只做单向（DSL → 画布），反向推迟
3. **Rhai 表达式复用边界**：`preconditions` / `scale` / `condition` 用 Rhai 表达式，需确定求值上下文（哪些变量可访问、sandbox 限制）
4. **Device DSL 与 ConnectionManager 耦合**：当前连接配置是前端 JSON 文件直写。Device DSL 引入后，连接配置是否也从 DSL 生成？还是保持双轨？
5. **性能**：大量设备的 YAML 解析 + 编译开销。初期不考虑，后期按需优化（缓存编译结果）

---

## 未解决的问题

- DSL 文件的存放位置（项目目录结构？数据库？）——依赖 RFC-0003 边缘存储层
- Capability 的 `implementation` 字段是否需要支持组合能力（一个 capability 由多个底层操作组成）
- Workflow DSL 的 `timeout` / `retry` 策略是否在 DSL 层声明，还是由 ADR-0016 边级可观测性覆盖
- 是否需要一个 `dsl-lsp` crate 为编辑器提供补全/校验（远期）
