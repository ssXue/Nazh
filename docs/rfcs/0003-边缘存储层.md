# RFC-0003：边缘存储层

> **状态：** 提议中
> **日期：** 2026-04-30
> **决策者：** ssXue
> **关联：** 偏离-06（持久化）、偏离-03（edge-daemon）、偏离-10（审计）、RFC-0002（分层内核）、ADR-0012（工作流变量 Phase 3）、蓝图文档 §12.5

---

## 动机

当前项目所有运行时状态（DataStore 载荷、工作流变量、节点生命周期）纯内存，进程退出即清零。可观测日志以 JSONL 追加写落地，但无索引、无 schema 约束、无高效查询。配置类数据靠 JSON 文件直写。

蓝图文档 §12.5 明确推荐 SQLite 作为本地存储，覆盖：项目配置、设备模型、工作流版本、部署记录、日志索引、报警记录、审批记录、仿真结果。当前项目无对应实现。

随着偏离项推进（尤其是 edge-daemon 独立运行时、设备模型持久化、审计追溯），结构化本地存储成为前置依赖。需要一次设计讨论来确定存储层的定位、边界和实施策略。

---

## 需求与约束

### 硬需求

1. **进程重启恢复**——工作流变量、运行时快照在 daemon 重启后可恢复
2. **结构化查询**——可观测数据按 workflow_id / 时间范围 / 事件类型检索，不扫全文件
3. **审计追溯**——安全拒绝、人工审批、状态转移记录可索引查询
4. **desktop / daemon 共用**——同一套存储 crate，两个可执行体都是消费方
5. **Ring 纯净性**——`rusqlite` / `sqlx` 不得进入 Ring 0（`crates/core/`）

### 可协商

- 是否替换现有 JSONL 可观测日志（还是双写期后迁移）
- schema 版本迁移策略（手动 SQL vs 内建 migration 框架）
- 事务粒度（单变量写入 vs 批量快照）
- 是否引入 ORM 抽象（diesel / sea-orm）还是裸 SQL

### 非目标

- 替代 DataStore（`ArenaDataStore`）的内存数据通道——载荷传输仍走 DashMap
- 引入远程数据库——纯本地 SQLite，不依赖网络
- 成为通用 KV 存储——变量持久化走专用表，不自建 KV 抽象

## 设计

### Crate 归属

```
crates/
  store/              # Ring 1 —— 本地 SQLite 存储层
```

**Ring 1 定位**：依赖 Ring 0（`EngineError`、`ExecutionEvent` 类型），不依赖其他 Ring 1 crate。`rusqlite` 仅此 crate 引入。

Ring 0 保持纯净——`crates/core/` 只定义 `DataStore` trait（已有）和事件类型，不感知 SQLite。

### 核心抽象

```rust
// crates/store/src/lib.rs

/// 本地存储引擎
pub struct Store {
    db: rusqlite::Connection,
}

/// 表域（schema namespace）
pub enum StoreDomain {
    Variables,       // 工作流变量快照
    Observability,   // 事件索引（替代 JSONL 扫描）
    Audit,           // 审批 / 安全拒绝记录
    Deployment,      // 部署包版本 / 部署记录
    Config,          // 设备模型 / 能力模型 / 连接配置（未来）
}
```

### 表结构（v0.1 最小集）

#### `variables` — 工作流变量持久化

```sql
CREATE TABLE IF NOT EXISTS variables (
    workflow_id TEXT NOT NULL,
    key         TEXT NOT NULL,
    value       TEXT NOT NULL,        -- JSON serialized TypedVariable
    var_type    TEXT NOT NULL,        -- "i64" | "f64" | "bool" | "string"
    updated_at  TEXT NOT NULL,        -- ISO 8601
    PRIMARY KEY (workflow_id, key)
);
```

#### `events` — 可观测事件索引

```sql
CREATE TABLE IF NOT EXISTS events (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    workflow_id TEXT NOT NULL,
    trace_id    TEXT NOT NULL,
    node_id     TEXT,
    event_type  TEXT NOT NULL,        -- "Started" | "Completed" | "Failed" | ...
    payload     TEXT,                 -- JSON，可选
    timestamp   TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_events_workflow_time
    ON events(workflow_id, timestamp);
```

#### `audit_log` — 审计记录

```sql
CREATE TABLE IF NOT EXISTS audit_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    workflow_id TEXT NOT NULL,
    action      TEXT NOT NULL,        -- "deploy" | "undeploy" | "variable_write" | "safety_reject" | "manual_approve"
    actor       TEXT,                 -- "user" | "system" | "ai"
    detail      TEXT,                 -- JSON
    timestamp   TEXT NOT NULL
);
```

#### `deployments` — 部署记录

```sql
CREATE TABLE IF NOT EXISTS deployments (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    workflow_id TEXT NOT NULL,
    version     INTEGER NOT NULL DEFAULT 1,
    ast_hash    TEXT NOT NULL,        -- AST 内容哈希，用于变更检测
    status      TEXT NOT NULL,        -- "active" | "superseded" | "rolled_back"
    deployed_at TEXT NOT NULL
);
```

#### `schema_version` — 迁移管理

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    version     INTEGER PRIMARY KEY,
    applied_at  TEXT NOT NULL
);
```

### 与现有系统的集成点

```
┌─────────────────────────────────────────────────┐
│ desktop-tauri                                    │
│   └── commands/ → Store (读/写)                  │
│   └── observability.rs → Store (事件索引)         │
│   └── deployment_session.rs → Store (部署记录)    │
└─────────────────────────────────────────────────┘
         ↕ 共用 crates/store
┌─────────────────────────────────────────────────┐
│ edge-daemon（未来）                               │
│   └── 变量恢复 → Store (启动时读)                  │
│   └── 事件写入 → Store (运行时写)                  │
│   └── 审计记录 → Store (运行时写)                  │
└─────────────────────────────────────────────────┘
```

### 写入策略

| 数据 | 写入时机 | 事务语义 |
|------|----------|----------|
| 变量 | `set_workflow_variable` 调用时同步写 | 单行 UPSERT |
| 事件索引 | `ExecutionEvent` 发射时异步写（mpsc → writer task） | 批量 INSERT（100ms 或 100 条 flush） |
| 审计 | 关键动作触发时同步写 | 单行 INSERT |
| 部署记录 | `deploy_workflow` / `undeploy_workflow` 时同步写 | 事务包裹（status 更新 + 新记录） |

### Migration 策略

内建 `migrations/` 目录，按版本号顺序执行：

```
crates/store/src/migrations/
  001_initial.sql      -- 建表
  002_xxx.sql          -- 未来增量
```

`Store::open(path)` 时检查 `schema_version`，自动执行未应用的 migration。不引入 diesel / sea-orm，手写 SQL 保持依赖最小。

## 备选方案考虑

### 方案 A：SQLite 共享 crate（本 RFC 推荐）

新增 `crates/store/` Ring 1 crate，封装 `rusqlite`，提供 typed API。

优势：
- desktop / daemon 共用，一份 schema 一份 migration
- Ring 0 保持纯净
- 与现有 `rusqlite` workspace 依赖一致（`nodes-io` 已验证 bundled 编译）

劣势：
- 新增 crate 维护成本
- `rusqlite` bundled 编译 ~5MB 产物（ADR-0018 已记录）

### 方案 B：扩展 JSONL + 内存索引

保持 JSONL 追加写，启动时加载到内存 HashMap 索引。

优势：
- 无新依赖，无 SQLite
- 追加写性能高

劣势：
- 无事务语义（变量快照可能半写）
- 文件膨胀后启动加载慢
- 查询能力受限于内存索引结构
- 无法支持 edge-daemon 的断网自治 + 重启恢复场景

### 方案 C：sled / redb 等 embedded KV

优势：
- 纯 Rust，无 C 编译
- KV 语义简单

劣势：
- 项目已有 `rusqlite` 依赖，引入第二个存储引擎增加复杂度
- 无 SQL 查询能力，复杂查询需应用层实现
- 社区生态不如 SQLite 成熟

### 方案 D：延迟到 edge-daemon 实施时再建

当前 desktop 阶段 JSONL 够用，存储层随 daemon 一起建。

优势：
- 减少当前工作量
- daemon 需求明确后再设计

劣势：
- 变量持久化、审计追溯等能力继续缺失
- daemon 立项时存储层成为阻塞项
- desktop 已积累的 JSONL 数据需要迁移

## 实施拆解

### Phase 1：最小 crate + 变量持久化

- 新建 `crates/store/`，引入 `rusqlite` workspace 依赖
- 实现 `Store::open()` + migration 执行器
- 实现 `variables` 表 + `StoreVariable` 读/写 API
- `WorkflowVariables` 集成：启动时从 Store 恢复、写入时同步持久化
- Tauri shell 启动时初始化 Store，传入引擎
- 合约测试

### Phase 2：可观测事件索引

- 实现 `events` 表 + 批量写入 writer task
- `observability.rs` 集成：事件同时写 JSONL（兼容期）+ Store
- IPC `query_observability` 改为查询 Store 而非读文件
- JSONL 保留为 fallback，标记 deprecated

### Phase 3：审计 + 部署记录

- 实现 `audit_log` + `deployments` 表
- 部署/卸载动作写入审计记录
- 变量变更写入审计记录
- 部署版本管理（ast_hash 变更检测）
- IPC 暴露审计查询命令

### Phase 4：配置迁移（随偏离-01/04 推进）

- 连接配置从 `connections.json` 迁入 Store
- 设备模型 / 能力模型（DSL 落地后）存入 Store
- AI 配置迁移考虑（可选）

## 风险与未知

1. **`rusqlite` bundled 编译在交叉编译场景（ARM 边缘设备）的行为**——需验证 `CC` / `AR` 环境变量配置。当前 `nodes-io` 仅在 x86_64 macOS 验证过。
2. **变量恢复时机**——daemon 重启后变量从 Store 恢复，但节点生命周期（Timer/Serial/MQTT 的 `on_deploy`）需要重新触发。恢复 ≠ 热续，语义需明确。
3. **事件写入吞吐**——高频场景（1000+ events/s）下批量写入的 flush 策略需实测调优。
4. **JSONL → Store 迁移期**——Phase 2 双写期间磁盘占用翻倍，需设截止日期移除 JSONL。
5. **Store crate 与 Ring 0 的耦合边界**——`Store` 需要引用 `ExecutionEvent` 类型做序列化，但 `ExecutionEvent` 在 Ring 0。方案：Ring 0 定义序列化 trait，Store 消费；或 Store 直接 serde 序列化为 JSON TEXT 列。后者更简单。
